## 二、标准推进顺序
请确认当前项目按以下顺序推进，而不是跳步进入实现：

- [ ] **第一性原理确认需求与目标**
- [ ] **收口 MVP 范围与后续迭代边界**
- [ ] **建立正式工作流与治理入口**
- [ ] **整理待拍板事项并完成启动拍板**
- [ ] **团队搭建与多角色面具配置**
- [ ] **多角色协作产出高质量设计方案**
- [ ] **完成设计拍板**
- [ ] **进入正式实现与联调验证**
- [ ] **执行进度维护、验证复盘与资产沉淀**

## 二、阶段门禁 Checklist

### 2.1 第一性原理确认

- [ ] 模块 / 项目的**本质一句话定义**已经明确
- [ ] 已明确它在整体工作流中的职责
- [ ] 已明确上游 / 下游边界
- [ ] 已明确当前模块**不负责什么**
- [ ] 已形成 `FIRST_PRINCIPLES_AND_ROLE_ESSENCE.md` 或等价文档

**未满足以上任一项时，不进入 MVP 收口。**

### 2.2 MVP 范围收口

- [ ] 已明确当前 MVP 的主对象 / 主闭环
- [ ] 已明确当前阶段**必须做什么**
- [ ] 已明确当前阶段**不做什么**
- [ ] 已明确后续增强项 / P1 / P2 / 下一阶段迭代方向
- [ ] 已形成 `MVP_SCOPE_AND_ITERATION_ALIGNMENT.md` 或等价文档

**未完成范围收口时，不进入正式治理与设计。**

### 2.3 工作流与治理入口建立

- [ ] 已有正式导航文档，如 `工作流总览与协作导航.md`
- [ ] 已写清当前阶段判断
- [ ] 已写清推荐阅读顺序
- [ ] 已写清接手顺序与协作方式
- [ ] 已写清后续要补的治理文档

**没有治理入口时，不进入多人协作设计。**

### 2.4 启动拍板

- [ ] 已整理关键待拍板事项
- [ ] 每个拍板项都包含：可选方案 / 推荐方案 / 推荐理由 / 延后风险 / 结论
- [ ] 已明确哪些边界、策略、契约必须冻结
- [ ] 已形成 `待拍板决策清单.md` 与 `启动与拍板.md` 或等价文档
- [ ] 玩家 / 用户 / owner 已完成关键拍板

**关键分歧未冻结时，不进入正式设计。**

### 2.5 团队搭建与角色面具配置

- [ ] 已明确需要覆盖哪些职责视角
- [ ] 已明确标准配置与压缩配置
- [ ] 已明确各角色的职责边界、输入输出与非职责
- [ ] 已形成 `角色面具配置方案.md` 与 `roles.md` 或等价文档
- [ ] 已明确哪些事项仍需要玩家拍板，不能由执行端替代

**角色依据未落档时，不进入高质量方案设计。**

### 2.6 设计方案产出

- [ ] 已形成正式 `设计方案.md` 草案
- [ ] 已明确主流程
- [ ] 已明确关键对象 / Schema / 输入输出
- [ ] 已明确当前设计严格服务 MVP，而非未来全部想象空间
- [ ] 已明确哪些开放问题可带入实现，哪些必须先关闭

**没有正式设计基线时，不进入实现。**

### 2.7 设计拍板

- [ ] 已确认主流程可按当前方案推进
- [ ] 已确认关键对象骨架 / Schema 可以冻结
- [ ] 已确认实现策略未越过 MVP 边界
- [ ] 已确认遗留开放问题不会导致实现期大范围返工
- [ ] 已记录正式设计拍板结论

**设计未拍板通过时，不启动正式实现。**

### 2.8 实现与联调验证

- [ ] 已实现 MVP 最小功能闭环
- [ ] 已保留运行记录 / 验证记录 / 联调结果
- [ ] 已形成 `执行进度.md` 或等价进度文档
- [ ] 关键变更已回写到正式执行轨文档
- [ ] 验证材料与 MVP 主闭环直接绑定

**只写代码、不留验证记录，视为未完成实现阶段。**

### 2.9 复盘与资产沉淀

- [ ] 已沉淀复盘结论
- [ ] 已沉淀可复用模板 / 契约 / Prompt / Schema / 验证材料
- [ ] 已形成后续迭代优先级清单
- [ ] 已形成可迁移到下个项目的流程资产

**没有沉淀资产时，项目只算一次性完成，不算方法论沉淀完成。**

## 四、启动前必问 10 个问题

- [ ] 这个模块为什么存在？
- [ ] 它的本质职责是什么？
- [ ] 它和上下游的边界是什么？
- [ ] 当前 MVP 只做什么？
- [ ] 当前明确不做什么？
- [ ] 哪些问题必须先拍板？
- [ ] 设计前谁来拍板，拍什么？
- [ ] 需要哪些正式职责视角参与设计？
- [ ] 当前设计方案是否真的服务 MVP？
- [ ] 最后要沉淀哪些可复用资产？

## 五、禁止跳过的原则

- [ ] **不允许跳过第一性原理直接写功能列表**
- [ ] **不允许未做 MVP 收口就直接展开大而全设计**
- [ ] **不允许没有拍板就默认关键问题已确定**
- [ ] **不允许没有角色依据就开始多角色协作设计**
- [ ] **不允许设计未冻结就直接进入正式实现**
- [ ] **不允许实现后不回写正式文档**
- [ ] **不允许项目结束后不沉淀复用资产**

## 六、适合直接复制使用的一段启动说明

> 新项目启动时，不直接从功能实现开始，而是先按标准 workflow 推进：先确认模块本质与职责边界，再收口 MVP 范围，随后建立工作流导航、待拍板清单与启动拍板文档；拍板完成后，再进行团队搭建、角色面具配置与设计方案产出；设计拍板通过后，才进入实现、联调、验证与复盘，并将关键模板、契约和验证资产沉淀为后续项目可复用材料。
