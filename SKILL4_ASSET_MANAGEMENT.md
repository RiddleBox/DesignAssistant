# Skill 4：项目资产统一管理

## 📋 技能概述

**技能名称**：项目资产统一管理  
**技能编号**：SKILL-004  
**版本**：v1.0.0  
**依赖**：Skill 1（数字员工招募与管理）、Skill 2（团队项目协作）  
**数据域**：`data-layer/assets/`（读写）、`projects/*/shared/`（读写）

**核心功能**：
1. 统一管理项目资产（文档、代码、设计稿、配置文件等）
2. 自动分类和元数据管理
3. 版本控制和变更追踪
4. 多维度检索和引用追踪
5. 资产统计和报告

**设计原则**：
- ✅ **领域无关**：通用资产分类，适用于任何领域
- ✅ **低耦合**：只写assets/和projects/*/shared/，通过接口读取其他数据
- ✅ **零依赖**：使用系统内置命令和AI工具
- ✅ **版本安全**：自动保留历史版本，防止数据丢失

---

## 🎯 使用场景

### 场景1：保存项目资产
```
用户："将这份需求文档保存到项目资产库"
→ 自动识别资产类型（文档）
→ 生成元数据（创建者、时间、版本、标签）
→ 保存到项目共享资源目录
→ 更新资产索引
```

### 场景2：检索资产
```
用户："查找所有UI设计稿"
→ 按类型检索
→ 展示资产列表（名称、创建者、时间、版本）
→ 支持预览和下载
```

### 场景3：版本管理
```
用户："查看需求文档的历史版本"
→ 展示版本列表
→ 显示变更日志
→ 支持版本对比和回滚
```

### 场景4：引用追踪
```
用户："删除这个API文档"
→ 检查引用关系
→ 发现被3个任务引用
→ 警告并询问确认
```

## 🔧 核心功能实现

### Step 0：环境检测与自动初始化

在开始任何操作前，必须先执行环境检测：

1. **检查资产数据层**：使用 `read_file` 检查 `data-layer/assets/index.json` 是否存在
2. **数据层不存在** → 执行零依赖自举初始化：
   - 告知用户：\"检测到资产数据层尚未初始化，正在自动创建...\"
   - 使用 `terminal` 创建 `data-layer/assets/` 目录
   - 使用 `edit_file` 创建 `index.json` 文件（初始化为空资产列表）

3. **检查项目资产配置**：
   - 如果用户操作涉及特定项目，检查该项目的 `config/assets.json` 是否存在
   - 不存在 → 自动创建项目资产配置文件

### Step 2A：保存资产

#### 阶段一：收集资产信息

1. **检查用户是否指定了项目**
   - 已指定 → 继续
   - 未指定 → 询问：\"请选择要保存资产的项目\"

2. **检查用户是否提供了资产内容**
   - 已提供（附件、代码块、文本） → 继续
   - 未提供 → 询问：\"请提供要保存的资产内容\"

3. **询问资产基本信息**：
   - \"请提供资产名称（例如：产品需求文档v1.0）\"
   - \"请选择资产类型（文档/代码/设计/配置/其他）\"
   - \"请提供资产描述（可选）\"
   - \"请添加标签（多个标签用逗号分隔，可选）\"

#### 阶段二：自动分类和生成元数据

1. **确定资产分类**：
   - 根据用户选择的类型，映射到对应的category
   - 文档 → docs
   - 代码 → code
   - 设计 → design
   - 配置 → config
   - 其他 → other

2. **生成资产ID**：
   - 格式：`asset_YYYYMMDD_NNN`
   - 示例：`asset_20260212_001`

3. **生成元数据**：
   - 创建者：从当前上下文获取（如果有员工身份）
   - 创建时间：当前时间
   - 版本：1.0（初始版本）
   - 文件大小：计算内容大小
   - 标签：用户提供的标签 + 自动生成的标签（基于类型和内容）

#### 阶段三：保存资产文件

1. **确定保存路径**：
   - 路径格式：`projects/{项目ID}/shared/{分类目录}/{资产名称}`
   - 示例：`projects/proj_20260212_001/shared/documents/产品需求文档v1.0.md`

2. **创建资产文件**：
   - 使用 `edit_file` 创建资产文件
   - 写入用户提供的内容

3. **更新目录元数据**：
   - 读取或创建 `metadata.json`
   - 添加新资产的元数据记录

#### 阶段四：更新资产索引

1. **更新全局资产索引**：
   - 使用 `replace_in_file` 在 `data-layer/assets/index.json` 中添加新资产记录
   - 更新统计数据：total +1, byCategory[分类] +1, byProject[项目ID] +1

2. **更新项目资产配置**：
   - 更新 `projects/{项目ID}/config/assets.json` 的统计数据

#### 阶段五：反馈用户

```
✅ 资产保存成功！

资产信息：
- 资产ID：asset_20260212_001
- 资产名称：产品需求文档v1.0
- 类型：文档
- 项目：软件开发团队项目
- 创建者：产品经理
- 版本：1.0
- 大小：15 KB
- 保存路径：projects/proj_20260212_001/shared/documents/产品需求文档v1.0.md

标签：需求、产品、v1.0

下一步操作：
- 查看资产详情
- 分享给团队成员
- 关联到任务
```

### Step 2E：更新资产

#### 阶段一：定位资产和收集信息

1. **检查用户是否指定了资产**
   - 已指定 → 继续
   - 未指定 → 展示资产列表，询问选择

2. **检查用户是否提供了新内容**
   - 已提供 → 继续
   - 未提供 → 询问：\"请提供更新后的资产内容\"

3. **询问变更信息**：
   - \"请描述本次变更内容（用于版本日志）\"

#### 阶段二：创建新版本

1. **生成新版本号**：
   - 读取当前版本号（例如：1.2）
   - 生成新版本号（例如：1.3）

2. **保存新版本文件**：
   - 备份旧版本：重命名为 `{资产名称}_v{旧版本号}.{扩展名}`
   - 保存新版本：使用原文件名

3. **更新元数据**：
   - 更新资产索引中的version、updatedAt、size
   - 在versionHistory中添加新版本记录

#### 阶段三：版本控制

1. **检查版本数量**：
   - 如果版本数量超过maxVersions（默认10），删除最旧的版本文件

2. **更新变更日志**：
   - 在versionHistory中记录变更信息

## 📋 工作准则

1. **数据访问规则**：
   - 只写 `data-layer/assets/` 和 `projects/*/shared/` 目录
   - 只读其他数据层目录
   - 通过接口调用Skill 1和Skill 2

2. **资产分类原则**：
   - 使用通用分类（docs/code/design/config/other）
   - 不预设特定领域的分类
   - 支持用户自定义标签

3. **版本控制规范**：
   - 每次更新自动创建新版本
   - 保留最近N个版本（默认10）
   - 版本号格式：主版本.次版本（例如：1.2）

4. **引用追踪规则**：
   - 删除前必须检查引用关系
   - 有引用时必须警告用户
   - 删除后不自动清理引用（需手动处理）

5. **错误处理**：
   - 资产不存在时，展示资产列表
   - 保存失败时，提示用户并回滚
   - 删除有引用的资产时，必须二次确认

6. **性能优化**：
   - 大文件（>10MB）不读取内容，只显示元数据
   - 资产列表分页显示（每页20条）
   - 搜索结果限制数量（最多100条）
