# 实施计划

本实施计划基于多Skill协作架构,按照优先级分阶段实现功能。首先完成P0核心功能,建立基础设施,然后逐步添加P1和P2增强功能。

**重要说明**:Skill 1将采用**渐进式重构策略**,复用现有SKILL.md的60%核心流程,同时在数据结构中预留扩展字段,为后续多Skill协作做好准备。

**架构设计原则**:
- **领域无关性**:所有任务和功能设计不针对特定行业,保持通用性
- **低耦合架构**:各Skill独立可插拔,用户按需组合
- **高度可扩展**:支持任意领域的员工类型和协作模式

## 实施建议

### 开发顺序
1. **第一迭代**(1-2周):完成阶段一和阶段二,建立基础设施和核心协作功能
   - 重点:复用现有SKILL.md,扩展数据结构,实现项目工作空间和团队模板
2. **第二迭代**(1-2周):完成阶段三,实现资产和任务管理
   - 重点:资产库、任务看板、依赖关系管理
3. **第三迭代**(1周):完成阶段四和阶段五,添加知识库和日志分析
   - 重点:团队知识库共享、工作日志自动记录、复盘报告
4. **第四迭代**(1周):完成阶段六,实现高级功能并进行集成测试
   - 重点:员工技能树、外部工具集成、系统集成测试

### 技术栈建议
- **数据存储**:JSON文件(轻量级、易于版本控制)
- **文件操作**:Node.js fs模块或Python pathlib
- **数据验证**:JSON Schema
- **日志系统**:Winston或Python logging
- **测试框架**:Jest(JS)或pytest(Python)

### 关键注意事项
1. **Skill 1复用策略**:保留现有SKILL.md的60%核心流程,在数据结构中预留扩展字段,避免后期大规模重构
2. **向后兼容**:扩展的字段和目录初始为空,不影响现有功能
3. **接口优先**:在实现具体功能前,先定义好跨Skill的数据接口
4. **数据访问规则**:各Skill必须严格遵守数据访问规则(只写自己的数据域)
5. **渐进增强**:新增的工作准则在相关Skill未实现时不会报错,只是不会被触发
6. **优先保证P0功能的稳定性**,再逐步添加P1/P2功能
7. **每个Skill应该能够独立部署和测试**
8. **领域无关性**:所有实现不绑定特定行业,保持通用性和可扩展性

### 复用现有SKILL.md的关键点
- **直接复用**:环境检测、花名册校验、招聘流程、提示词自维护、工作空间感知、联网搜索
- **扩展字段**:花名册增加"所属项目"和"技能标签",员工档案增加"参与项目"、"技能树"、"协作记录"模块
- **扩展目录**:工作空间增加`协作记录/`和`项目引用/`目录
- **扩展准则**:system_prompt.md增加"团队协作感知"、"知识库优先"、"协作记录维护"三条工作准则
- **预留接口**:实现"更新员工所属项目"、"查询员工技能标签"、"更新员工技能树"、"读取员工档案"四个跨Skill接口

### 领域适用性保证
- **资产分类**:使用通用分类(docs/code/design/config/other),不预设特定领域分类
- **知识库分类**:使用通用分类(技术方案/业务规范/行业趋势/最佳实践),适用于任何领域
- **团队模板**:提供示例模板但支持用户自定义,不限制特定领域
- **任务管理**:通用的任务看板和依赖关系管理,不绑定特定工作流
- **技能树**:根据岗位动态生成,不预设特定领域的技能体系
- **所有配置通过JSON文件管理**,用户可根据自己的领域需求灵活调整
