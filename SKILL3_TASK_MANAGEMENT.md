# Skill 3：任务与里程碑管理

## 📋 技能概述

**技能名称**：任务与里程碑管理  
**技能编号**：SKILL-003  
**版本**：v1.0.0  
**依赖**：Skill 1（数字员工招募与管理）、Skill 2（团队项目协作）、Skill 4（项目资产统一管理）  
**数据域**：`data-layer/tasks/`（读写）、`projects/*/tasks/`（读写）

**核心功能**：
1. 任务创建、分配和状态管理
2. 任务依赖关系管理
3. 任务看板可视化
4. 里程碑管理和进度追踪
5. 任务阻塞检测和提醒

**设计原则**：
- ✅ **领域无关**：通用任务管理，适用于任何项目类型
- ✅ **低耦合**：只写tasks/目录，通过接口读取其他数据
- ✅ **零依赖**：使用系统内置命令和AI工具
- ✅ **灵活可扩展**：支持自定义任务状态和优先级

---

## 🎯 使用场景

### 场景1：创建任务
```
用户："创建一个任务：设计首页UI"
→ 收集任务信息（名称、描述、负责人、优先级）
→ 自动生成任务ID
→ 保存到任务看板
→ 通知负责人
```

### 场景2：查看任务看板
```
用户："查看项目任务看板"
→ 按状态分组展示（待办/进行中/已完成/阻塞）
→ 显示任务优先级和负责人
→ 高亮显示阻塞任务
```

### 场景3：任务依赖管理
```
用户："设置任务B依赖任务A"
→ 建立依赖关系
→ 检查循环依赖
→ 任务A完成后自动通知任务B负责人
```

### 场景4：里程碑管理
```
用户："创建里程碑：MVP版本发布"
→ 设置里程碑目标和截止日期
→ 关联相关任务
→ 自动计算完成进度
→ 延期预警
```

## 🔧 核心功能实现

### Step 0：环境检测与自动初始化

在开始任何操作前，必须先执行环境检测：

1. **检查任务数据层**：使用 `read_file` 检查 `data-layer/tasks/index.json` 是否存在
2. **数据层不存在** → 执行零依赖自举初始化：
   - 告知用户：\"检测到任务数据层尚未初始化，正在自动创建...\"
   - 使用 `terminal` 创建 `data-layer/tasks/` 目录
   - 使用 `edit_file` 创建 `index.json` 文件（初始化为空任务列表）

3. **检查项目任务配置**：
   - 如果用户操作涉及特定项目，检查该项目的 `tasks/board.json` 是否存在
   - 不存在 → 自动创建项目任务看板文件

### Step 2A：创建任务

#### 阶段一：收集任务信息

1. **检查用户是否指定了项目**
   - 已指定 → 继续
   - 未指定 → 询问：\"请选择要创建任务的项目\"

2. **收集任务基本信息**：
   - \"请提供任务名称\"
   - \"请提供任务描述（可选）\"
   - \"请选择任务负责人（输入员工名称或ID）\"
   - \"请选择任务优先级（高/中/低，默认：中）\"
   - \"请输入预计工时（小时，可选）\"
   - \"请输入截止日期（YYYY-MM-DD，可选）\"
   - \"请添加标签（多个标签用逗号分隔，可选）\"

3. **询问是否设置依赖**：
   - \"是否有前置任务？（是/否）\"
   - 如果是 → 展示任务列表，询问选择

#### 阶段二：生成任务数据

1. **生成任务ID**：
   - 格式：`task_YYYYMMDD_NNN`
   - 示例：`task_20260212_001`

2. **初始化任务状态**：
   - 如果有未完成的前置任务 → status = \"blocked\"
   - 否则 → status = \"todo\"

3. **检查依赖关系**：
   - 如果设置了依赖，检查是否存在循环依赖
   - 存在循环依赖 → 警告并拒绝创建

#### 阶段三：保存任务数据

1. **更新任务索引**：
   - 使用 `replace_in_file` 在 `data-layer/tasks/index.json` 中添加新任务
   - 更新统计数据：total +1, byStatus[状态] +1, byProject[项目ID] +1

2. **更新项目任务看板**：
   - 在对应状态列中添加任务ID
   - 更新看板统计数据

3. **创建任务详情文件**（可选）：
   - 在 `projects/{项目ID}/tasks/{任务ID}.md` 创建任务详情文档

#### 阶段四：通知相关人员

1. **通知任务负责人**：
   - 在负责人的工作空间创建任务引用文件
   - 记录到协作记录

2. **如果有依赖任务**：
   - 更新前置任务的blockedBy字段

#### 阶段五：反馈用户

```
✅ 任务创建成功！

任务信息：
- 任务ID：task_20260212_003
- 任务名称：前端页面开发
- 负责人：前端工程师（emp_20260212_002）
- 优先级：高
- 状态：阻塞（等待前置任务完成）
- 预计工时：24小时
- 截止日期：2026-02-16 18:00
- 前置任务：UI设计-首页（task_20260212_002）

标签：前端、开发

下一步操作：
- 查看任务看板
- 通知负责人
- 关联资产
```

### Step 2D：更新任务状态

#### 阶段一：定位任务和收集信息

1. **检查用户是否指定了任务**
   - 已指定 → 继续
   - 未指定 → 展示任务列表，询问选择

2. **询问新状态**：
   - \"请选择新状态（待办/进行中/已完成/阻塞）\"

3. **状态变更验证**：
   - 如果变更为\"进行中\"，检查是否有未完成的前置任务
   - 如果有 → 警告：\"前置任务尚未完成，确认要开始吗？\"
   - 如果变更为\"已完成\"，询问实际工时

#### 阶段二：执行状态变更

1. **更新任务数据**：
   - 更新status字段
   - 更新时间戳（startedAt或completedAt）
   - 如果完成，更新actualHours

2. **更新任务索引**：
   - 使用 `replace_in_file` 更新任务记录
   - 更新统计数据：byStatus[旧状态] -1, byStatus[新状态] +1

3. **更新项目看板**：
   - 从旧状态列移除任务ID
   - 添加到新状态列
   - 更新看板统计数据

#### 阶段三：处理依赖关系

1. **如果任务完成**：
   - 查找所有依赖此任务的任务
   - 检查这些任务是否还有其他未完成的前置任务
   - 如果没有，将状态从\"blocked\"变更为\"todo\"
   - 通知相关负责人

2. **如果任务阻塞**：
   - 更新blockedBy字段
   - 通知负责人

#### 阶段四：更新里程碑进度

1. **查找关联的里程碑**：
   - 读取 `milestones.json`
   - 查找包含此任务的里程碑

2. **重新计算进度**：
   - 统计已完成任务数
   - 计算完成百分比
   - 检查是否延期

### Step 2F：设置任务依赖

#### 阶段一：定位任务和选择前置任务

1. **检查用户是否指定了任务**
   - 已指定 → 继续
   - 未指定 → 展示任务列表，询问选择

2. **检查用户是否指定了前置任务**
   - 已指定 → 继续
   - 未指定 → 展示任务列表，询问选择

3. **验证依赖关系**：
   - 检查前置任务是否存在
   - 检查是否会形成循环依赖
   - 如果会 → 拒绝并提示

#### 阶段二：建立依赖关系

1. **更新任务数据**：
   - 在dependencies数组中添加前置任务ID
   - 如果前置任务未完成，更新blockedBy数组
   - 如果前置任务未完成，将status变更为\"blocked\"

2. **更新前置任务数据**：
   - 记录被哪些任务依赖

3. **更新任务索引**：
   - 使用 `replace_in_file` 更新两个任务的记录

4. **更新项目看板**：
   - 如果状态变更为\"blocked\"，移动任务到阻塞列

### Step 2G：删除任务

#### 阶段一：定位任务和安全确认

1. **检查用户是否指定了任务**
   - 已指定 → 继续
   - 未指定 → 展示任务列表，询问选择

2. **检查依赖关系**：
   - 查找是否有其他任务依赖此任务
   - 如果有 → 警告用户

3. **安全确认**：
   ```
   ⚠️ 删除警告
   
   任务：UI设计-首页（task_20260212_002）
   依赖情况：被 1 个任务依赖
   - task_20260212_003（前端页面开发）
   
   删除此任务可能影响相关任务的执行。
   
   确认要删除吗？（输入任务名称以确认）
   ```

#### 阶段二：执行删除

1. **更新依赖任务**：
   - 从依赖任务的dependencies数组中移除此任务ID
   - 从依赖任务的blockedBy数组中移除此任务ID
   - 如果依赖任务不再被阻塞，变更状态为\"todo\"

2. **删除任务数据**：
   - 使用 `replace_in_file` 从任务索引中移除任务记录
   - 更新统计数据：total -1, byStatus[状态] -1, byProject[项目ID] -1

3. **更新项目看板**：
   - 从对应状态列移除任务ID
   - 更新看板统计数据

4. **删除任务详情文件**（如果有）：
   - 使用 `terminal` 删除 `projects/{项目ID}/tasks/{任务ID}.md`

5. **清理任务引用**：
   - 删除负责人工作空间的任务引用文件

#### 阶段三：反馈用户

```
✅ 任务删除成功！

- 任务：UI设计-首页（task_20260212_002）

影响：

  - task_20260212_003（前端页面开发）的前置任务已移除

提示：
- 任务数据已永久删除
- 相关任务的依赖关系已自动更新
```

## 📋 工作准则

1. **数据访问规则**：
   - 只写 `data-layer/tasks/` 和 `projects/*/tasks/` 目录
   - 只读其他数据层目录
   - 通过接口调用其他Skill

2. **任务状态管理**：
   - 严格遵循状态转换规则
   - 自动检测和处理阻塞状态
   - 任务完成时自动解除依赖任务的阻塞

3. **依赖关系管理**：
   - 创建依赖前必须检查循环依赖
   - 依赖任务完成时自动通知相关任务
   - 删除任务时自动清理依赖关系

4. **里程碑管理**：
   - 自动计算里程碑进度
   - 自动检测延期情况
   - 任务状态变更时自动更新里程碑进度

5. **错误处理**：
   - 任务不存在时，展示任务列表
   - 循环依赖时，拒绝创建并提示
   - 删除有依赖的任务时，必须警告

6. **性能优化**：
   - 任务列表分页显示（每页20条）
   - 看板视图只加载必要数据
   - 统计数据增量更新
