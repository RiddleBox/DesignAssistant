# Skill 5：工作日志与分析

## 📋 技能概述

**技能名称**：工作日志与分析  
**技能编号**：SKILL-005  
**版本**：v1.0.0  
**依赖**：Skill 1（数字员工招募与管理）、Skill 2（团队项目协作）、Skill 3（任务与里程碑管理）  
**数据域**：`data-layer/logs/`（读写）、`employees/*/工作日志/`（读写）

**核心功能**：
1. 工作日志自动记录（时间、员工、任务、产出、耗时、问题）
2. 日志查询和筛选（按时间、员工、任务、项目）
3. 复盘报告生成（任务完成率、平均耗时、高频问题）
4. 可视化图表数据（时间分布、任务类型占比、效率趋势）
5. 效率异常检测和预警（重复性问题、效率下降）
6. 员工间消息与协作日志（P2功能）

**设计原则**：
- ✅ **领域无关**：通用日志格式，适用于任何领域
- ✅ **低耦合**：只写logs/目录，通过接口读取其他数据
- ✅ **零依赖**：使用系统内置命令和AI工具
- ✅ **自动化**：最小化人工干预，自动记录和分析

---

## 🎯 使用场景

### 场景1：自动记录工作日志
```
数字员工："我完成了API设计任务"
→ 系统自动记录日志（时间、员工、任务、耗时）
→ 保存到员工工作日志目录
→ 更新全局日志索引
```

### 场景2：查询工作日志
```
用户："查看后端工程师本周的工作日志"
→ 筛选后端工程师的日志
→ 按时间排序展示
→ 显示任务完成情况和耗时
```

### 场景3：生成复盘报告
```
用户："生成本周的项目复盘报告"
→ 分析本周所有工作日志
→ 统计任务完成率、平均耗时
→ 识别高频问题和效率趋势
→ 生成可视化报告
```

### 场景4：效率异常预警
```
系统自动检测：
→ 发现后端工程师连续3天效率下降
→ 分析原因：重复遇到"数据库连接超时"问题
→ 主动提醒用户并建议解决方案
```

### 场景5：员工间消息（P2功能）
```
前端工程师："向后端工程师发送消息：API接口文档在哪里？"
→ 创建消息记录
→ 在后端工程师的消息队列中添加待处理消息
→ 后端工程师响应后记录到协作日志
```

## 🔧 核心功能实现

### Step 0：环境检测与自动初始化

在开始任何操作前，必须先执行环境检测：

1. **检查日志数据层**：使用 `read_file` 检查 `data-layer/logs/index.json` 是否存在
2. **数据层不存在** → 执行零依赖自举初始化：
   - 告知用户：\"检测到日志数据层尚未初始化，正在自动创建...\"
   - 使用 `terminal` 创建 `data-layer/logs/` 目录
   - 使用 `terminal` 创建子目录：`reports/`、`analytics/`
   - 使用 `edit_file` 创建 `index.json` 文件（初始化为空日志列表）
   - 使用 `edit_file` 创建 `messages.json` 文件（初始化为空消息列表）

3. **检查员工工作日志目录**：
   - 如果用户操作涉及特定员工，检查该员工的 `工作日志/` 目录是否存在
   - 不存在 → 自动创建员工工作日志目录

### Step 2A：记录工作日志

#### 阶段一：收集日志信息

1. **检查用户是否提供了日志内容**
   - 已提供（任务名称、工作内容） → 继续
   - 未提供 → 询问：\"请提供工作日志内容\"

2. **询问日志基本信息**：
   - \"请提供任务名称（例如：API设计）\"
   - \"请选择任务状态（开始/进行中/已完成/阻塞）\"
   - \"请提供工作耗时（小时，可选）\"
   - \"请描述工作内容和产出物\"
   - \"是否遇到问题？（可选）\"

3. **检查用户是否指定了员工和项目**
   - 已指定 → 继续
   - 未指定 → 从当前上下文获取（如果有员工身份）或询问用户

#### 阶段二：生成日志ID和元数据

1. **生成日志ID**：
   - 格式：`log_YYYYMMDD_NNN`
   - 示例：`log_20260212_001`

2. **生成元数据**：
   - 时间戳：当前时间
   - 员工信息：从Skill 1获取
   - 项目信息：从Skill 2获取
   - 任务信息：从Skill 3获取（如果有）
   - 动作类型：task_started/task_in_progress/task_completed/task_blocked
   - 耗时：用户提供或自动计算
   - 产出物：用户提供
   - 问题：用户提供

#### 阶段三：保存日志文件

1. **确定保存路径**：
   - 格式：`employees/{员工ID}/工作日志/{日期}_{任务名称}.md`
   - 示例：`employees/emp_20260212_003/工作日志/2026-02-12_API设计.md`

2. **创建日志文件**：
   - 使用 `edit_file` 创建日志文件
   - 写入元数据头部（YAML格式）
   - 写入日志内容（Markdown格式）

3. **日志文件格式**：参考数据结构设计中的示例

#### 阶段四：更新全局日志索引

1. **更新日志索引**：
   - 使用 `replace_in_file` 在 `data-layer/logs/index.json` 中添加新日志记录
   - 更新统计数据：total +1, byEmployee[员工ID] +1, byProject[项目ID] +1, byAction[动作类型] +1

2. **如果任务状态变更**：
   - 调用Skill 3的接口更新任务状态

### Step 2B：查询工作日志

#### 阶段一：收集查询条件

1. **询问查询条件**：
   - \"请选择查询范围（员工/项目/全部，默认：全部）\"
   - \"请选择时间范围（今天/本周/本月/自定义，默认：本周）\"
   - \"请选择任务状态（开始/进行中/已完成/阻塞/全部，默认：全部）\"

#### 阶段二：执行查询

1. **读取日志索引**：使用 `read_file` 读取 `data-layer/logs/index.json`

2. **多维度筛选**：
   - 员工筛选：按employeeId筛选
   - 项目筛选：按projectId筛选
   - 时间筛选：按timestamp筛选
   - 状态筛选：按action筛选

3. **排序结果**：
   - 按时间倒序排序（最新的在前）

## 📋 工作准则

1. **数据访问规则**：
   - 只写 `data-layer/logs/` 和 `employees/*/工作日志/`、`employees/*/协作记录/` 目录
   - 只读其他数据层目录
   - 通过接口调用其他Skill

2. **自动化原则**：
   - 最小化人工干预，自动记录和分析
   - 在任务状态变更时自动记录日志
   - 定期自动生成复盘报告

3. **异常检测规则**：
   - 连续3天效率下降 → 触发预警
   - 重复性问题出现3次以上 → 触发预警
   - 任务阻塞超过24小时 → 触发预警

4. **消息处理规则**（P2功能）：
   - 消息未响应超过24小时 → 提醒用户
   - 消息响应后自动记录到协作日志
   - 协作日志定期归档

5. **错误处理**：
   - 日志记录失败时，提示用户并重试
   - 报告生成失败时，提示用户并回滚
   - 消息发送失败时，提示用户并重试

6. **性能优化**：
   - 日志列表分页显示（每页20条）
   - 报告生成异步执行（大数据量时）
   - 消息队列定期清理（已完成消息保留30天）
