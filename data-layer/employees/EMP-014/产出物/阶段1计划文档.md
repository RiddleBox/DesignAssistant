# 阶段1 工作计划文档（草案）

## 1. 目的与背景
基于阶段0成果（能力清单/交付清单/成功标准），阶段1的目标是：
- 输出可执行的阶段1工作计划（里程碑、任务、输出、验收）
- 形成可供阶段2构建使用的目标说明与工作框架

## 2. 阶段1目标
1. 明确阶段1应完成的核心输出与验收标准
2. 设计阶段1团队的工作里程碑与任务清单
3. 生成阶段2目标说明，包含关键工作、风险与依赖

## 2.1 与阶段0目标对齐（关键映射）
本节将阶段1计划与阶段0目标内容直接映射，确保输出与阶段0成功标准/交付清单一致。

### 2.1.1 阶段0能力清单 → 阶段1输出
- **AI Native 思考**：产出“工作分解与流程设计文档”（阶段1计划文档 + 协作流程图），体现需求分解与 Agent/工具链设计逻辑。
- **第一性原理分析**：在阶段1计划中增加“验收标准与指标定义”，并在阶段2目标说明中列出可验证的关键指标。
- **系统化设计**：在阶段1交付模板包中提供“任务-输出-验收”模板，确保各角色输出具备可复用流程。
- **协作能力**：通过协作流程图、责任地图与审查节点，实现多角色协作可追踪。
- **可复用方法论**：形成“交付模板+检查表+迭代计划”组合，作为后续复用资产。

## 2.2 为什么“系统设计/RAG 方案”既是阶段1又是阶段2的内容？
- **阶段1（计划设计）负责定义“应该是什么”**：
  - 确定需要什么样的 RAG/检索-生成架构（数据源、可信度、反馈循环、输出结构）。
  - 明确该系统必须满足的验收标准（例如：对于输入是否覆盖、答案可信度、结果可验证、可组合性）。
  - 形成“RAG 需求说明文档”（作为阶段2的输入），以便阶段2执行时不会迷失方向。
- **阶段2（团队构建/实现）负责“怎么做”**：
  - 选择具体的技术栈（如向量数据库、检索层、生成模型、工具链）、实现细节、Prompt 结构。
  - 生成用于执行的数字员工（Prompt 模板 + 验收样例 + 运行脚本）。

> 结论：阶段1要把“我们需要一个怎样的 RAG / 系统方案”说清楚，阶段2则负责把这个方案落实成可运行的“数字员工 + 工作流”。

## 2.3 阶段1当前要完成的“具体产出清单”
本阶段（阶段1，计划设计）应输出以下可交付文档/模板，作为阶段2执行的“可用输入”。

### 2.3.1 当前必须产出的文档/模板（“现在要做什么”）
- **阶段1计划文档（本文件）**：包含阶段1目标、里程碑、任务、输出、验收标准、阶段2目标说明。
- **阶段1协作流程图 + 责任地图**（EMP-015 输出）：必须回答“谁在什么时候交什么给谁，谁来验收”。
- **阶段1交付模板包 + 验收检查表**（EMP-016 输出）：用于规范后续角色输出格式，需包含“任务-输出-验收”模板。
- **阶段1计划验证报告**（EMP-017 输出）：基于验收标准评估上述文档/模板是否满足阶段0成功标准，并提出改进。
- **阶段1成果包说明 + 阶段2启动输入结构**（EMP-018 输出）：把上述输出整理成“阶段1成果包”，并定义阶段2的启动条件与输入文档结构。

### 2.3.2 当前阶段的“约束条件”
这些产出需符合以下约束：
- 产出必须是“可交付/可验证”的文档，而不是仅仅“结构化的框架”。
- 所有输出要能回答“如果交给下一阶段的执行团队（阶段2），他们需要做什么？哪里不用再问？”
- 阶段1输出的每个文档要能“直接拿来写入简历/作品集”，至少包含“案例结构”“关键指标”“执行结果可展示的格式”。

## 2.4 后续阶段（2/3/4）的关键交付与约束（“后面要做什么”）
### 2.4.1 阶段2（团队构建/实现）
- **关键交付**：数字员工角色（Prompt + 验收样例 + 运行说明）、可运行的 workflow 实现（可执行流程脚本/运行说明）。
- **约束**：阶段2的输出必须满足阶段1的“验收标准/模板”——换言之，阶段2产出的结果必须能被阶段1指定的验收报告直接评估。

### 2.4.2 阶段3（执行交付/案例输出）
- **关键交付**：一次完整的“真实评估案例输出包”包含结果报告、评分表、决策建议、执行日志。
- **约束**：案例输出必须可直接用于展示（简历/面试），并符合阶段0定义的“能力验证点”（AI 思考、第一性原理、系统化、协作、可复用）。

### 2.4.3 阶段4（产品化/复用）
- **关键交付**：可复用的 workflow 模板、执行手册、角色/Prompt 资产包。
- **约束**：任何人拿到该模板/手册，应能在 1-2 天内复制一次评估流程并产出相似质量的结果。

### 2.4.4 为“后续阶段设计的阶段1输出”举例（避免把责任模糊）
- 阶段1输出的“阶段2启动输入文档”应明确列出：
  - 需要跑的评估场景（目标/输入/期望输出）
  - 关键指标（要达到什么程度算成功）
  - 运行流程（谁投什么 Prompt，怎么校验结果）
  - 结果要如何格式化为“可展示案例”

### 2.4.5 你当前的角色与可做的事
如果你是“阶段1推进者/发起者”，你的主要工作是：
- 确保阶段1输出可以被直接“手递交给”阶段2执行。
- 用最直接的方式把“现在要做什么”“后续要做什么”写成文档（如本文件的这部分）。
- 针对执行阶段的需求，提前加上必要的“约束/验收标准/输出结构”，以防后续出现“执行团队不知道该怎么做”的情况。

> 这就是为什么你会感觉这是“一个空架子”：阶段1本应该把这些边界条件填进去，让它变成一个“可执行工作包”。

### 2.1.2 阶段0成功标准 → 阶段1交付物
- **可写入简历/面试的输出**：阶段1成果包需包含“可展示/可讲述的案例与指标”，作为阶段2输入。
  - 示例：补充“可展示产出”模板，如“评估案例摘要（目标/输入/输出/指标/结果）”，并定义“可讲述故事线”（问题→行动→结果）。
- **workflow 可跑通并输出评估结论**：阶段1需产出“阶段1计划文档 + 验收报告”，并通过阶段1验证官的评估。
- **方案可复用**：交付模板包必须具备“替换输入即可复用”的结构（模板中明确输入字段）。

### 2.1.3 阶段0交付清单 → 阶段1计划交付品
- 岗位能力映射报告 → 阶段1计划文档中“角色责任与能力映射”模块（见 4. 关键任务）
- 核心工作流结构图 → 阶段1协作流程图（由 EMP-015 输出）
- 真实评估案例输出 → 阶段1验证报告（EMP-017 输出）
- 可复用 workflow 模板 → 阶段1交付模板包（EMP-016 输出）
- 复盘报告 + 迭代计划 → 阶段1成果包中的“迭代记录”与“阶段2启动输入”（EMP-018 输出）

## 7. 阶段2目标说明（草案）

### 7.1 阶段2目标（推荐格式）
- **阶段2目标**：基于阶段1成果完成阶段2执行方案与初版成果，验证阶段2路径可行性并产出可交付样本。
- **阶段2关键输出**：阶段2执行方案文档、阶段2初版交付物样本、阶段2风险/缓解建议。

### 7.2 阶段2主要工作（推荐结构）
1. 根据阶段1成果（计划、模板、验收标准），明确阶段2要“做什么”：
   - 设定阶段2目标与成功标准（包括量化指标）。
   - 确定阶段2的关键交付物（例如：评估报告、执行方案、案例输出）。
2. 细化阶段2工作内容（里程碑、任务、时间表、角色）。
3. 设计阶段2执行流程与角色分工，并明确阶段2审查/验收节点。
4. 输出阶段2执行方案文档（含时间表、责任人、验收标准）。
5. 生成阶段2初版交付物样本（根据阶段1交付模板定义）。
6. 输出阶段2风险/依赖与缓解建议。

### 7.2.1 阶段2关键要点（需达成的关键指标/可展示内容）
- **可展示案例**：阶段2应至少输出一个可直接用于“简历/面试”场景的案例摘要（问题→行动→结果）。
- **关键指标**：阶段2输出需包含可量化的验证指标（例如：准确率、效率提升、交付完成率）。
- **可复用成果**：阶段2交付物应包括可直接复用的模板或流程组件（“输入-输出-验收”一体化文档）。
- **可讲述输出**：提供“成果故事线”（背景/目标→行动→结果→下一步）以支持展示与复盘。 

### 7.3 阶段2关键风险/依赖（建议补充）
- **依赖**：阶段1计划文档完整且包含可执行的阶段2目标说明。
- **依赖**：阶段1交付模板包已可直接用于阶段2任务。
- **风险**：若阶段1输出缺乏量化指标，阶段2执行可能失去方向。
- **风险**：若阶段1交付文档结构不一致，阶段2团队启动时需要额外对齐成本。

### 7.4 阶段2启动条件（建议）
- 阶段1成果包已形成并包含：
  - 阶段1计划文档（含阶段2目标说明）
  - 阶段1协作流程与责任地图
  - 阶段1交付模板包（含验收检查表）
  - 阶段1计划验证报告（含改进建议）
- 阶段1验收报告确认“可交付阶段2启动输入”。
