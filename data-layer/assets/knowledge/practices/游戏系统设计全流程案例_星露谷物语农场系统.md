# 游戏系统设计全流程案例：星露谷物语农场系统

## 📋 案例概述

本文以经典独立游戏《星露谷物语》的农场经营系统为例，完整模拟从0到1的游戏系统设计过程。通过"旁观者视角"观察设计师 Eric Barone（ConcernedApe）的设计思考，学习成熟设计师的思维模式和工作方法。

**适用场景**：
- 学习游戏系统设计方法论
- 理解模拟经营游戏的设计逻辑
- 掌握从概念到实现的完整流程
- 培养系统化设计思维

**设计时间线**：2012-2016（4年开发周期）

---

## 🎯 第一阶段：概念与核心循环设计

### Day 1：确定游戏愿景

**🧠 设计师内心独白：**

> "我想做一款像《牧场物语》那样温馨的农场游戏，但要加入更多RPG元素和自由度。玩家应该能感受到从零开始建立农场的成就感，同时有丰富的社交和探索内容。"

**✍️ 设计师的第一步行动：**

```
【设计文档 v0.1】

游戏核心概念：
- 类型：农场模拟经营 + RPG + 社交
- 核心体验：从荒废农场到繁荣庄园的成长过程
- 目标玩家：喜欢休闲经营、收集养成的玩家
- 游戏时长：100+ 小时

核心问题：
1. 玩家每天要做什么？
2. 为什么要持续玩下去？
3. 如何保持新鲜感？
```

**👁️ 旁观者观察：**
> 注意到设计师并没有直接开始设计具体系统，而是先明确了**游戏的核心定位**和**要解决的问题**。这是成熟设计师的标志——先想清楚"为什么做"，再考虑"怎么做"。

---

### Day 2-3：设计核心循环（Core Loop）

**🧠 设计师思考过程：**

> "农场游戏的核心是什么？是种植循环。但如果只有种植会很单调，我需要给玩家更多事情做..."

**✍️ 设计师画出第一版核心循环：**

```
【核心循环 v1.0】

早上起床
    ↓
浇水照料作物
    ↓
收获成熟作物
    ↓
出售获得金币
    ↓
购买新种子/工具
    ↓
种植新作物
    ↓
（循环）
```

**🤔 设计师自我质疑：**

> "等等，这个循环太简单了，玩家很快就会厌倦。我需要在循环中加入更多变化..."

**✍️ 设计师修改为多层循环：**

```
【核心循环 v2.0 - 多层次设计】

┌─────────────────────────────────────┐
│        日循环（每天）                 │
├─────────────────────────────────────┤
│ 1. 照料农场（浇水、收获）             │
│ 2. 探索/社交/钓鱼（自由选择）         │
│ 3. 出售产品                          │
│ 4. 规划明天                          │
└─────────────────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│        周循环（7天）                  │
├─────────────────────────────────────┤
│ • 作物生长周期（3-7天）               │
│ • 动物产出周期                        │
│ • NPC好感度积累                       │
└─────────────────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│        季循环（28天）                 │
├─────────────────────────────────────┤
│ • 季节更替（不同作物）                │
│ • 节日活动                           │
│ • 季度目标达成                        │
└─────────────────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│        年循环（4季）                  │
├─────────────────────────────────────┤
│ • 农场扩张                           │
│ • 社区中心修复                        │
│ • 结婚生子                           │
└─────────────────────────────────────┘
```

**👁️ 旁观者观察：**
> 设计师通过**迭代**改进了核心循环。第一版太单薄，第二版加入了**时间层次**的概念——短期、中期、长期目标交织，这就是"成熟的节奏设计能力"的体现。

---

### Day 4-5：识别核心系统模块

**🧠 设计师思考：**

> "现在我知道玩家要做什么了，接下来要拆解出需要哪些系统来支撑这些玩法..."

**✍️ 设计师列出系统清单：**

```
【系统模块清单 v1.0】

必需系统（MVP）：
✅ 种植系统 - 核心玩法
✅ 时间系统 - 驱动循环
✅ 资源系统 - 经济基础
✅ 工具系统 - 玩家能力
✅ 地图系统 - 空间载体

扩展系统（增加深度）：
⭐ 动物养殖系统
⭐ 钓鱼系统
⭐ 采矿系统
⭐ 社交系统
⭐ 烹饪系统

体验系统（增加粘性）：
🎯 任务系统
🎯 成就系统
🎯 收集系统
```

**🤔 设计师的优先级判断：**

> "我不能一次做完所有系统，那样会永远做不完。我要先做出最小可玩版本（MVP），然后逐步扩展..."

**✍️ 设计师制定开发优先级：**

```
【开发阶段规划】

Phase 1（3个月）- 核心可玩：
- 种植系统
- 时间系统
- 基础资源系统
- 简单工具系统
→ 目标：能种菜、能收获、能卖钱

Phase 2（6个月）- 丰富内容：
- 动物养殖
- 钓鱼
- 采矿
→ 目标：玩法多样化

Phase 3（12个月）- 深度体验：
- 社交系统
- 任务系统
- 节日活动
→ 目标：长期可玩性
```

**👁️ 旁观者观察：**
> 这里体现了**系统架构能力**的关键——**分层设计**和**优先级管理**。设计师没有试图一次性设计完美系统，而是采用**迭代式开发**，先保证核心循环可玩，再逐步添加深度。

---

## 🔧 第二阶段：核心系统详细设计

### Week 2：设计种植系统

**🧠 设计师开始深入设计第一个核心系统：**

> "种植系统是整个游戏的基础，我需要仔细设计它的每个环节..."

#### **Step 1：定义系统边界**

**✍️ 设计师写下系统职责：**

```
【种植系统职责定义】

负责：
✅ 作物的生长逻辑
✅ 作物的状态管理（种子→幼苗→成熟）
✅ 浇水机制
✅ 收获机制
✅ 季节限制

不负责：
❌ 金钱计算（由经济系统负责）
❌ 工具耐久（由工具系统负责）
❌ 玩家体力（由角色系统负责）
```

**👁️ 旁观者观察：**
> 设计师明确了**系统边界**，这是避免系统间耦合的关键。注意他不仅定义了"做什么"，还定义了"不做什么"。

---

#### **Step 2：设计数据结构**

**🧠 设计师思考：**

> "我需要什么数据来描述一个作物？"

**✍️ 设计师设计作物数据结构：**

```javascript
【作物数据结构 v1.0】

Crop {
    id: string,              // 作物ID
    name: string,            // 作物名称
    growthStages: number,    // 生长阶段数
    daysPerStage: number[],  // 每阶段天数
    seasons: string[],       // 可种植季节
    sellPrice: number,       // 售价
    seedPrice: number,       // 种子价格
}

// 示例：防风草
{
    id: "parsnip",
    name: "防风草",
    growthStages: 4,
    daysPerStage: [1, 1, 1, 1],  // 总共4天成熟
    seasons: ["spring"],
    sellPrice: 35,
    seedPrice: 20,
}
```

**🤔 设计师发现问题：**

> "等等，这样设计有个问题——如果我想加入'连续收获'的作物（像草莓），这个结构就不够用了..."

**✍️ 设计师迭代数据结构：**

```javascript
【作物数据结构 v2.0 - 支持连续收获】

Crop {
    id: string,
    name: string,
    growthStages: number,
    daysPerStage: number[],
    seasons: string[],
    sellPrice: number,
    seedPrice: number,
    
    // 新增字段
    regrowthDays: number | null,  // 重新生长天数（null表示一次性）
    harvestAmount: {min: number, max: number},  // 收获数量范围
    isGiantCropPossible: boolean,  // 能否长成巨型作物
}

// 示例：草莓（连续收获）
{
    id: "strawberry",
    name: "草莓",
    growthStages: 5,
    daysPerStage: [2, 2, 2, 2, 2],  // 8天首次成熟
    regrowthDays: 4,  // 之后每4天收获一次
    seasons: ["spring"],
    sellPrice: 120,
    seedPrice: 100,
    harvestAmount: {min: 1, max: 1},
    isGiantCropPossible: false,
}
```

**👁️ 旁观者观察：**
> 设计师展现了**可扩展性思维**——他没有满足于当前需求，而是思考"未来可能需要什么功能"。这种前瞻性设计避免了后期大规模重构。

---

#### **Step 3：设计系统流程**

**🧠 设计师绘制种植流程图：**

```
【种植系统流程图】

玩家使用锄头 → 耕地
    ↓
玩家选择种子 → 种植
    ↓
作物进入生长状态
    ↓
每日更新循环：
    ├─ 检查是否浇水 → 是：生长进度+1
    │                  否：生长进度不变
    ├─ 检查生长进度 → 达到阶段：切换到下一阶段
    ├─ 检查是否成熟 → 是：显示可收获状态
    └─ 检查季节变化 → 跨季未成熟：作物枯萎
    ↓
玩家收获 → 获得产物 → 清除作物
    ↓
（如果是连续收获作物）→ 重置生长进度 → 继续循环
```

**🤔 设计师思考边界情况：**

> "我需要考虑一些特殊情况..."

**✍️ 设计师列出边界情况处理：**

```
【边界情况处理】

1. 跨季节问题：
   - 春季作物在夏季第一天会枯萎
   - 但温室内的作物不受季节影响
   → 需要标记"是否在温室内"

2. 浇水问题：
   - 下雨天自动浇水
   - 洒水器自动浇水
   → 需要检查天气和洒水器覆盖范围

3. 品质问题：
   - 作物品质受什么影响？
   → 决定：受土壤品质、肥料影响
   → 需要添加品质计算系统

4. 特殊作物：
   - 巨型作物（3x3成熟作物有概率合并）
   - 需要特殊检测逻辑
```

**👁️ 旁观者观察：**
> 这是**系统设计成熟度**的重要体现——考虑边界情况和异常处理。新手设计师往往只考虑"正常流程"，而忽略了各种特殊情况。

---

#### **Step 4：设计系统接口**

**🧠 设计师思考：**

> "种植系统需要和其他系统交互，我需要定义清晰的接口..."

**✍️ 设计师定义系统接口：**

```javascript
【种植系统对外接口】

// 输入接口（其他系统调用种植系统）
PlantingSystem {
    // 种植作物
    plantSeed(position, seedId, isFertilized) → boolean
    
    // 浇水
    waterCrop(position) → boolean
    
    // 收获
    harvestCrop(position) → Item[]
    
    // 查询状态
    getCropInfo(position) → CropInfo
    
    // 每日更新（由时间系统调用）
    onDayUpdate()
    
    // 季节变化（由时间系统调用）
    onSeasonChange(newSeason)
}

// 输出接口（种植系统需要其他系统提供）
需要的外部接口：
- TimeSystem.getCurrentDay() → 获取当前日期
- TimeSystem.getCurrentSeason() → 获取当前季节
- WeatherSystem.isRaining() → 是否下雨
- InventorySystem.addItem(item) → 添加物品到背包
- EconomySystem.calculatePrice(item, quality) → 计算价格
```

**👁️ 旁观者观察：**
> 设计师定义了**清晰的系统接口**，这是**模块化设计**的核心。通过接口隔离，种植系统可以独立开发和测试，不需要依赖其他系统的具体实现。

---

### Week 3：设计经济系统

**🧠 设计师转向经济系统设计：**

> "现在玩家能种菜了，但经济系统要怎么设计？我需要让玩家感受到成长，但又不能太容易..."

#### **Step 1：定义经济目标**

**✍️ 设计师写下经济系统的设计目标：**

```
【经济系统设计目标】

1. 玩家成长曲线：
   - 第1年：艰难起步，精打细算
   - 第2年：稳定发展，开始扩张
   - 第3年：富裕阶段，追求优化

2. 金钱的价值感：
   - 前期：每一笔钱都很重要
   - 中期：开始有余裕做选择
   - 后期：金钱不再是限制，追求效率

3. 避免的问题：
   ❌ 通货膨胀（后期金钱贬值）
   ❌ 单一最优策略（只种最赚钱的作物）
   ❌ 过度重复劳动（无脑刷钱）
```

**👁️ 旁观者观察：**
> 设计师在设计具体数值前，先明确了**经济系统的体验目标**。这是"目标驱动设计"的体现——先想清楚要给玩家什么感受，再设计实现方式。

---

#### **Step 2：设计资源流动模型**

**🧠 设计师绘制资源流动图：**

```
【资源流动模型】

┌─────────────────────────────────────┐
│          资源输入（获得金钱）          │
├─────────────────────────────────────┤
│ • 出售作物（主要收入）                │
│ • 出售动物产品（稳定收入）             │
│ • 出售鱼类（补充收入）                │
│ • 出售矿物（爆发收入）                │
│ • 完成任务（少量收入）                │
└─────────────────────────────────────┘
            ↓
      【金钱池】
            ↓
┌─────────────────────────────────────┐
│          资源输出（花费金钱）          │
├─────────────────────────────────────┤
│ • 购买种子（必要支出）                │
│ • 购买动物（投资）                    │
│ • 升级工具（提升效率）                │
│ • 建造建筑（扩张）                    │
│ • 购买装饰（可选）                    │
└─────────────────────────────────────┘
```

**🤔 设计师思考平衡性：**

> "我需要确保玩家的收入和支出是平衡的，不能让玩家太穷也不能太富..."

**✍️ 设计师设计第一年的经济曲线：**

```
【第一年经济曲线设计】

春季（起步期）：
- 初始资金：500金
- 预期收入：2000-3000金/月
- 主要支出：种子（1000金）、工具升级（2000金）
- 目标：让玩家感受到"紧张但有希望"

夏季（发展期）：
- 预期收入：5000-8000金/月
- 主要支出：动物（4000金）、建筑（10000金）
- 目标：玩家开始有余裕做选择

秋季（收获期）：
- 预期收入：10000-15000金/月
- 主要支出：为冬季做准备
- 目标：玩家感受到成长

冬季（调整期）：
- 预期收入：3000-5000金/月（无法种植）
- 主要支出：社交、探索
- 目标：让玩家期待下一年
```

**👁️ 旁观者观察：**
> 设计师设计了**动态的经济曲线**，而不是线性增长。注意冬季的收入下降是**刻意设计**的——这创造了节奏变化，避免玩家疲劳。

---

#### **Step 3：设计作物定价策略**

**🧠 设计师开始设计具体数值：**

> "每种作物应该卖多少钱？我需要一个定价公式..."

**✍️ 设计师设计定价公式：**

```
【作物定价公式 v1.0】

基础公式：
售价 = 种子成本 × 利润率 × 时间系数

其中：
- 利润率：1.5-2.5（保证有利可图）
- 时间系数：生长天数越长，单日收益越高

示例计算：

防风草：
- 种子成本：20金
- 生长天数：4天
- 售价 = 20 × 1.75 × 1.0 = 35金
- 净利润 = 35 - 20 = 15金
- 日均收益 = 15 / 4 = 3.75金/天

花椰菜：
- 种子成本：80金
- 生长天数：12天
- 售价 = 80 × 1.75 × 1.2 = 168金
- 净利润 = 168 - 80 = 88金
- 日均收益 = 88 / 12 = 7.33金/天
```

**🤔 设计师发现问题：**

> "这样计算的话，长周期作物的日均收益更高，玩家会只种长周期作物。我需要加入风险因素..."

**✍️ 设计师调整设计：**

```
【作物定价公式 v2.0 - 加入平衡因素】

考虑因素：
1. 时间风险：长周期作物占用土地时间长
2. 资金压力：长周期作物回本慢
3. 灵活性：短周期作物可以快速调整策略

调整后的设计：
- 短周期作物：低利润，但灵活，适合新手
- 中周期作物：平衡收益，主流选择
- 长周期作物：高收益，但有风险

特殊设计：
- 连续收获作物：首次投入高，但长期收益好
  → 鼓励玩家做长期规划
  
示例：草莓
- 种子成本：100金
- 首次成熟：8天
- 之后每4天收获一次
- 单次售价：120金
- 春季28天，可收获约5次
- 总收益：120 × 5 = 600金
- 净利润：500金
- 日均收益：500 / 28 = 17.8金/天（非常高！）
- 但需要前期投入和耐心等待
```

**👁️ 旁观者观察：**
> 设计师展现了**平衡性设计能力**——他不是简单地设定数值，而是考虑了**多维度的权衡**（收益vs风险、短期vs长期）。这创造了**有意义的选择**，而不是单一最优解。

### Week 4：设计时间与节奏系统

**🧠 设计师思考：**

> "时间系统是驱动整个游戏的引擎，我需要仔细设计时间的流逝速度和节奏..."

#### **Step 1：设计时间刻度**

**✍️ 设计师设计时间结构：**

```
【时间系统结构】

最小单位：10分钟（游戏内）
- 1小时 = 6个时间单位
- 1天 = 24小时 = 144个时间单位
- 实际游戏时间：1天 ≈ 12-15分钟

时间段划分：
06:00 - 早晨（起床）
08:00 - 商店开门
12:00 - 中午
18:00 - 商店关门
20:00 - 傍晚（NPC回家）
24:00 - 深夜（强制睡觉）
02:00 - 过晚不睡会昏倒
```

**🤔 设计师思考节奏：**

> "一天12分钟够吗？玩家能完成多少事情？"

**✍️ 设计师进行时间预算：**

```
【玩家一天时间预算】

假设玩家有20块农田：
- 浇水：20 × 5秒 = 100秒 ≈ 1.5分钟
- 收获：10 × 3秒 = 30秒
- 种植：10 × 3秒 = 30秒
- 合计：2分钟

剩余时间（10分钟）可以：
- 社交：拜访2-3个NPC（3分钟）
- 探索：采集/钓鱼/采矿（5分钟）
- 购物：买种子/卖东西（2分钟）

结论：12-15分钟一天是合理的
- 不会太赶（有选择空间）
- 不会太闲（保持紧凑感）
```

**👁️ 旁观者观察：**
> 设计师进行了**量化分析**——他没有凭感觉设定时间，而是计算了玩家的实际行动时间。这是**数据驱动设计**的体现。

---

#### **Step 2：设计季节系统**

**🧠 设计师设计季节循环：**

```
【季节系统设计】

每季28天（4周）
- 为什么是28天？
  → 4的倍数，便于设计周循环
  → 足够长，让玩家体验完整的种植周期
  → 不会太长，保持新鲜感

季节特性：
春季：
- 主题：新生、希望
- 作物：基础作物（防风草、花椰菜）
- 节日：复活节（13日）、花舞节（24日）

夏季：
- 主题：繁荣、活力
- 作物：高价值作物（蓝莓、甜瓜）
- 节日：夏日集市（11日）、月光水母节（28日）

秋季：
- 主题：收获、准备
- 作物：特殊作物（南瓜、蔓越莓）
- 节日：秋季集市（16日）、万灵节（27日）

冬季：
- 主题：休整、社交
- 作物：无法种植（除非有温室）
- 节日：冬季集市（8日）、冬星节（25日）
- 特殊玩法：冰钓、采集、社交
```

**🤔 设计师思考冬季设计：**

> "冬季不能种植，玩家会不会觉得无聊？我需要给冬季特殊的意义..."

**✍️ 设计师为冬季设计特殊内容：**

```
【冬季特殊设计】

目的：让冬季成为"调整期"而非"无聊期"

冬季可做的事：
1. 社交重点：
   - NPC在冬季更容易找到
   - 冬星节送礼增加好感度
   - 适合推进恋爱线

2. 探索重点：
   - 矿洞探索（不受季节影响）
   - 冰钓（冬季特有鱼类）
   - 采集冬季特有物品

3. 规划重点：
   - 整理仓库
   - 规划下一年农场布局
   - 升级工具和建筑

4. 经济设计：
   - 冬季作物（冬季种子）需要采集制作
   - 鼓励玩家探索而非单纯种植
```

**👁️ 旁观者观察：**
> 设计师将**限制转化为特色**——冬季不能种植本来是个问题，但通过设计特殊内容，反而创造了**节奏变化**。这体现了"限制激发创意"的设计哲学。

## 🔗 第三阶段：系统整合与平衡

### Month 2：系统联动设计

**🧠 设计师思考：**

> "现在各个系统都设计好了，但它们是孤立的。我需要设计系统间的联动，让游戏产生化学反应..."

#### **设计系统联动矩阵**

**✍️ 设计师绘制系统联动图：**

```
【系统联动矩阵】

种植系统 ←→ 经济系统
- 作物售价受品质影响
- 肥料需要购买

种植系统 ←→ 社交系统
- 送作物给NPC增加好感
- NPC会送种子作为礼物

种植系统 ←→ 烹饪系统
- 作物可以作为食材
- 烹饪品售价更高

动物系统 ←→ 社交系统
- 动物产品是NPC喜欢的礼物
- 好感度高的NPC会送动物

采矿系统 ←→ 工具系统
- 矿物用于升级工具
- 更好的工具提升采矿效率

所有系统 ←→ 时间系统
- 时间驱动所有系统更新
- 季节影响可用内容
```

**🤔 设计师发现设计机会：**

> "如果我让系统间产生更深的联动，玩家会有更多策略选择..."

**✍️ 设计师设计深度联动：**

```
【深度联动设计案例：品质系统】

品质影响链：
土壤品质 → 作物品质 → 售价/好感度 → 收益

具体设计：
1. 使用肥料提升土壤品质
2. 高品质土壤产出高品质作物
3. 高品质作物：
   - 售价更高（经济收益）
   - 送礼效果更好（社交收益）
   - 烹饪加成更高（生存收益）

这创造了投资回报循环：
花钱买肥料 → 产出高品质作物 → 获得更多收益 → 有钱买更多肥料
```

**👁️ 旁观者观察：**
> 设计师通过**系统联动**创造了**涌现玩法**——单个系统很简单，但组合起来产生了复杂的策略空间。这是"系统设计"的精髓——**整体大于部分之和**。

### Month 3-4：数值平衡与迭代

**🧠 设计师进入测试阶段：**

> "理论设计完成了，现在需要实际测试和调整数值..."

#### **测试与调整循环**

**✍️ 设计师的测试日志：**

```
【测试日志 - 第1周】

问题1：玩家第一年春季就破产了
- 原因：种子太贵，收益太低
- 解决：降低初始种子价格20%

问题2：玩家只种防风草（最高效）
- 原因：其他作物性价比太低
- 解决：提升长周期作物收益，增加多样性奖励

问题3：冬季太无聊
- 原因：可做的事太少
- 解决：增加冬季特有内容（冰钓、冬季集市）

问题4：时间太紧张
- 原因：农田太多，浇水占用所有时间
- 解决：增加洒水器（自动浇水工具）
```

**✍️ 设计师调整后的数值表：**

```
【作物数值表 v3.0 - 平衡后】

作物名称 | 成本 | 天数 | 售价 | 日均收益 | 特点
--------|------|------|------|----------|------
防风草   | 20   | 4    | 35   | 3.75     | 新手友好
花椰菜   | 80   | 12   | 175  | 7.92     | 高收益，有风险
土豆     | 50   | 6    | 80   | 5.00     | 有概率多产
草莓     | 100  | 8+4  | 120  | 17.8     | 连续收获，长期投资

设计意图：
- 防风草：保底选择，适合新手
- 花椰菜：高风险高回报，适合赌一把
- 土豆：稳定收益，有惊喜
- 草莓：需要规划，长期收益最高
→ 每种作物都有存在价值，没有绝对最优解
```

**👁️ 旁观者观察：**
> 设计师通过**迭代测试**不断调整数值，这是**数据驱动设计**的实践。注意他不是一次性设计完美数值，而是通过**测试-反馈-调整**的循环逐步优化。

## 📚 总结：设计师的思维模式

通过这个完整的设计过程模拟，我们可以总结出**成熟设计师的思维模式**：

### 🎯 **1. 自顶向下的设计流程**

```
愿景/目标
    ↓
核心循环
    ↓
系统模块
    ↓
详细设计
    ↓
数值平衡
    ↓
扩展内容
```

### 🔄 **2. 迭代式设计思维**

- 不追求一次性完美
- 先做MVP，再逐步扩展
- 通过测试不断调整

### 🎨 **3. 多维度平衡能力**

- 收益 vs 风险
- 短期 vs 长期
- 简单 vs 深度
- 自由 vs 引导

### 🔗 **4. 系统联动思维**

- 单个系统简单清晰
- 系统间产生化学反应
- 创造涌现玩法

### 📊 **5. 数据驱动决策**

- 量化分析玩家行为
- 用数据验证设计假设
- 持续监控和调整

---

## 💡 **关键启示**

1. **先想清楚"为什么"，再考虑"怎么做"**
   - 每个设计决策都有明确目的

2. **限制是创意的源泉**
   - 冬季不能种植 → 特殊玩法
   - 时间有限 → 策略选择

3. **系统边界清晰，接口简洁**
   - 避免系统间耦合
   - 便于独立开发和测试

4. **平衡不是数值相等，而是选择有意义**
   - 每个选项都有存在价值
   - 没有绝对最优解

5. **迭代优于完美**
   - 先做出来，再优化
   - 测试是设计的一部分
