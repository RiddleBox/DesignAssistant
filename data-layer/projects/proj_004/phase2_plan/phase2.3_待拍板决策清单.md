## 一、为什么要先补这份清单

在 `Phase 2.3` 当前阶段，我们已经完成了三类前置工作：

- `2.3` 的第一性原理与岗位本质收口；
- `2.3 MVP` 范围与后续迭代边界的初步收口；
- `2.3` 的治理入口与协作顺序说明。

这意味着当前已经不再适合继续泛讨论“`2.3` 理论上还能做什么”，而是要进一步进入：

> **把真正会影响启动方向、设计边界、协作方式、与 `2.2 / 2.4 / 2.5` 联动方式的关键问题，显式整理成待拍板决策清单。**

如果缺少这份文档，`2.3` 后续非常容易出现以下问题：

- 另一端接手时仍按旧理解，把 `2.3` 做成“大而全计划书生成器”；
- 设计阶段一边推进，一边继续修改模块本体，导致反复返工；
- 本来应该用户明确拍板的问题，被执行端在实现中默认决定；
- 行动姿态、分阶段承诺、`Go/No-Go` 与退出条件这些最关键的对象骨架，还没定清楚，就先被长模板、细预算和复杂编排分散注意力。

因此，这份清单的目的不是把所有问题一次性讨论完，而是先固定：

- 哪些问题**必须现在定**；
- 哪些问题**本周最好定**；
- 哪些问题**可以条件性后置**；
- 哪些问题**在未拍板前不得被执行端自行扩大解释**。

## 三、当前拍板的基本原则

基于现有定位文档、范围文档与治理入口文档，`2.3` 当前的拍板应遵循以下原则：

### 3.1 先拍板模块本体与主产物，再拍板增强项

当前最重要的不是先决定：

- 资源模型要不要做得更细；
- 报告最后写几章；
- 是否一开始就把多角色机制工程化为复杂多 Agent 编排。

当前最重要的是先决定：

- `2.3` 到底是不是行动设计层 / 资源承诺层；
- `2.3` 当前主产物到底是什么；
- `2.3` 当前最小闭环是什么；
- `2.3` 与 `2.2 / 2.4 / 2.5` 的职责边界在哪里。

### 3.2 先拍板 `MVP` 必做项，再拍板 `P1 / P2`

当前阶段必须严格区分：

- 没有它，`2.3 MVP` 就无法成立的内容；
- 有它更好，但不应作为启动前置的内容。

### 3.3 拍板项必须能约束后续设计与实现

每个待拍板项都不应只是抽象讨论题，而应能直接影响：

- 另一端后续如何出 `phase2.3_roles.md`；
- `phase2.3_设计方案.md` 中哪些内容可以先做；
- 哪些内容必须暂缓；
- 未来验收时用什么口径判断是否偏航。

### 3.4 旧目标中与新定位冲突的内容必须显式收口

`phase2.3_目标说明.md` 中保留了较强的旧实现导向，例如：

- 短 / 中 / 长期路径直接作为中心；
- 人力 / 资金 / 时间估算直接作为核心能力；
- `9` 章节决策建议模板作为验收口径；
- 代码模块拆分先于对象边界冻结。

这些内容不是完全无价值，但当前必须明确：

- 哪些继续保留；
- 哪些降级为派生视图或增强项；
- 哪些暂不作为当前 `MVP` 硬范围。

### 3.5 未拍板内容不得被默认扩大解释

如果某个问题仍未拍板，则默认：

- 不进入当前强制范围；
- 不由执行端自行扩展解释；
- 不在设计文档中被写成既定结论。

## 四、当前拍板的上位判断

在进入逐项清单之前，先把 `2.3` 当前阶段最关键的上位判断写清楚：

> **`2.3` 当前的核心任务，是以“结构化行动决策对象 + 最小行动设计闭环”为中心完成启动前收口，而不是一次性实现完整计划书、完整资源系统或复杂编排系统。**

因此，当前所有拍板项都应围绕这条主线服务，而不是偏离这条主线。

| **输入 / 输出契约与 Schema 的推进时机** | A. 在工作流与角色依据明确后收口；B. 现在先把 Schema 全写死；C. 一直拖到实现后再补 | **A** | 若过早写死，后续角色与流程一变会大量返工；若过晚补齐，又会影响联调稳定性 | ✅ 已拍板：采用 **A** |

**已拍板口径**：

## 八、当前不建议默认推进的事项

为了防止另一端在未拍板前直接沿旧思路开工，建议明确写出当前不建议默认推进的方向。

### 8.1 不建议默认把 `2.3` 直接做成完整计划书生成模块

原因：
- 这会削弱结构化行动对象作为主产物的地位；
- 会让 `2.3` 更像展示层而不是承诺层；
- 会削弱后续 `2.5` 对真实行动对象的验证与归因价值。

### 8.2 不建议默认把完整资源系统作为当前设计起点

原因：
- 会把注意力从行动骨架转移到细节估算；
- 会显著增加实现复杂度和归因难度；
- 会让 `MVP` 在启动前就膨胀。

### 8.3 不建议默认把 `2.4` 当成现成答案拼装层

原因：
- `2.4` 当前更适合作为 context packet / 决策支撑包；
- 如果过早把 `2.4` 当成答案层，`2.3` 就会失去真正的行动设计职责。

### 8.4 不建议默认让 `2.3` 越级覆盖 `2.5`

原因：
- 会模糊整个 Phase 2 链路的分工；
- 会导致“行动设计”和“现实验证 / 复盘归因”混在一起，后续模块边界难以维护。

### 8.5 不建议默认把多 Agent 编排作为当前实现前提

原因：
- 会把注意力从对象定义和承诺逻辑转移到编排细节；
- 当前阶段更适合先通过角色面具保障职责覆盖，而不是直接进入复杂工程化编排。

### 8.6 不建议默认让多角色机制在 `2.3` 当前阶段的定位

原因：
- 会把注意力从对象定义和承诺逻辑转移到编排细节；
- 当前阶段更适合先通过角色面具保障职责覆盖，而不是直接进入复杂工程化编排。
