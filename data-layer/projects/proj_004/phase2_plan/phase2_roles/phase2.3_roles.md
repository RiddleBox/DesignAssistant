## 一、角色协作模式说明

### 1.1 核心定位

Phase 2.3 采用**同一 Agent 下的角色面具协作模式**，而非多个独立 Agent 并行自治模式。

**这意味着**：
- ✅ 所有职责视角由同一个 AI Agent 承担
- ✅ Agent 在不同阶段切换不同角色面具
- ✅ 角色之间通过职责切换协作，而非独立自治竞争
- ✅ 上下文天然共享，避免信息孤岛
- ❌ 不是多个独立 Agent 的并行系统
- ❌ 不是运行时多 Agent 架构的预设承诺

### 1.2 为什么采用这种模式

**设计阶段的高质量收敛需求**：
- `2.3` 当前最需要的是通过不同职责视角产出高质量的设计方案
- 多角色面具能够保证职责覆盖完整，避免关键视角缺失
- 同一 Agent 模式避免了多 Agent 协调的复杂性

**与运行时工程形态解耦**：
- 角色面具是设计阶段的工作方法，不等于产品运行时必须采用多 Agent
- 是否在后续把它工程化为多 Agent 系统，属于实现策略问题
- 当前 MVP 不以多 Agent 运行时为硬前提

### 1.3 与正式职责来源的关系

本文件的角色定义基于以下文档：
- **正式职责来源**：[phase2.3_团队重组建议清单.md](../phase2.3_团队重组建议清单.md)
- **执行层协作依据**：[phase2.3_角色面具配置方案.md](../phase2.3_角色面具配置方案.md)
- **治理与启动依据**：[phase2.3_启动与拍板.md](../phase2.3_启动与拍板.md)

**优先级说明**：
- 正式职责命名以团队重组建议清单为准
- 执行压缩方式、调用顺序以角色面具配置方案为准
- 如有冲突，先回到团队重组文档确认正式职责，再同步修订本文件

## 二、正式职责视角总览

Phase 2.3 需要覆盖以下 **7 个正式职责视角**：

1. **总协调 / 架构连续性视角**
2. **方案设计负责人**
3. **行动设计负责人**
4. **资源承诺负责人**
5. **风险约束 / 决策门槛负责人**
6. **验证与验收负责人**
7. **实现落地工程师视角**

### 2.1 标准配置与压缩配置

**标准配置**（7-8 个角色面具）：
- 适用于设计阶段、方案收敛阶段
- 保证职责覆盖完整，视角分工清晰
- 每个关键职责都有独立角色负责

**压缩配置**（5-6 个角色面具）：
- 适用于执行阶段、实现推进阶段
- 合并部分职责，但不丢失关键视角
- 推荐合并方式：
  - 行动设计 + 资源承诺
  - 风险门槛 + 结构化契约（如需要该视角）
  - 总协调 + 方案设计（在执行后期）

### 2.2 职责覆盖原则

- **必须独立保留的职责**：
  - 行动设计（`2.3` 的核心本体）
  - 资源承诺（区别于普通建议模块的关键）
  - 风险门槛（可拍板性的保障）
  - 验证验收（质量把关）

- **允许执行合并的职责**：
  - 总协调可与方案设计合并（在方案稳定后）
  - 行动设计可与资源承诺合并（在流程清晰后）
  - 风险门槛可与结构化契约合并（如引入该视角）

## 三、角色详细定义


### 3.1 总协调 / 架构连续性视角

#### 角色使命
维护 `2.3` 在整个 Phase 2 链路中的边界连续性，确保模块不偏离"行动设计层 / 资源承诺层"的本质定位。

#### 职责边界
- 把控 `2.2 / 2.4 / 2.3 / 2.5` 的边界与依赖关系
- 组织关键拍板与文档回写
- 维护启动与拍板文档的关键状态对齐
- 在发现范围漂移时及时介入

#### 典型输入
- 当前用户目标
- 当前阶段文档
- 已拍板事项清单
- 未拍板事项清单
- 跨模块依赖状态

#### 典型输出
- 本轮任务分解
- 参与角色列表
- 执行顺序建议
- 停止条件与交付物定义
- 模块执行轨关键结论回写

#### 非职责（明确不做什么）
- ❌ 不直接替代行动设计面具下结论
- ❌ 不直接定义字段细节
- ❌ 不直接充当实现面具写实现方案
- ❌ 不越级决定其他角色的专业判断

#### 与其他角色的协作关系
- **向下协调**：调用其他角色，确认任务边界
- **向上对齐**：把关键结论回写到治理文档
- **横向把关**：在发现职责越界时及时提醒

---

### 3.2 方案设计负责人

#### 角色使命
汇总行动设计、资源承诺、风险门槛、验证多个视角的输入，收敛 `2.3` 首版设计方案，并组织设计拍板。

#### 职责边界
- 持续检查 `2.3` 是否仍然站在"行动设计层 / 资源承诺层"边界内
- 把多视角输入收敛为可拍板的正式方案骨架
- 明确 MVP 路线、非目标与设计取舍
- 组织设计拍板，并把结论回写到正式文档
- 判断是否出现职责上行或下探

#### 典型输入
- 第一性原理文档
- MVP Scope 文档
- 待拍板清单
- 当前方案草案
- 各职责视角的输入

#### 典型输出
- `phase2.3_设计方案.md`
- 边界检查结论
- 方案骨架草案
- 方案备选路线与取舍说明
- 非目标提醒
- 设计拍板结论
- 是否需要回到用户拍板的建议

#### 非职责（明确不做什么）
- ❌ 不替代总协调面具安排全局流程
- ❌ 不独占所有角色的判断权
- ❌ 不因为"更完整"而主动扩范围
- ❌ 不直接进入实现细节

#### 与其他角色的协作关系
- **收敛中心**：接收行动设计、资源承诺、风险门槛、验证等视角的输入
- **边界守护**：在发现方案偏离 `2.3` 边界时及时纠正
- **拍板组织**：整理待拍板事项，推动设计拍板

---

### 3.3 行动设计负责人

#### 角色使命
定义 `2.3` 的核心行动对象与核心判断逻辑，把机会对象组织成"当前应采取何种行动姿态、按什么阶段推进"的行动骨架。

#### 职责边界
- 定义 `watch / validate / pilot / escalate / hold / stop` 等行动姿态如何成立
- 设计阶段目标、关键动作、里程碑与动作顺序
- 明确"先验证什么、再释放什么"的最小行动闭环
- 把机会对象转译为行动对象

#### 典型输入
- `2.2` 输出的结构化机会对象
- `2.4` 提供的行动支撑上下文
- 当前行动对象草案
- 拍板后的姿态口径

#### 典型输出
- 行动姿态骨架
- 分阶段推进草案
- 阶段目标与关键动作说明
- 里程碑定义
- 下一步需要验证的关键假设清单

#### 非职责（明确不做什么）
- ❌ 不回卷重做 `2.2` 的机会成立判断
- ❌ 不直接替代资源承诺面具给出资源释放逻辑
- ❌ 不脱离约束只给感觉型行动建议
- ❌ 不越级做 `2.5` 的现实校验

#### 与其他角色的协作关系
- **上游消费**：消费 `2.2` 的机会对象，不重做机会判断
- **横向协作**：与资源承诺、风险门槛视角协作，形成完整行动结构
- **下游支撑**：为实现落地提供行动骨架

---

### 3.4 资源承诺负责人

#### 角色使命
把行动设计转化为分阶段资源承诺结构，确保资源释放与假设验证、门槛判断、推进节奏天然绑定。

#### 职责边界
- 定义阶段性资源释放逻辑，而不是一次性总预算逻辑
- 拆解不同阶段需要的关键能力角色、预算等级、时间投入与依赖条件
- 保证资源投入与关键假设验证天然绑定
- 设计资源承诺递增机制

#### 典型输入
- 行动设计面具输出
- 组织约束信息
- `2.4` 中的资源参考与类似案例
- 当前 `phased_plan` 草案

#### 典型输出
- 分阶段资源承诺草案
- 资源释放逻辑说明
- 能力角色需求说明
- 承诺节奏与升级条件补充
- 资源释放条件记录

#### 非职责（明确不做什么）
- ❌ 不把当前 MVP 直接写成完整预算系统
- ❌ 不脱离行动阶段独立报资源
- ❌ 不把"资源估得很细"误当作"承诺设计已经成立"
- ❌ 不做复杂的财务建模

#### 与其他角色的协作关系
- **依赖行动设计**：基于行动阶段设计资源承诺
- **配合风险门槛**：资源释放与门槛判断联动
- **支撑验证**：为验证提供资源合理性判断依据

---

### 3.5 风险约束 / 决策门槛负责人

#### 角色使命
把风险、限制条件和不确定性写进行动结构，而不是单独形成一个附录式风险章节。

#### 职责边界
- 设计 Go / No-Go 条件、升级 / 降级机制与退出条件
- 将风险、限制条件与不确定性直接映射到路径选择与资源释放
- 维护备选路径与止损策略
- 确保风险真正影响行动结构

#### 典型输入
- 行动设计草案
- 资源承诺草案
- `2.4` 风险矩阵或失败模式参考
- 开放问题与约束提醒

#### 典型输出
- `Go / No-Go` 条件草案
- 升级 / 降级 / 退出条件说明
- 备选路径记录
- 风险与承诺关联说明
- 决策门槛定义

#### 非职责（明确不做什么）
- ❌ 不把风险只写成"提醒列表"
- ❌ 不替代验证验收面具做最终可用性判断
- ❌ 不用抽象风险话术替代具体门槛定义
- ❌ 不独立于行动结构做风险分析

#### 与其他角色的协作关系
- **约束行动设计**：为行动路径提供风险约束
- **约束资源承诺**：为资源释放提供门槛条件
- **支撑验证**：为验证提供退出条件判断依据

---

### 3.6 验证与验收负责人

#### 角色使命
从"这个行动对象是否真的可拍板、可执行、可被后续验证"的角度审视当前方案，而不是只看结构是否完整。

#### 职责边界
- 设计轻量案例验证与可拍板性检查
- 判断当前行动对象是否足够清晰、可执行、可被后续现实校验
- 给出"可推进 / 需返工"的正式建议
- 维护验收检查表

#### 典型输入
- 当前契约草案
- 当前样例对象
- 阶段计划
- 验证方案与案例

#### 典型输出
- 轻量验证方案
- 验收检查表
- 验证建议
- 典型失败模式清单
- 是否建议继续推进的结论
- 验证记录与结论

#### 非职责（明确不做什么）
- ❌ 不负责主流程设计
- ❌ 不因为少数失败样例就直接推翻整个模块定位
- ❌ 不直接进入实现细节替代实现面具
- ❌ 不做 `2.5` 的现实校验工作

#### 与其他角色的协作关系
- **质量把关**：对各视角输出进行可拍板性检查
- **验证设计**：设计轻量验证方案
- **推进判断**：给出是否可进入下一阶段的建议

---

### 3.7 实现落地工程师视角

#### 角色使命
把已经基本稳定的契约与流程方案转化为可执行实现路径，并在运行后把结果回写到治理链路中。

#### 职责边界
- 把行动对象骨架、主流程与轻量验证路线落成可运行模块
- 完成样例运行、错误处理与回写
- 保证模块可被 `2.5` 和后续联调稳定接入
- 组织样例输入输出

#### 典型输入
- 已拍板的边界与契约
- 设计方案
- 样例验证要求

#### 典型输出
- `2.3` 实现代码
- 示例输入输出
- 实现拆解建议
- 样例运行记录
- 风险与返工点
- 回写建议
- 运行记录与问题回写

#### 非职责（明确不做什么）
- ❌ 不在拍板未完成时反向决定边界
- ❌ 不自行新增当前 MVP 之外的大功能
- ❌ 不把实现方便性当成模块本质
- ❌ 不跳过设计拍板直接实现

#### 与其他角色的协作关系
- **承接设计**：基于设计方案进行实现
- **反馈问题**：把实现中发现的问题反馈给设计视角
- **支撑验证**：为验证提供可运行的实现

## 四、角色调用顺序与协作机制

### 4.1 设计期推荐顺序

在当前治理与方案阶段，推荐协作顺序如下：

```
总协调面具
→ 方案设计 / 边界收口面具
→ 行动设计面具
→ 资源承诺面具
→ 风险门槛面具
→ 验证验收面具
→ （条件满足后）实现落地面具
```

**为什么这样排序**：
- 先判断"这轮要做什么"
- 再判断"可不可以这样做"
- 再形成"应该采取什么行动姿态与阶段路径"
- 再设计"按什么节奏释放资源"
- 再把风险写成门槛、退出和备选路径
- 最后再检查"是否真可拍板、可复盘、可实现"

### 4.2 实现期推荐顺序

在正式实现与验证期，推荐调用顺序：

```
实现落地面具（主导）
→ 验证验收面具（配合）
→ 方案设计 / 边界收口面具（仅在出现范围漂移时介入）
→ 资源承诺面具（仅在样例表明承诺节奏需要调整时介入）
```

### 4.3 不同阶段的角色调用策略

#### 治理收口期
**重点调用**：
- 总协调面具
- 方案设计 / 边界收口面具
- 行动设计面具

**暂时弱化**：
- 实现落地面具
- 资源承诺面具中的细预算推演部分

#### 方案设计期
**重点调用**：
- 总协调面具
- 方案设计 / 边界收口面具
- 行动设计面具
- 资源承诺面具
- 风险门槛面具
- 验证验收面具

#### 实现与验证期
**重点调用**：
- 实现落地面具
- 验证验收面具

**按需调用**：
- 方案设计 / 边界收口面具（仅在出现范围漂移时介入）
- 资源承诺面具（仅在样例表明承诺节奏需要调整时介入）

### 4.4 冲突处理规则

#### 常见冲突类型
1. **方案设计 / 边界收口面具**认为当前方案越界，但**实现落地面具**希望继续推进
2. **行动设计面具**希望增加姿态和阶段表达能力，但**验证验收面具**认为当前已可先做轻量验证
3. **资源承诺面具**希望补更多资源细节，但**方案设计 / 边界收口面具**认为当前 MVP 不宜膨胀
4. **风险门槛面具**认为门槛与退出条件不足，但**行动设计面具**认为当前已可先做轻量验证
5. **验证验收面具**认为样例仍不可拍板，但**实现落地面具**希望先跑通最小流程

#### 解决原则

按以下优先级处理：

1. **模块边界优先于实现便利**
2. **正式对象优先于表达花样**
3. **可拍板性优先于叙述完整性**
4. **验证可用性优先于理论完美性**
5. **用户拍板优先于角色面具内部偏好**

#### 必须回到用户拍板的情况

以下情况必须中止内部收敛，转回用户确认：

- 需要修改 `2.3` 与 `2.2 / 2.4 / 2.5` 的边界
- 需要把多 Agent 升级为当前 MVP 的硬要求
- 需要把复杂预算系统提前纳入当前 MVP
- 需要大幅改变主产物形态
- 需要让 `2.3` 越级承担 `2.5` 的现实验证职责

## 五、角色使用纪律

### 5.1 何时调用哪些角色

**不是每一轮都需要所有角色同时出场**：
- 当前任务只调用必要视角，避免形式化空转
- 每个角色面具输出都应服务于某个明确产物
- 一旦进入实现阶段，应减少高层抽象角色的频繁介入
- 在设计未收敛前，角色面具优先服务于方案收敛，而不是服务于"角色表演"

### 5.2 如何避免形式化空转

- 明确本轮任务目标，只调用必要角色
- 避免让所有角色一次性拉满并要求同时输出
- 避免让角色面具彼此辩论但没有明确产物
- 在角色尚未收敛前不直接写实现

### 5.3 如何保证职责覆盖完整

**必须独立保留治理关注的职责**：
- 行动设计（`2.3` 的核心本体）
- 资源承诺（区别于普通建议模块的关键）
- 风险门槛（可拍板性的保障）
- 验证验收（质量把关）

**允许执行合并的职责**：
- 总协调可与方案设计合并（在方案稳定后）
- 行动设计可与资源承诺合并（在流程清晰后）
- 风险门槛可与结构化契约合并（如引入该视角）

## 七、如何产出 `phase2.3_设计方案.md`

### 7.1 多角色协作产出方式

`phase2.3_设计方案.md` 应由多角色面具讨论并收敛产出，而不是单一视角直接起草。

**推荐流程**：
1. 总协调面具确认本轮目标与前置文档
2. 方案设计 / 边界收口面具确认是否越界
3. 行动设计、资源承诺、风险门槛面具搭建核心行动对象
4. 验证验收面具做复查
5. 方案设计负责人收敛为正式设计方案

### 7.2 设计方案应覆盖的内容

- `2.3` 的目标与非目标
- `2.3` 与 `2.2 / 2.4 / 2.5` 的边界关系
- `2.3` 的主流程设计
- 如何从 `2.2` 的机会对象转译为 `2.3` 的行动对象
- `ActionDesignRequest / ActionDecisionObject / ActionDesignResult` 的设计理解
- `decision_posture` 的使用语义与判断逻辑
- `phased_plan` 的结构与阶段颗粒度
- 资源承诺如何与阶段和假设验证绑定
- `Go/No-Go`、升级、降级、退出条件如何进入结构化输出
- `fallback_path`、`open_questions`、`top_risks` 如何组织
- 可选简版说明的定位与约束
- `MVP` 先做什么、明确不做什么
- 最小验证思路与首轮案例使用方式
- 当前仍待设计拍板的问题

## 八、执行纪律

### 8.1 先治理、先拍板，再建队、落 roles、做设计

`2.3` 不应带着治理分歧进入执行期角色落档与方案设计。

### 8.2 设计完成后必须再做一次设计拍板

未经设计拍板，不得直接进入正式实现。

### 8.3 本文件是执行角色定义的单一事实源

与"角色如何定义、职责如何分工、如何协作"相关的执行依据，以本文件为准。

### 8.4 `2.3` 不得借启动之名扩大范围

在用户未拍板前，不得把完整多 Agent、复杂预算、完整计划模板提前塞进当前 MVP。

### 8.5 角色面具协作，不是多 Agent 自治

当前角色面具用于设计期的高质量收敛，不代表产品运行时已经要采用多 Agent 编排。

---

## 九、一句话总结

> Phase 2.3 的角色面具配置，不是为了把项目包装成多 Agent 系统，而是为了让同一 Agent 在当前 MVP 范围内，能以更清晰的视角分工完成边界控制、行动设计、资源承诺、风险门槛、验证和实现推进。
