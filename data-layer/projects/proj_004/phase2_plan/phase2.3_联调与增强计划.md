# Phase 2.3 联调与增强计划

> **文档类型**：后续行动计划
> **适用模块**：Phase 2.3 行动设计与资源承诺模块
> **当前状态**：MVP 已完成，进入联调与增强阶段
> **最后更新**：2026-03-16

---

## 一、总体背景

Phase 2.3 MVP 已完成验收（3个案例全部通过），核心能力闭环：
- 姿态判断（watch / validate / pilot / escalate / hold / stop）
- 分阶段资源承诺（1-3阶段，含 Go/No-Go 门）
- 退出条件与 fallback 路径

下一步分两条主线推进：
1. **联调主线**：打通与 2.2（上游）、2.4（知识支持）、2.5（下游复盘）的接口
2. **增强主线**：在联调稳定后，按优先级补充 P1 增强能力

---

## 二、联调计划（优先级：高）

### 2.1 与 Phase 2.2 的联调（上游接口）

**目标**：2.3 直接消费 2.2 产出的 `OpportunityObject`，不再使用手工构造的测试数据

**接口现状**：
- 2.2 输出：`OpportunityObject`（12字段，含 priority_level / key_assumptions / uncertainty_map）
- 2.3 输入：`ActionDesignRequest`（依赖 OpportunityObject 字段）
- 当前缺口：`ActionDesignRequest` 尚未直接从 `OpportunityObject` 自动构造

**分步骤**：

| 步骤 | 内容 | 负责角色视角 | 输出物 |
|------|------|------------|--------|
| A1 | 确认 `OpportunityObject → ActionDesignRequest` 字段映射关系 | 方案设计视角 | 字段映射表 |
| A2 | 实现 `from_opportunity()` 构造方法或适配函数 | 实现落地视角 | `src/adapters.py` |
| A3 | 用 2.2 真实输出样本跑通 2.3 全流程 | 评测验收视角 | 联调测试报告 |
| A4 | 确认边界：2.3 不修改 OpportunityObject，只消费 | 总协调视角 | 拍板记录 |

**关键拍板项**：
- [ ] `priority_level` 映射到 2.3 姿态判断的规则是否与 2.2 的分级口径对齐？
- [ ] `key_assumptions` 数量阈值是否统一（当前 2.3 以 ≥3 个假设触发 validate）？

### 2.2 与 Phase 2.4 的联调（知识支持接口）

**目标**：2.3 在判断姿态和设计阶段计划时，可以向 2.4 检索相关领域知识（行业先例、资源基准等）

**接口现状**：
- 2.4 已暴露 `POST /api/v1/retrieve` 接口（纯向量检索，top_k=5）
- 2.3 当前完全规则驱动，不调用任何外部知识
- 当前缺口：2.3 尚无 RAG 集成层

**分步骤**：

| 步骤 | 内容 | 负责角色视角 | 输出物 |
|------|------|------------|--------|
| B1 | 确认 2.3 哪些判断点需要知识支持（姿态判断？资源估算？） | 方案设计视角 | 知识需求清单 |
| B2 | 确认 2.4 当前知识库覆盖范围是否满足 B1 需求 | 评测验收视角 | 覆盖度评估报告 |
| B3 | 实现 `KnowledgeClient` 封装 2.4 retrieve 接口 | 实现落地视角 | `src/knowledge_client.py` |
| B4 | 在 `ActionDesigner` 中增加可选知识增强入口（不改变主流程） | 实现落地视角 | `action_designer.py` 更新 |
| B5 | 跑通知识增强联调测试（至少 1 个真实案例） | 评测验收视角 | 联调测试记录 |

**关键拍板项**：
- [ ] MVP 阶段知识增强是「可选增强」还是「必须集成」？（推荐：可选，降级时走纯规则）
- [ ] 知识检索失败时的降级策略是什么？

### 2.3 与 Phase 2.5 的联调（下游接口）

**目标**：2.3 输出的 `ActionDesignResult` 可以被 2.5 的 `OutputChecker` 直接消费和校验

**接口现状**：
- 2.3 输出：`ActionDesignResult`（含 decision / phased_plan / risks / fallback_path）
- 2.5 消费：`OutputChecker` 做输出检查，`ProblemAttributor` 做问题归因
- 当前缺口：2.5 尚未有针对 `ActionDesignResult` 的 schema 检查规则

**分步骤**：

| 步骤 | 内容 | 负责角色视角 | 输出物 |
|------|------|------------|--------|
| C1 | 确认 2.5 `OutputChecker` 对 2.3 输出的检查维度 | 方案设计视角 | 检查维度清单 |
| C2 | 在 2.5 中注册 2.3 输出的 schema 检查规则 | 实现落地视角 | `output_checker.py` 更新 |

**关键拍板项**：
- [ ] `ActionDesignResult` 是否需要增加 `confidence_score` 字段以支持 2.5 归因？
- [ ] 2.3 的 fallback_path 是否需要结构化为 2.5 可读的格式？

## 三、增强计划（优先级：中，联调完成后启动）

### 3.1 P1 增强项（推荐在联调完成后 1 周内评估）

#### 增强A：姿态判断规则优化

**现状**：当前规则基于 `priority_level` + `key_assumptions` 数量，较为粗粒度

**增强方向**：
- 引入 `uncertainty_map` 中的不确定性等级作为判断权重
- 引入 `counter_evidence` 数量作为保守化因子
- 新增 `escalate` 姿态的判断条件（当前仅 `stop` 作为最高响应）

**实现范围**：仅修改 `_determine_posture()` 方法，不改接口

**验收标准**：新规则下，原有 3 个验证案例结果不退化，新增 2 个 escalate 案例通过

---

#### 增强B：分阶段计划细化

**现状**：当前每阶段资源估算为固定模板（"1-2人, 10-20天"），缺乏差异化

**增强方向**：
- 根据 `opportunity_thesis` 领域（AI/区块链/传统行业）差异化资源估算
- 根据 `priority_level` 调整阶段时间窗口
- pilot 姿态支持 2-3 阶段计划，每阶段有独立 Go/No-Go 准则

**实现范围**：修改 `_design_phased_plan()` 方法

**验收标准**：pilot 姿态案例产出 2 阶段计划，资源数字与案例描述匹配

---

#### 增强C：开放问题生成质量提升

**现状**：`open_questions` 直接从 `key_assumptions` 转换，质量偏机械

**增强方向**：
- 基于 `uncertainty_map` 中的不确定性类型生成更有针对性的问题
- 区分「验证型问题」（可做实验验证）和「调研型问题」（需外部数据）
- 为每个问题增加优先级标注

**实现范围**：修改 `_collect_open_questions()` 方法，扩展 `OpenQuestion` 数据模型

**验收标准**：每个案例生成的问题中，至少 50% 有明确的问题类型标注

---

### 3.2 P2 增强项（可选，按需评估）

| 增强项 | 描述 | 触发条件 |
|--------|------|----------|
| 历史决策对比 | 相似机会的历史姿态记录，支持「为什么这次选 validate 而不是 pilot」的解释 | 2.4 知识库接入后 |
| 姿态变更追踪 | 支持对同一机会在不同时间点重新设计，记录姿态变化轨迹 | 有重复评估需求时 |
| 批量处理接口 | 支持一次性处理多个 OpportunityObject 输入 | 有批量评估需求时 |
| Prompt 模式迁移 | 将规则引擎的姿态判断迁移为 LLM prompt 驱动 | 规则覆盖不足时 |

## 四、执行顺序与节奏建议

```
联调阶段（建议 3-5 天）：
  Day 1-2: A1-A4（打通与 2.2 的接口）
  Day 2-3: C1-C4（打通与 2.5 的接口）
  Day 3-4: B1-B5（打通与 2.4 的知识支持接口，可选）
  Day 4-5: D1-D5（端到端链路联调）

增强阶段（联调稳定后）：
  Week 2: 增强A（姿态判断规则优化）
  Week 2: 增强B（分阶段计划细化）
  Week 3: 增强C（开放问题生成质量提升）
  Week 3+: P2 增强项按需评估
```

## 五、联调完成标准

| 条件 | 验收标准 |
|------|----------|
| 2.2 → 2.3 接口联调 | 使用 OpportunityObject 真实输出，跑通 2 个案例，姿态判断结果合理 |
| 2.4 → 2.3 知识支持 | 知识检索可调用，降级路径验证通过 |
| 2.3 → 2.5 接口联调 | ActionDesignResult 被 OutputChecker 消费，无 schema 报错 |
| 端到端链路 | 至少 1 个真实案例完成完整链路，各模块输出记录存档 |
