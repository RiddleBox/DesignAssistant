# 阶段2.3目标说明：行动设计与资源承诺模块

> **制定者**：EMP-019 阶段2规划师
> **制定时间**：2026-03-12
> **工作周期**：Week 4-5（2周）
> **版本**：v1.1

---

## 1. 子阶段目标

### 1.1 核心目标

实现 `2.3` 行动设计与资源承诺模块，将 `2.2` 形成的机会判断对象升级为**可拍板、可推进、可回退**的结构化行动决策对象。

### 1.2 具体目标

> **口径校正说明（研发方法层 vs 系统功能层）**：
> 本文档保留了 `2.3` 早期“决策建议模块”导向的任务拆分痕迹，但当前正式口径应以“**行动设计层 / 资源承诺层**”理解 `2.3`，而不是把它当作长篇建议文档生成器。
> 当前 `2.3 MVP` 的中心应是：**结构化行动决策对象 + 最小行动设计闭环**，即围绕 `decision_posture -> phased_plan -> go_no_go_criteria -> exit_conditions` 建立稳定中层对象。
> 如后续讨论 `多角色`、`多Agent` 等内容，也应区分清楚：前者当前主要服务于**研发方法层**的设计收敛，后者若纳入，则属于**系统功能层**的运行时增强项，而不是当前研发阶段的默认实现前提。

1. **行动姿态判断**：能够形成 `watch / validate / pilot / escalate / hold / stop` 等首版行动姿态
2. **分阶段承诺设计**：能够生成面向推进与回退的 `phased_plan`
3. **风险与退出条件设计**：能够识别关键风险，并形成 `go / no-go` 判断依据与退出条件
4. **轻量资源承诺表达**：能够给出阶段级资源承诺逻辑，而不是完整预算系统
5. **结构化行动对象生成**：产出符合当前 `2.3 MVP` 口径的结构化行动决策对象

### 1.3 成功标准

- ✅ 能生成可拍板的结构化行动决策对象
- ✅ 能形成最小行动设计闭环：`decision_posture -> phased_plan -> go_no_go_criteria -> exit_conditions`
- ✅ 能提供阶段级轻量资源承诺表达
- ✅ 至少完成2个真实机会对象的轻量验证案例

## 3. 核心工作内容

### 3.1 行动对象骨架实现（Week 4, Day 1-3）

**负责人**：决策逻辑工程师

**工作内容**：
1. 实现行动姿态判断逻辑
   - 识别 `watch / validate / pilot / escalate / hold / stop`
   - 将机会判断结果映射为行动姿态
   - 输出姿态选择理由与关键前提

2. 实现分阶段承诺骨架
   - 以阶段目标、关键假设、动作、门槛为主组织 `phased_plan`
   - 保持“可拍板、可推进、可回退”的中层颗粒度
   - 不直接膨胀为完整长周期经营路线图

3. 实现 `Go / No-Go` 与退出条件逻辑
   - 基于行动姿态给出推进条件
   - 明确暂停、止损和回退触发条件
   - 形成首版行动闭环对象

**输出物**：
- `decision_advisor.py`（核心模块）
- `path_generator.py`（分阶段承诺生成器）
- `milestone_planner.py`（阶段门槛规划器）
- `decision_logic.md`（行动设计逻辑说明）

### 3.2 轻量资源承诺实现（Week 4, Day 3-5）

**负责人**：资源规划工程师

**工作内容**：
1. 实现阶段级人力承诺表达
   - 基于行动姿态估算团队带宽需求
   - 定义关键角色类型与参与强度
   - 形成粗粒度人力承诺口径

2. 实现阶段级资金与时间表达
   - 给出预算带与时间带，而不是完整财务建模
   - 结合关键门槛描述资源释放节奏
   - 保留后续扩展到更细资源模型的空间

3. 实现资源承诺逻辑
   - 将资源投入与阶段门槛绑定
   - 体现“先验证、再升级投入”的承诺策略
   - 输出可服务拍板的轻量资源说明

**输出物**：
- `resource_planner.py`（资源规划器）
- `budget_estimator.py`（轻量预算带估算器）
- `timeline_calculator.py`（时间带计算器）
- `resource_models.json`（资源估算模型）

### 3.3 风险、回退与应对设计（Week 5, Day 1-3）

**负责人**：风险分析工程师

**工作内容**：
1. 实现关键风险识别逻辑
   - 技术风险（可行性、复杂度）
   - 市场风险（需求、竞争）
   - 团队风险（能力、稳定性）
   - 资金风险（融资、现金流）

2. 实现 `go / no-go` 判断支撑
   - 概率评估（高/中/低）
   - 影响评估（高/中/低）
   - 风险等级（红/黄/绿）

3. 实现退出条件与应对策略
   - 预防措施（降低概率）
   - 缓解措施（降低影响）
   - 回退 / 暂停 / 止损触发条件

**输出物**：
- `risk_analyzer.py`（风险分析器）
- `risk_matrix.json`（风险矩阵）
- `mitigation_strategies.json`（应对策略库）
- `risk_guide.md`（风险分析指南）

### 3.4 案例生成与轻量验证（Week 5, Day 3-7）

**负责人**：全体团队成员

**工作内容**：
1. 基于 `2.2` 的评估报告生成结构化行动对象
   - 运行 `2.3` 模块
   - 生成完整的行动决策对象
   - 检查字段、逻辑闭环与可拍板性

2. 验证行动对象质量
   - 行动姿态是否合理
   - 分阶段承诺是否清晰可推进
   - 轻量资源表达与风险退出条件是否匹配

3. 收集反馈和改进
   - 识别对象骨架不足
   - 优化行动设计逻辑
   - 更新文档与验证结论

**输出物**：
- `cases/case1_decision.md`（案例1行动决策对象）
- `cases/case2_decision.md`（案例2行动决策对象）
- `cases/validation_report.md`（验证报告）

## 7. 风险与应对

### 7.1 关键风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 资源估算不准确 | 高 | 中 | 参考历史数据，建立估算模型 |
| 行动路径过于理想化 | 中 | 中 | 增加风险分析，提供备选方案 |
| 决策建议缺乏可执行性 | 中 | 高 | 增加具体步骤和检查点 |
