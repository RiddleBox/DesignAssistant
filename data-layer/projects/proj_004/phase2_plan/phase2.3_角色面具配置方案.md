> **文档类型**：执行视角配置文档  
> **适用模块**：`Phase 2.3` 行动设计与资源承诺模块  
> **文档目标**：为 `2.3` 在同一 Agent 下的协作执行提供一套清晰、可压缩、可落地的角色面具配置方案，确保后续方案设计、验证和实现能够在明确的视角分工下进行。

## 一、文档定位说明

### 1.1 这份文档解决什么问题

这份文档回答的是：

- `2.3` 需要哪些执行视角来完成当前阶段工作
- 每个角色面具分别负责什么输入、输出和判断任务
- 这些角色在同一 Agent 内如何协作，而不是互相覆盖或互相抢职责
- 后续另一端接手细方案时，应该如何使用这套角色视角
- 正式职责清单如何被转译成执行期可切换、可压缩的面具体系

### 1.2 这份文档不解决什么问题

这份文档**不**负责以下内容：

- 不决定是否需要团队重组
- 不重新定义 `2.3` 的宏观阶段责任归属
- 不替代 [phase2.3_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.3_团队重组建议清单.md)
- 不把角色面具误写成运行时多 Agent 架构设计
- 不在当前阶段为每个角色直接写出最终 Prompt 成品
- 不越俎代庖替另一端先写完整 `phase2.3_设计方案.md`

### 1.3 与团队重组建议清单的差异

- **团队重组建议清单**：组织层，回答“`2.3` 正式需要覆盖哪些职责视角，这些职责为什么必须存在，以及它们在阶段推进中如何构成正式团队结构”
- **角色面具配置方案**：执行层，回答“同一 Agent 内部应如何把这些正式职责转成可切换、可压缩、可协作的执行视角”

因此，这份文档会重点写：

- 角色定义
- 输入输出
- 使用时机
- 协作顺序
- 冲突解决机制
- 正式职责如何映射为执行面具

而不会重复写大量组织配置内容。

### 1.4 与 `phase2_roles/phase2.3_roles.md` 的关系

这份文档还需要和另一份正式角色文件区分开：

- **[phase2.3_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.3_团队重组建议清单.md)**：给出 `2.3` 的**正式职责来源**
- **本文件**：说明这些正式职责在**同一 Agent 下如何执行化、如何压缩、如何协作**
- **`phase2_roles/phase2.3_roles.md`**：由另一端基于前两份文档正式落档，作为后续设计方案与实现阶段的执行角色依据

换句话说：

> **本文件不是最终角色落档文件，而是正式角色落档前的执行化桥接文档。**

## 二、配置总原则

### 2.1 核心原则

`2.3` 当前采用的是：

> **同一 Agent 下的角色面具协作模式，而不是多 Agent 自治模式。**

这意味着：

- 所有角色面具共享同一总体目标
- 角色面具之间是职责切换，不是互相竞争的独立智能体
- 角色面具的价值在于提高设计质量、判断完整度与交接稳定性
- 角色面具当前服务于 `2.3 MVP` 收口，不服务于“做一个看起来很复杂的多 Agent showcase” [[memory:rebp86gg]]

### 2.2 当前配置目标

本轮角色面具配置要服务于五件事：

- **帮助 `2.3` 守住模块边界**
- **帮助 `2.3` 建立结构化行动对象**
- **帮助 `2.3` 把资源承诺、风险门槛和退出条件写进行动结构**
- **帮助 `2.3` 形成可拍板、可验证、可被 `2.5` 消费的输出**
- **帮助另一端在接手时有统一执行视角**

### 2.3 当前配置约束

- 不以完整多 Agent 辩论为前提
- 不让角色面具替代用户拍板
- 不让角色面具绕开治理文档自行扩范围
- 不在 `MVP` 未稳之前设计过多“炫技型”视角
- 不把角色面具误当成当前产品运行时系统形态的预设承诺

### 2.4 判断层级边界说明

为避免与 `2.2`、`2.4`、`2.5` 发生语义混淆，本文件中的“判断”一词采用与团队重组文档一致的层级定义：

- **`2.2 = 机会级判断`**：判断一组信号与证据是否能构成值得继续跟踪、深挖或升级的机会对象
- **`2.3 = 行动级判断 / 资源承诺判断`**：判断基于机会对象当前应采取什么行动姿态、释放什么阶段资源、设置什么推进门槛与退出条件
- **`2.4 = 行动支撑上下文`**：提供案例、方法、约束、风险与资源参考，不直接替代 `2.3` 做行动设计
- **`2.5 = 现实校验与闭环归因`**：检验 `2.3` 设计出的行动对象在真实推进中是否成立、偏差在哪里、如何回写系统改进

因此，当前角色面具中的：

- **行动设计面具**，处理的是**行动级判断**，不是 `2.2` 的机会判断
- **资源承诺面具**，处理的是**承诺节奏与释放结构**，不是泛化预算系统
- **风险门槛面具**，处理的是**Go / No-Go / 升级 / 降级 / 退出条件**，不是独立风险章节写作
- **结构化契约面具**，服务的是**行动对象及其下游交接契约**，不是提前把所有 Schema 工程化做满

## 三、推荐角色面具总览

### 3.1 标准配置

建议 `2.3` 标准配置采用 `7-8` 个角色面具：

1. **总协调面具**
2. **方案设计 / 边界收口面具**
3. **行动设计面具**
4. **资源承诺面具**
5. **风险门槛面具**
6. **结构化契约面具**
7. **验证验收面具**
8. **实现落地面具**

### 3.2 压缩配置

如果执行阶段需要压缩，可合并为 `5-6` 个角色面具：

- 总协调面具
- 方案设计 / 边界收口面具
- 行动设计 + 资源承诺面具
- 风险门槛 + 结构化契约面具
- 验证验收面具
- 实现落地面具

### 3.3 角色面具使用原则

- 并不是每一轮都需要所有角色同时出场
- 当前任务只调用必要视角，避免形式化空转
- 每个角色面具输出都应服务于某个明确产物
- 一旦进入实现阶段，应减少高层抽象角色的频繁介入
- 在设计未收敛前，角色面具优先服务于方案收敛，而不是服务于“角色表演”

## 四、角色面具详细定义

## 4.1 总协调面具

### 角色使命

负责在每一轮任务中确认当前目标、调用顺序、输入前提和结束条件，避免团队视角在细节中失焦。

### 关注问题

- 当前这一轮到底要产出什么
- 需要哪些前置文档先对齐
- 哪些问题属于当前阶段，哪些应后置
- 当前应该调用哪些角色，而不是把所有角色都拉进来

### 典型输入

- 当前用户目标
- 当前阶段文档
- 已拍板事项
- 未拍板事项

### 典型输出

- 本轮任务分解
- 参与角色列表
- 执行顺序
- 停止条件与交付物定义

### 不应该做的事

- 不直接替代行动设计面具下结论
- 不直接定义字段细节
- 不直接充当实现面具写实现方案

---

## 4.2 方案设计 / 边界收口面具

### 角色使命

负责持续检查 `2.3` 是否仍然站在“行动设计层 / 资源承诺层”边界内，并把多视角输入收敛为可拍板的正式方案骨架。

### 关注问题

- 当前设计是否还在 `2.3` 范围内
- 是否出现职责上行或下探
- 是否误把未来增强项纳入当前 `MVP`
- 行动设计、资源承诺、风险门槛、验证是否已形成统一方案骨架

### 典型输入

- 第一性原理文档
- `MVP Scope` 文档
- 待拍板清单
- 当前方案草案

### 典型输出

- 边界检查结论
- 方案骨架草案
- 非目标提醒
- 是否需要回到用户拍板的建议

### 不应该做的事

- 不替代总协调面具安排全局流程
- 不独占所有角色的判断权
- 不因为“更完整”而主动扩范围

---

## 4.3 行动设计面具

### 角色使命

负责定义 `2.3` 的核心行动对象与核心判断逻辑，把机会对象组织成“当前应采取何种行动姿态、按什么阶段推进”的行动骨架。

### 关注问题

- 当前应该采用什么行动姿态
- 为什么是 `watch / validate / pilot / escalate / hold / stop` 中的某一种
- 当前最小行动闭环应该长什么样
- 哪些阶段目标、关键动作和里程碑最值得先冻结

### 典型输入

- `2.2` 输出的结构化机会对象
- `2.4` 提供的行动支撑上下文
- 当前行动对象草案
- 拍板后的姿态口径

### 典型输出

- 行动姿态骨架
- 分阶段推进草案
- 关键动作与里程碑草案
- 下一步需要验证的关键假设清单

### 不应该做的事

- 不回卷重做 `2.2` 的机会成立判断
- 不直接替代资源承诺面具给出资源释放逻辑
- 不脱离约束只给感觉型行动建议

---

## 4.4 资源承诺面具

### 角色使命

负责把行动设计转化为分阶段资源承诺结构，确保资源释放与假设验证、门槛判断、推进节奏天然绑定。

### 关注问题

- 当前阶段应该释放什么级别的资源
- 资源承诺为什么不能一次性给满
- 角色、带宽、时间与粗粒度预算带如何表达
- 哪些资源释放必须绑定到特定验证结果之后

### 典型输入

- 行动设计面具输出
- 组织约束信息
- `2.4` 中的资源参考与类似案例
- 当前 `phased_plan` 草案

### 典型输出

- 分阶段资源承诺草案
- 资源释放逻辑说明
- 能力角色需求说明
- 承诺节奏与升级条件补充

### 不应该做的事

- 不把当前 `MVP` 直接写成完整预算系统
- 不脱离行动阶段独立报资源
- 不把“资源估得很细”误当作“承诺设计已经成立”

---

## 4.5 风险门槛面具

### 角色使命

负责把风险、限制条件和不确定性写进行动结构，而不是单独形成一个附录式风险章节。

### 关注问题

- 当前阶段继续推进的门槛是什么
- 哪些情况应暂停、降级或退出
- 备选路径和止损路径应该怎样定义
- 哪些风险会直接改变资源释放与行动姿态

### 典型输入

- 行动设计草案
- 资源承诺草案
- `2.4` 风险矩阵或失败模式参考
- 开放问题与约束提醒

### 典型输出

- `Go / No-Go` 条件草案
- 升级 / 降级 / 退出条件说明
- 备选路径记录
- 风险与承诺关联说明

### 不应该做的事

- 不把风险只写成“提醒列表”
- 不替代验证验收面具做最终可用性判断
- 不用抽象风险话术替代具体门槛定义

---

## 4.6 结构化契约面具

### 角色使命

负责把行动设计逻辑转换成稳定可消费的输入 / 输出契约，确保 `2.3` 可以被下游稳定接入，而不是停留在抽象描述层。

### 关注问题

- `2.3` 的请求对象应该长什么样
- `ActionDecisionObject` 的最小字段是什么
- 哪些字段是正式字段，哪些只是辅助字段
- `2.3 -> 2.5` 的交接需要哪些最小信息

### 典型输入

- 启动与拍板文档中的契约草案
- 行动设计 / 资源承诺 / 风险门槛面具输出
- 上下游模块边界说明

### 典型输出

- 输入 Schema 草案
- 输出 Schema 草案
- 字段注释与消费说明
- 示例对象样本

### 不应该做的事

- 不直接判断行动方案是否应该成立
- 不把契约设计成过度依赖 `2.4` 的强耦合结构
- 不在当前阶段过早设计复杂版本治理体系

---

## 4.7 验证验收面具

### 角色使命

负责从“这个行动对象是否真的可拍板、可执行、可被后续验证”的角度审视当前方案，而不是只看结构是否完整。

### 关注问题

- 输出对象是否稳定合法
- 人类是否能理解并讨论这个行动对象
- `2.5` 是否真的可消费这些字段
- 当前样例验证是否足以支撑继续推进
- 是否已经具备进入下一阶段的条件

### 典型输入

- 当前契约草案
- 当前样例对象
- 阶段计划
- 验证方案与案例

### 典型输出

- 验收检查表
- 验证建议
- 典型失败模式清单
- 是否建议继续推进的结论

### 不应该做的事

- 不负责主流程设计
- 不因为少数失败样例就直接推翻整个模块定位
- 不直接进入实现细节替代实现面具

---

## 4.8 实现落地面具

### 角色使命

负责把已经基本稳定的契约与流程方案转化为可执行实现路径，并在运行后把结果回写到治理链路中。

### 关注问题

- 哪些部分已经足够稳定，可以开始做
- 先实现哪条最小路径
- 如何组织样例输入输出
- 运行结果要回写到哪些文档中

### 典型输入

- 已拍板的边界与契约
- 设计方案
- 样例验证要求

### 典型输出

- 实现拆解建议
- 样例运行记录
- 风险与返工点
- 回写建议

### 不应该做的事

- 不在拍板未完成时反向决定边界
- 不自行新增当前 `MVP` 之外的大功能
- 不把实现方便性当成模块本质

## 五、正式职责与执行面具的映射关系

### 5.1 映射原则

本节的作用，是帮助另一端在建立 `phase2_roles/phase2.3_roles.md` 时，不会把“正式职责视角”和“执行期角色面具”混成两套互相竞争的命名系统。

### 5.2 映射表

| 正式职责视角 | 对应执行面具 | 使用说明 |
|--------------|--------------|----------|
| **总协调 / 架构连续性视角** | **总协调面具** | 作为全局收口与跨阶段连续性的正式职责来源 |
| **方案设计负责人** | **方案设计 / 边界收口面具** | 在执行层负责方案收敛、边界检查与取舍整理 |
| **行动设计负责人** | **行动设计面具** | 对应 `2.3` 的行动级判断核心职责 |
| **资源承诺负责人** | **资源承诺面具** | 负责资源释放节奏与承诺结构 |
| **风险约束 / 决策门槛负责人** | **风险门槛面具** | 负责门槛、止损、备选路径与风险约束 |
| **验证与验收负责人** | **验证验收面具** | 负责轻量验证、验收检查与可推进判断 |
| **实现落地工程师视角** | **实现落地面具** | 负责从已拍板方案进入可运行实现 |
| **契约 / 下游交接支撑职责** | **结构化契约面具** | 作为执行层的专项支撑面具出现，优先服务 `2.2 -> 2.3` 消费与 `2.3 -> 2.5` 交接稳定性 |

### 5.3 如何理解 `结构化契约面具`

这里要特别说明：

- 在 `2.3` 当前治理体系里，**正式职责来源仍优先以团队重组建议清单为准**
- 但从执行层看，`2.3` 又确实需要一个专门处理输入输出契约、字段骨架与下游可消费性的专项视角
- 因此，本文件将 **`结构化契约面具` 定义为执行层专项支撑面具**，用于帮助另一端把行动对象与交接契约做稳
- 后续另一端在建立 `phase2_roles/phase2.3_roles.md` 时，可以将其：
  - 作为独立支撑角色保留；或
  - 归并到 **方案设计 / 边界收口面具** 与 **验证验收面具** 的职责组合中

**推荐口径**：

- 标准配置：保留 `结构化契约面具`
- 压缩配置：将其并入“风险门槛 + 结构化契约面具”或“方案设计 / 边界收口面具”

### 5.4 落档优先级说明

1. `phase2_roles/phase2.3_roles.md` 的正式职责命名，优先以 [phase2.3_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.3_团队重组建议清单.md) 为准。
2. 各职责在同一 Agent 下如何协作、何时调用、如何压缩执行，以本文件为准。
3. 如果角色命名或职责颗粒度出现差异，应先回到团队重组文档确认正式职责，再同步修订执行层角色映射。

## 六、推荐协作顺序

### 6.1 设计期推荐顺序

在当前治理与方案阶段，推荐协作顺序如下：

```text
总协调面具
→ 方案设计 / 边界收口面具
→ 行动设计面具
→ 资源承诺面具
→ 风险门槛面具
→ 结构化契约面具
→ 验证验收面具
→ （条件满足后）实现落地面具
```

### 6.2 为什么这样排序

这个顺序确保：

- 先判断“这轮要做什么”
- 再判断“可不可以这样做”
- 再形成“应该采取什么行动姿态与阶段路径”
- 再设计“按什么节奏释放资源”
- 再把风险写成门槛、退出和备选路径
- 再把方案转换成可消费对象契约
- 最后再检查“是否真可拍板、可复盘、可实现”

而不是一开始就从实现、预算细化或 Prompt 细节入手。

## 八、冲突处理规则

### 8.1 常见冲突类型

- **方案设计 / 边界收口面具**认为当前方案越界，但**实现落地面具**希望继续推进
- **行动设计面具**希望增加姿态和阶段表达能力，但**结构化契约面具**希望保持对象简洁
- **资源承诺面具**希望补更多资源细节，但**方案设计 / 边界收口面具**认为当前 `MVP` 不宜膨胀
- **风险门槛面具**认为门槛与退出条件不足，但**行动设计面具**认为当前已可先做轻量验证
- **验证验收面具**认为样例仍不可拍板，但**实现落地面具**希望先跑通最小流程

### 8.2 解决原则

按以下优先级处理：

1. **模块边界优先于实现便利**
2. **正式对象优先于表达花样**
3. **可拍板性优先于叙述完整性**
4. **验证可用性优先于理论完美性**
5. **用户拍板优先于角色面具内部偏好**

### 8.3 必须回到用户拍板的情况

以下情况必须中止内部收敛，转回用户确认：

- 需要修改 `2.3` 与 `2.2 / 2.4 / 2.5` 的边界
- 需要把多 Agent 升级为当前 `MVP` 的硬要求
- 需要把复杂预算系统提前纳入当前 `MVP`
- 需要大幅改变主产物形态
- 需要让 `2.3` 越级承担 `2.5` 的现实验证职责

## 九、另一端接手时的使用方式

另一端后续接手时，不应把这份文档理解为“必须每轮都演完整角色戏”，而应理解为：

- 在做设计时，知道当前需要从哪些视角检查方案
- 在写方案时，知道哪些判断应由哪个角色面具负责
- 在发现冲突时，知道该按什么顺序处理
- 在进入实现时，知道什么时候应减少高层角色，更多交给实现与验证视角

### 9.1 最推荐的使用方法

- 先用总协调面具确认任务边界
- 用方案设计 / 边界收口面具确认是否越界
- 用行动设计、资源承诺、风险门槛面具搭建核心行动对象
- 用结构化契约面具把对象变成稳定输入输出骨架
- 用验证验收面具做复查
- 条件满足后，再调用实现落地面具推进

### 9.2 最不推荐的使用方法

- 把所有角色一次性拉满并要求同时输出
- 让角色面具彼此辩论但没有明确产物
- 在角色尚未收敛前直接写实现
- 把角色面具文档误当成运行时多 Agent 系统设计稿
- 让“资源承诺面具”变成“预算细化器”

## 十、配置结论

### 10.1 当前建议

- `2.3` 当前适合采用 **`7-8` 个标准角色面具 / `5-6` 个压缩角色面具** 的方案
- 当前角色面具配置应服务于**行动设计质量、承诺逻辑稳定性与边界控制**，而不是为了展示复杂性
- 角色面具的核心价值在于让另一端接手时，能够按稳定视角做高质量方案设计与实现
- `结构化契约面具` 适合作为 `2.3` 的执行层专项支撑视角保留，以降低后续 `2.2 -> 2.3 -> 2.5` 交接成本

### 10.2 一句话总结

> `Phase 2.3` 的角色面具配置，不是为了把项目包装成多 Agent 系统，而是为了让同一 Agent 在当前 `MVP` 范围内，能以更清晰的视角分工完成边界控制、行动设计、资源承诺、风险门槛、契约设计、验证和实现推进。
