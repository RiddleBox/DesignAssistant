# 阶段2.4目标说明：知识库&RAG系统

> **制定者**：EMP-019 阶段2规划师
> **制定时间**：2026-03-12
> **工作周期**：Week 2-5（4周，并行执行）
> **版本**：v1.0

---

## 1. 子阶段目标

### 1.1 核心目标

构建行业知识库并实现RAG检索引擎，支持其他模块的知识查询，提供高质量的上下文信息。

### 1.2 具体目标

1. **知识库构建**：构建至少100条高质量知识条目
2. **RAG检索引擎**：实现混合检索（向量+关键词）
3. **向量化系统**：实现文档向量化和索引
4. **检索优化**：检索准确率≥75%，速度<2秒

### 1.3 成功标准

- ✅ 知识库包含至少100条高质量知识
- ✅ 检索准确率 ≥ 75%（Top-5）
- ✅ 检索速度 < 2秒
- ✅ 支持元数据过滤（时间、类别、来源）

## 3. 核心工作内容

### 3.1 知识库架构设计（Week 2, Day 1-2）

**负责人**：知识库架构师

**工作内容**：
1. 设计知识库结构
   - 定义目录结构（行业情报、案例库、方法论库、团队画像）
   - 定义元数据标准（时间、类别、来源、标签）
   - 定义知识条目格式（Markdown + YAML）

2. 设计知识分类体系
   - 行业情报分类（新闻、报告、趋势）
   - 案例分类（成功、失败、范式转移）
   - 方法论分类（评估框架、决策模型、风险矩阵）

3. 设计知识质量标准
   - 完整性标准
   - 准确性标准
   - 时效性标准

**输出物**：
- `knowledge_base/README.md`（知识库说明）
- `knowledge_base/schema.yaml`（元数据Schema）
- `docs/kb_architecture.md`（架构设计文档）
- `docs/quality_standards.md`（质量标准）

### 3.2 知识采集与清洗（Week 2-4，持续进行）

**负责人**：数据工程师

**工作内容**：
1. 采集知识来源
   - 行业报告（Newzoo、Sensor Tower等）
   - 游戏资讯（GameLook、触乐等）
   - 技术文档（OpenAI、Anthropic文档）
   - 案例资料（公开报道、分析文章）

2. 清洗和结构化
   - 提取关键信息
   - 统一格式（Markdown + YAML）
   - 添加元数据（时间、来源、标签）

3. 质量控制
   - 去重
   - 事实核查
   - 格式验证

**输出物**：
- `knowledge_base/industry_intelligence/`（至少30条）
- `knowledge_base/case_library/`（至少30条）
- `knowledge_base/methodology/`（至少20条）
- `knowledge_base/team_profiles/`（至少20条）
- `data/kb_statistics.json`（知识库统计）

### 3.3 向量化系统实现（Week 2-3）

**负责人**：向量化工程师

**工作内容**：
1. 实现文档向量化
   - 使用text-embedding-3-large模型
   - 分块策略（chunk size: 512 tokens）
   - 批量处理优化

2. 构建向量索引
   - 选择向量数据库（Qdrant / Pinecone）
   - 创建索引
   - 优化检索性能

3. 实现增量更新
   - 新文档自动向量化
   - 索引自动更新
   - 版本管理

**输出物**：
- `rag/vectorizer.py`（向量化模块）
- `rag/vector_db.py`（向量数据库接口）
- `rag/indexer.py`（索引管理器）
- `docs/vectorization_guide.md`（向量化指南）

### 3.4 RAG检索引擎实现（Week 3-4）

**负责人**：RAG工程师

**工作内容**：
1. 实现混合检索
   - 向量检索（语义相似度）
   - 关键词检索（BM25）
   - 混合策略（向量0.7 + 关键词0.3）

2. 实现Reranker
   - 使用LLM对检索结果重排序
   - 提高Top-K准确率

3. 实现元数据过滤
   - 时间过滤（最近N天）
   - 类别过滤（特定类型）
   - 来源过滤（特定来源）

**输出物**：
- `rag/retriever.py`（检索引擎）
- `rag/reranker.py`（重排序器）
- `rag/filters.py`（过滤器）
- `docs/retrieval_guide.md`（检索指南）

### 3.5 检索效果测试与优化（Week 4-5）

**负责人**：全体团队成员

**工作内容**：
1. 准备测试查询
   - 设计20个测试查询
   - 标注标准答案（Top-5）

2. 执行测试
   - 测试检索准确率
   - 测试检索速度
   - 测试不同参数配置

3. 优化检索效果
   - 调整混合检索权重
   - 优化分块策略
   - 优化Reranker

**输出物**：
- `tests/retrieval_test_cases.json`（测试用例）
- `tests/retrieval_test_results.md`（测试报告）
- `docs/optimization_log.md`（优化记录）

## 5. 技术栈

### 5.1 核心技术

- **Embedding模型**：text-embedding-3-large
- **向量数据库**：Qdrant / Pinecone
- **检索策略**：混合检索（向量0.7 + 关键词0.3）
- **知识格式**：Markdown + YAML元数据
- **语言**：Python 3.10+

### 5.2 依赖库

- `openai`：Embedding API
- `qdrant-client` / `pinecone-client`：向量数据库
- `langchain`：RAG框架
- `rank-bm25`：关键词检索
- `pyyaml`：元数据解析

## 7. 风险与应对

### 7.1 关键风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 知识采集耗时超预期 | 高 | 中 | 先构建MVP（50条），后续迭代 |
| 检索效果不达标 | 中 | 中 | 准备降级方案（纯LLM） |
| 向量数据库成本高 | 中 | 低 | 使用开源方案（Qdrant） |
| 知识质量参差不齐 | 中 | 中 | 建立质量审核流程 |

### 7.2 依赖风险

- **对外部数据源的依赖**：需要高质量的行业数据
  - **应对**：提前准备多个数据源，避免单点依赖
