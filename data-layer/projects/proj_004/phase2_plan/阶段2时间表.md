### 1.3 模块启动与团队细化节奏原则

- **总规划先行**：阶段2先以《阶段2团队构建方案》作为整体资源规划、角色基线与阶段节奏依据。
- **执行团队后置细化**：`2.1`、`2.2`、`2.3` 在各自正式启动时，再结合当前真实上下游依赖、契约冻结情况和待拍板事项，补充对应的执行版团队配置与职责拆分。
- **按模块启动逐步细化**：不在 Week 1 一次性把所有执行版团队全部定死，而是在模块进入启动窗口时完成拍板，减少因上游未稳定导致的返工。
- **上游输出优先**：`2.2` 的执行团队细化应以 `2.1` 已阶段性冻结的输入输出契约为依据；`2.3` 的执行团队细化应以 `2.2` 已稳定的评估输出与字段口径为依据。
- **节奏要求**：每个模块进入实现前，必须先完成“启动与拍板 / 执行团队细化 / 边界与验收确认”，再进入主实现阶段。

## 3. 关键路径分析

### 3.1 关键路径

```
2.0 → 2.1 → 2.2 → 2.3 → 2.5
```

**说明**：
- 2.1→2.2→2.3是串行依赖，构成关键路径
- 2.4与关键路径并行，不在关键路径上
- 关键路径总时长：1周 + 2周 + 2周 + 2周 + 3周 = 10周

**实际总时长**：8周（通过并行执行压缩）

### 3.2 时间压缩策略

1. **2.1和2.2错峰启动**：`2.2` 在 `2.1` 形成阶段性冻结契约后启动拍板与团队细化，随后进入主实现，以减少等待时间
2. **2.4并行执行**：`2.4` 与 `2.1` / `2.2` / `2.3` 并行，优先提供术语、案例与评估知识支持
3. **预留缓冲时间**：每个子阶段预留10-15%缓冲

## 5. 风险时间管理

### 5.1 时间风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 某子阶段延期1-2天 | 高 | 低 | 使用缓冲时间吸收 |
| 某子阶段延期3-5天 | 中 | 中 | 调整后续计划，压缩非关键路径 |
| 某子阶段延期>5天 | 低 | 高 | 启动应急预案，增加资源 |
| 整体延期超过8周 | 低 | 高 | 压缩2.5时间（最少2周） |

### 5.2 缓冲时间使用规则

**原则**：
- 缓冲时间仅用于应对意外，不用于常规工作
- 使用缓冲时间需要EMP-020批准
- 缓冲时间用完后，必须调整计划

**使用流程**：
1. 团队识别延期风险
2. 向EMP-020申请使用缓冲时间
3. EMP-020评估并批准
4. 团队使用缓冲时间
5. 每周汇报缓冲时间使用情况

## 8. 应急预案

### 8.1 延期应急预案

**轻度延期（1-2天）**：
- 使用缓冲时间吸收
- 团队加班赶进度
- 不调整整体计划

**中度延期（3-5天）**：
- 使用缓冲时间
- 调整后续计划
- 压缩非关键路径时间
- 增加资源（如有可能）

**重度延期（>5天）**：
- 启动应急预案
- 重新评估整体计划
- 可能需要延长总时间或削减范围
- 上报并寻求支持

### 8.2 加速预案

如某子阶段提前完成，可以：
1. 提前启动下游子阶段
2. 增加质量检查时间
3. 补充文档和测试
4. 为后续子阶段提供更多支持
