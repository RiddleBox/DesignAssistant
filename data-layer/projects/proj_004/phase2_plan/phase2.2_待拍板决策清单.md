## 一、为什么要先补这份清单

在 `Phase 2.2` 当前阶段，我们已经完成了两类前置工作：

- `2.2` 的第一性原理与岗位本质收口；
- `2.2 MVP` 范围与后续迭代边界的初步收口。

这意味着当前已经不是“继续泛讨论 `2.2` 要做什么”的阶段，而是进入了更具体的一步：

> **把真正会影响启动方向、设计边界、协作方式和后续实现顺序的关键问题，显式整理成待拍板决策清单。**

如果缺少这份文档，`2.2` 后续非常容易出现以下问题：

- 另一端接手时仍按旧理解，把 `2.2` 做成“评估报告模块”或“多 Agent 辩论器”；
- 设计阶段一边推进，一边继续修改模块边界，导致反复返工；
- 本来应该用户明确拍板的问题，被执行端在实现中默认决定；
- 当前该先做的对象契约、判断闭环，被多 Agent、复杂评分或长报告模板分散注意力。

因此，这份清单的目的不是为了把所有问题一次性讨论完，而是为了先固定：

- 哪些问题**必须现在定**；
- 哪些问题**本周最好定**；
- 哪些问题**可以条件性后置**；
- 哪些问题**在未拍板前不得被执行端自行扩张**。

## 三、当前拍板的基本原则

基于现有定位文档与范围文档，`2.2` 当前的拍板应遵循以下原则：

### 3.1 先拍板对象与边界，再拍板增强项

当前最重要的不是先决定：

- 要不要做完整多 Agent；
- 报告写成几段；
- 是否做复杂权重评分。

当前最重要的是先决定：

- `2.2` 消费什么输入；
- `2.2` 产出什么主对象；
- `2.2` 当前最小闭环是什么；
- `2.2` 与 `2.1 / 2.4 / 2.3` 的职责边界在哪里。

### 3.2 先拍板 `MVP` 必做项，再拍板 `P1 / P2`

当前阶段必须严格区分：

- 没有它，`2.2 MVP` 就无法成立的内容；
- 有它更好，但不应作为启动前置的内容。

### 3.3 拍板项必须能约束后续设计与实现

每个待拍板项都不应只是“抽象讨论题”，而应能直接影响：

- 另一端后续如何出设计方案；
- 方案中哪些内容可以先做；
- 哪些内容必须暂缓；
- 未来验收时用什么口径判断是否偏航。

### 3.4 未拍板内容不得被默认扩大解释

如果某个问题仍未拍板，则默认：

- 不进入当前强制范围；
- 不由执行端自行扩展解释；
- 不在设计文档中被写成既定结论。

## 八、当前不建议默认推进的事项

为了防止另一端在未拍板前直接沿旧思路开工，建议明确写出当前不建议默认推进的方向。

### 8.1 不建议默认把 `2.2` 直接做成报告生成模块

原因：
- 这会削弱结构化机会对象作为主产物的地位；
- 会让 `2.2` 更像展示层而不是判断层；
- 会削弱后续 `2.3` 的稳定消费接口。

### 8.2 不建议默认把完整多 Agent 作为当前设计起点

原因：
- 会把注意力从对象契约和判断骨架转移到 Agent 角色与编排细节；
- 会显著增加归因难度；
- 会让 MVP 范围在启动前就膨胀。

### 8.3 不建议默认把 `2.2` 与 `2.4` 做强耦合

原因：
- 当前 `2.4` 更适合作为证据包 / context packet 的供应层；
- 如果过早强耦合，后续 `2.4` 一旦调整，`2.2` 会被迫反复返工。

### 8.4 不建议默认让 `2.2` 越级覆盖 `2.3`

原因：
- 会模糊整个 Phase 2 链路的分工；
- 会导致“机会判断”和“行动设计”混在一起，后续模块边界难以维护。

为了让拍板更稳，建议不要把所有问题混在一起一次性讨论，而是按顺序推进。

5. 多 Agent 在 `2.2 MVP` 中的地位 = `A`
6. `2.2` 与 `2.3` 的当前边界 = `A`

### 9.2 第二轮再拍板

在第一轮结论明确后，再定：

1. 输出字段骨架
2. 优先级分级口径
3. 轻量评分是否进入 `MVP`
4. 首轮案例验证强度
5. 首轮设计方案的正式产物范围

### 9.3 第三轮按验证结果再决定

后续再根据首轮方案与验证情况决定：

1. 多 Agent 的具体实现形态
2. 是否进入复杂评分体系
3. 是否补完整报告模板
4. 是否把多 Agent 重点展示放在 `2.5`
