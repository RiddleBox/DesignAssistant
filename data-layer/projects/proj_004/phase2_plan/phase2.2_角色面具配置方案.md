# Phase 2.2 角色面具配置方案

> **文档类型**：执行视角配置文档

> **文档目标**：为 `2.2` 在同一 Agent 下的协作执行提供一套清晰、可压缩、可落地的角色面具配置方案，确保后续方案设计、验证和实现能够在明确的视角分工下进行。  
> **最后更新**：2026-03-15

## 一、文档定位说明

### 1.1 这份文档解决什么问题

这份文档回答的是：

- `2.2` 需要哪些执行视角来完成当前阶段工作
- 每个角色面具分别负责什么输入、输出和判断任务
- 这些角色在同一 Agent 内如何协作，而不是彼此打架或互相覆盖
- 后续另一端接手细方案时，应该如何使用这套角色视角

### 1.2 这份文档不解决什么问题

这份文档**不**负责以下内容：

- 不决定是否需要团队重组
- 不定义宏观阶段责任归属
- 不替代 [phase2.2_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.2_团队重组建议清单.md)
- 不把角色面具误写成运行时多 Agent 架构设计
- 不在当前阶段为每个角色直接写出最终 Prompt 成品

### 1.3 与团队重组建议清单的差异

- **团队重组建议清单**：组织层，回答“`2.2` 正式需要覆盖哪些职责视角，这些职责为什么必须存在，以及它们在阶段推进中如何构成正式团队结构”
- **角色面具配置方案**：执行层，回答“同一 Agent 内部应如何把这些正式职责转成可切换、可压缩、可协作的执行视角”

因此，这份文档会重点写：

- 角色定义
- 输入输出
- 使用时机
- 协作顺序
- 冲突解决机制
- **正式职责如何映射为执行面具**

而不会重复写大量组织配置内容。

### 1.4 与 `phase2_roles/phase2.2_roles.md` 的关系

这份文档还需要和另一份正式角色文件区分开：

- **[phase2.2_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.2_团队重组建议清单.md)**：给出 `2.2` 的**正式职责来源**
- **本文件**：说明这些正式职责在**同一 Agent 下如何执行化、如何压缩、如何协作**
- **`phase2_roles/phase2.2_roles.md`**：由另一端基于前两份文档正式落档，作为后续设计方案与实现阶段的执行角色依据

换句话说：

> **本文件不是最终角色落档文件，而是正式角色落档前的执行化桥接文档。**

## 二、配置总原则

### 2.1 核心原则

`2.2` 当前采用的是：

> **同一 Agent 下的角色面具协作模式，而不是多 Agent 自治模式。**

这意味着：

- 所有角色面具共享同一总体目标
- 角色面具之间是职责切换，不是互相竞争的独立智能体
- 角色面具的价值在于提高设计质量与判断完整度
- 角色面具当前服务于 `2.2 MVP` 收口，不服务于“做一个看起来很复杂的多 Agent showcase” [[memory:rebp86gg]]

### 2.2 当前配置目标

本轮角色面具配置要服务于四件事：

- **帮助 `2.2` 守住模块边界**
- **帮助 `2.2` 建立结构化机会对象**
- **帮助 `2.2` 形成可解释的判断闭环**
- **帮助另一端在接手时有统一执行视角**

### 2.3 当前配置约束

- 不以完整多 Agent 辩论为前提
- 不让角色面具替代用户拍板
- 不让角色面具绕开治理文档自行扩范围
- 不在 `MVP` 未稳之前设计过多“炫技型”视角

### 2.4 判断层级边界说明

为避免与 `2.1`、`2.3` 发生语义混淆，本文件中的“判断”一词也采用与团队重组文档一致的层级定义：

- **`2.1 = 信号级准入判断`**：判断外部事实是否能成为正式信号
- **`2.2 = 机会级判断 / 机会升级判断`**：判断一组信号与证据是否能构成值得继续跟踪、深挖或升级的机会对象
- **`2.3 = 行动级判断`**：判断基于机会对象应该采取什么行动、投入什么资源、进入什么决策路径

因此，当前角色面具中的：

- **机会判断面具**，处理的是**机会级判断**，不是 `2.1` 的信号识别
- **结构化契约面具**，服务的是**机会对象及其下游交接契约**，不是 `2.1` 的抽取 Schema
- **方案设计 / 边界收口面具**，守的是 `2.2` 的模块边界，不是替 `2.3` 先做行动方案

## 三、推荐角色面具总览

### 3.1 标准配置

建议 `2.2` 标准配置采用 `6-7` 个角色面具：

1. **总协调面具**
2. **方案设计 / 边界收口面具**
3. **机会判断面具**
4. **结构化契约面具**
5. **证据整合面具**
6. **验证验收面具**
7. **实现落地面具**

### 3.2 压缩配置

如果执行阶段需要压缩，可合并为 `5-6` 个角色面具：

- 总协调面具
- 方案设计 / 边界收口 + 结构化契约面具
- 机会判断面具
- 证据整合面具
- 验证验收面具
- 实现落地面具

### 3.3 角色面具使用原则

- 并不是每一轮都需要所有角色同时出场
- 当前任务只调用必要视角，避免形式化空转
- 每个角色面具输出都应服务于某个明确产物
- 一旦进入实现阶段，应减少高层抽象角色的频繁介入

## 四、角色面具详细定义

## 4.1 总协调面具

### 角色使命

负责在每一轮任务中确认当前目标、调用顺序、输入前提和结束条件，避免团队视角在细节中失焦。

### 关注问题

- 当前这一轮到底要产出什么
- 需要哪些前置文档先对齐
- 哪些问题属于当前阶段，哪些应后置
- 当前应该调用哪些角色，而不是把所有角色都拉进来

### 典型输入

- 当前用户目标
- 当前阶段文档
- 已拍板事项
- 未拍板事项

### 典型输出

- 本轮任务分解
- 参与角色列表
- 执行顺序
- 停止条件与交付物定义

### 不应该做的事

- 不直接替代机会判断面具下结论
- 不直接设计字段细节
- 不直接充当实现面具写实现方案

---

## 4.2 边界收口面具

### 角色使命

负责持续检查 `2.2` 是否仍然站在“机会判断层”边界内，避免设计向 `2.1`、`2.3` 或展示型多 Agent 偏移。

### 关注问题

- 当前设计是否还在 `2.2` 范围内
- 是否出现职责上行或下探
- 是否误把未来增强项纳入当前 `MVP`
- 是否因为追求复杂度而牺牲主线

### 典型输入

- 第一性原理文档
- `MVP Scope` 文档
- 待拍板清单
- 当前方案草案

### 典型输出

- 边界检查结论
- 非目标提醒
- 当前设计是否越界的判定
- 是否需要回到用户拍板的建议

### 不应该做的事

- 不替代总协调面具安排全局流程
- 不负责具体字段设计
- 不因为“更完整”而主动扩范围

---

## 4.3 机会判断面具

### 角色使命

负责定义 `2.2` 的核心对象与核心判断逻辑，把“信号”组织成“机会对象”，并给出可解释的判断骨架。

### 关注问题

- 一组信号为什么能构成一个机会主题
- 当前机会论点是什么
- 支持与反对证据怎样组织
- 优先级与升级建议如何表达
- 哪些不确定性必须明确写出

### 典型输入

- `2.1` 输出的结构化信号
- `2.4` 提供的轻量证据包
- 当前机会对象 Schema 草案
- 分级口径草案

### 典型输出

- 机会对象骨架
- 论点与证据结构
- 分级建议草案
- 下一步验证问题清单

### 不应该做的事

- 不直接负责接口字段命名规范
- 不直接替代 `2.3` 产出行动方案
- 不脱离证据只给感觉型判断

---

## 4.4 结构化契约面具

### 角色使命

负责把机会判断逻辑转换成稳定可消费的输入 / 输出契约，确保 `2.2` 可以被下游稳定接入，而不是停留在抽象描述层。

### 关注问题

- `2.2` 的请求对象应该长什么样
- `OpportunityObject` 最小字段是什么
- 哪些字段是正式字段，哪些只是辅助字段
- `2.2 -> 2.3` 的交接需要哪些最小信息

### 典型输入

- 启动与拍板文档中的契约草案
- 机会判断面具输出
- 上下游模块边界说明

### 典型输出

- 输入 Schema 草案
- 输出 Schema 草案
- 字段注释与消费说明
- 示例对象样本

### 不应该做的事

- 不直接判断机会是否成立
- 不把契约设计成过度依赖 `2.4` 的强耦合结构
- 不在当前阶段过早设计复杂版本管理体系

---

## 4.5 证据整合面具

### 角色使命

负责检查 `2.2` 的判断是否有足够证据支撑，并把 `2.4` 提供的内容正确地放到“支持、反对、边界提醒、限制条件”中。

### 关注问题

- 当前机会对象的证据是否足够
- 哪些证据只是背景，不应被误写成正式支撑
- 哪些限制条件需要显式提醒
- 当前是否存在过度自信或证据边界不清的问题

### 典型输入

- `2.4` 证据包
- 机会对象草案
- 支持 / 反对证据列表

### 典型输出

- 证据补充建议
- 反证与限制条件补充
- 风险提示
- 是否应降低判断强度的建议

### 不应该做的事

- 不替代机会判断面具形成主论点
- 不把 `2.4` 的信息直接上升为最终判断结论
- 不把弱证据包装成强证据

---

## 4.6 验证验收面具

### 角色使命

负责从“这个东西是否真的可用”的角度审视当前方案，而不是只看结构是否完整。

### 关注问题

- 输出对象是否稳定合法
- 人类是否能理解这个机会对象
- 下游 `2.3` 是否真的可消费
- 当前样例验证是否足以支撑继续推进
- 是否已经具备进入下一阶段的条件

### 典型输入

- 当前 Schema 草案
- 当前样例对象
- 分级结果
- 验证方案与案例

### 典型输出

- 验收检查表
- 验证建议
- 典型失败模式清单
- 是否建议继续推进的结论

### 不应该做的事

- 不负责主流程设计
- 不因为少数失败样例就直接推翻整个模块定位
- 不直接进入实现细节替代实现面具

---

## 4.7 实现落地面具

### 角色使命

负责把已经基本稳定的契约与流程方案转化为可执行实现路径，并在运行后把结果回写到治理链路中。

### 关注问题

- 哪些部分已经足够稳定，可以开始做
- 先实现哪条最小路径
- 如何组织样例输入输出
- 运行结果要回写到哪些文档中

### 典型输入

- 已拍板的边界与契约
- 设计方案
- 样例验证要求

### 典型输出

- 实现拆解建议
- 样例运行记录
- 风险与返工点
- 回写建议

### 不应该做的事

- 不在拍板未完成时反向决定边界
- 不自行新增当前 `MVP` 之外的大功能
- 不把实现方便性当成模块本质

## 五、推荐协作顺序

### 5.1 设计期推荐顺序

在当前治理与方案阶段，推荐协作顺序如下：

```text
总协调面具
→ 边界收口面具
→ 机会判断面具
→ 结构化契约面具
→ 证据整合面具
→ 验证验收面具
→ （条件满足后）实现落地面具
```

### 5.2 为什么这样排序

这个顺序确保：

- 先判断“这轮要做什么”
- 再判断“可不可以这样做”
- 再形成“应该输出什么对象”
- 再把对象变成“可消费契约”
- 再补足证据完整性
- 最后再检查“是否真可用”

而不是一开始就从实现或提示词入手。

## 六、不同阶段的角色调用策略

### 6.1 治理收口期

重点调用：

- 总协调面具
- 边界收口面具
- 机会判断面具
- 结构化契约面具

暂时弱化：

- 实现落地面具

### 6.2 方案设计期

重点调用：

- 总协调面具
- 机会判断面具
- 结构化契约面具
- 证据整合面具
- 验证验收面具

### 6.3 实现与验证期

重点调用：

- 实现落地面具
- 验证验收面具
- 结构化契约面具

按需调用：

- 边界收口面具
  - 仅在出现范围漂移时介入

## 七、冲突处理规则

### 7.1 常见冲突类型

- **边界收口面具**认为当前方案越界，但**实现落地面具**希望继续推进
- **机会判断面具**希望增加表达能力，但**结构化契约面具**希望保持对象简洁
- **证据整合面具**认为证据不足，但**验证验收面具**认为当前仍可先做轻量验证

### 7.2 解决原则

按以下优先级处理：

1. **模块边界优先于实现便利**
2. **正式对象优先于表达花样**
3. **验证可用性优先于理论完整性**
4. **用户拍板优先于角色面具内部偏好**

### 7.3 必须回到用户拍板的情况

以下情况必须中止内部收敛，转回用户确认：

- 需要修改 `2.2` 与 `2.3` 边界
- 需要把多 Agent 升级为当前 `MVP` 的硬要求
- 需要将 `2.4` 从增强依赖改为强依赖
- 需要大幅改变主产物形态

## 八、另一端接手时的使用方式

另一端后续接手时，不应把这份文档理解为“必须每轮都演完整角色戏”，而应理解为：

- 在做设计时，知道当前需要从哪些视角检查方案
- 在写方案时，知道哪些判断应由哪个角色面具负责
- 在发现冲突时，知道该按什么顺序处理
- 在进入实现时，知道什么时候应减少高层角色，更多交给实现与验证视角

### 8.1 最推荐的使用方法

- 先用总协调面具确认任务边界
- 用边界收口面具确认是否越界
- 用机会判断面具和结构化契约面具搭建核心方案
- 用证据整合面具和验证验收面具做复查
- 条件满足后，再调用实现落地面具推进

### 8.2 最不推荐的使用方法

- 把所有角色一次性拉满并要求同时输出
- 让角色面具彼此辩论但没有明确产物
- 在角色尚未收敛前直接写实现
- 把角色面具文档误当成运行时多 Agent 系统设计稿

## 九、配置结论

### 9.1 当前建议

- `2.2` 当前适合采用 **`6-7` 个标准角色面具 / `5-6` 个压缩角色面具** 的方案
- 当前角色面具配置应服务于**设计质量与边界控制**，而不是为了展示复杂性
- 角色面具的核心价值在于让另一端接手时，能够按稳定视角做高质量方案设计与实现

### 9.2 一句话总结

> `Phase 2.2` 的角色面具配置，不是为了把项目包装成多 Agent 系统，而是为了让同一 Agent 在当前 `MVP` 范围内，能以更清晰的视角分工完成边界控制、机会判断、契约设计、证据整合、验证和实现推进。
