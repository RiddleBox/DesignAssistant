## 一、文档定位说明

### 1.1 这份文档解决什么问题

这份文档回答的是：

- `2.5` 需要哪些执行视角来完成当前阶段工作
- 每个角色面具分别负责什么输入、输出和判断任务
- 这些角色在同一 Agent 内如何协作，而不是互相覆盖、互相抢职责或演变成无序讨论
- 后续另一端接手细方案时，应该如何使用这套角色视角
- 正式职责清单如何被转译成执行期可切换、可压缩、可协作的面具体系

### 1.2 这份文档不解决什么问题

这份文档**不**负责以下内容：

- 不决定是否需要团队重组
- 不重新定义 `2.5` 的宏观阶段责任归属
- 不替代 [phase2.5_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.5_团队重组建议清单.md)
- 不把角色面具误写成运行时多 Agent 架构设计
- 不在当前阶段为每个角色直接写出最终 Prompt 成品
- 不越俎代庖替另一端先写完整 [phase2.5_设计方案.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.5_设计方案.md)
- 不绕过两次拍板门禁直接替实现阶段做超前承诺

### 1.3 与团队重组建议清单的差异

- **团队重组建议清单**：组织层，回答“`2.5` 正式需要覆盖哪些职责视角，这些职责为什么必须存在，以及它们在阶段推进中如何构成正式团队结构”
- **角色面具配置方案**：执行层，回答“同一 Agent 内部应如何把这些正式职责转成可切换、可压缩、可协作的执行视角”

因此，这份文档会重点写：

- 冲突解决机制
- 正式职责如何映射为执行面具

而不会重复写大量组织配置内容。

### 1.4 与 `phase2_roles/phase2.5_roles.md` 的关系

这份文档还需要和另一份正式角色文件区分开：

- **[phase2.5_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.5_团队重组建议清单.md)**：给出 `2.5` 的**正式职责来源**

## 二、配置总原则

### 2.1 核心原则

`2.5` 当前采用的是：

> **同一 Agent 下的角色面具协作模式，而不是多 Agent 自治模式。**

这意味着：

- 所有角色面具共享同一总体目标
- 角色面具之间是职责切换，不是互相竞争的独立智能体
- 角色面具的价值在于提高设计质量、归因完整度、回写稳定性与验证可信度
- 角色面具当前服务于 `2.5 MVP` 的现实校验闭环收口，而不是为了“做一个看起来很复杂的多 Agent showcase” [[memory:rebp86gg]]

### 2.2 当前配置目标

本轮角色面具配置要服务于六件事：

- **帮助 `2.5` 守住模块边界**
- **帮助 `2.5` 建立结构化系统复盘对象**
- **帮助 `2.5` 把真实链路问题组织成可解释的归因结构**
- **帮助 `2.5` 把关键发现稳定收口为阶段3优先级**
- **帮助 `2.5` 形成可拍板、可验证、可被后续联调消费的输出**
- **帮助另一端在接手时有统一执行视角**

### 2.3 当前配置约束

- 不以完整多 Agent 辩论为前提
- 不让角色面具替代用户拍板
- 不让角色面具绕开治理文档自行扩范围
- 不在 `MVP` 未稳之前设计过多“炫技型”视角
- 不把角色面具误当成当前产品运行时系统形态的预设承诺
- 不让角色面具把 `2.5` 从“现实校验层 / 闭环学习层”带回“整合测试收尾器”或“长篇复盘报告器”

### 2.4 职责层级边界说明

为避免与 `2.1 ~ 2.4` 发生语义混淆，本文件中的“验证”“复盘”“归因”“收口”采用与团队重组文档一致的层级定义：

- **`2.1 = 信号级对象形成`**：把外部材料整理成结构化信号
- **`2.2 = 机会级判断`**：把信号与证据组织成结构化机会对象
- **`2.3 = 行动级判断 / 资源承诺设计`**：把机会对象组织成结构化行动对象
- **`2.4 = 上下文与支撑能力层`**：提供证据、模板、方法、约束与辅助上下文
- **`2.5 = 现实校验与闭环归因`**：把上游对象压入真实案例与真实流程，验证系统级效果、形成结构化系统复盘对象，并回写阶段3优先级与上游优化方向

因此，当前角色面具中的：

- **现实校验面具**，处理的是**真实链路运行、检查与记录策略**，不是泛化测试工作
- **系统复盘对象面具**，处理的是**正式主产物对象骨架**，不是长文复盘表达
- **问题归因面具**，处理的是**系统问题的层级化解释结构**，不是感受型总结
- **阶段3收口面具**，处理的是**从复盘对象导出优先级项**，不是另起一个独立规划系统
- **结构化契约面具**，服务的是**`2.5` 输入 / 输出契约与联调骨架**，不是提前把所有协议工程化做满

## 三、推荐角色面具总览

### 3.1 标准配置

建议 `2.5` 标准配置采用 `8-9` 个角色面具：

1. **总协调面具**
2. **方案设计 / 边界收口面具**
3. **现实校验面具**
4. **系统复盘对象面具**
5. **问题归因面具**
6. **阶段3收口面具**
7. **验证验收面具**
8. **结构化契约面具**
9. **实现落地面具**

### 3.2 压缩配置

如果执行阶段需要压缩，可合并为 `5-6` 个角色面具：

- 总协调面具
- 方案设计 / 边界收口面具
- 现实校验 + 验证验收面具
- 系统复盘对象 + 结构化契约面具
- 问题归因 + 阶段3收口面具
- 实现落地面具

### 3.3 角色面具使用原则

- 并不是每一轮都需要所有角色同时出场
- 当前任务只调用必要视角，避免形式化空转
- 每个角色面具输出都应服务于某个明确产物
- 一旦进入实现阶段，应减少高层抽象角色的频繁介入
- 在设计未收敛前，角色面具优先服务于方案收敛，而不是服务于“角色表演”
- 在首轮设计前冻结拍板前，角色面具主要服务于**冻结边界、主产物形态、最小闭环与设计范围**
- 在设计拍板通过前，角色面具不得擅自把复杂观测、多 Agent 工程化、完整自动归因平台提前塞入当前 `MVP`

## 四、角色面具详细定义

## 4.1 总协调面具

### 角色使命

负责在每一轮任务中确认当前目标、调用顺序、输入前提和结束条件，避免团队视角在细节中失焦。

### 关注问题

- 当前这一轮到底要产出什么
- 需要哪些前置文档先对齐
- 哪些问题属于当前阶段，哪些应后置
- 当前应该调用哪些角色，而不是把所有角色都拉进来
- 当前处在设计前冻结拍板前、设计期，还是实现与验证期

### 典型输入

- 当前用户目标
- 当前阶段文档
- 已拍板事项
- 未拍板事项
- 当前设计阶段位置

### 典型输出

- 本轮任务分解
- 参与角色列表
- 执行顺序
- 停止条件与交付物定义
- 是否需要转回拍板的判断

### 不应该做的事

- 不直接替代现实校验面具或问题归因面具下结论
- 不直接定义字段细节
- 不直接充当实现面具写实现方案

---

## 4.2 方案设计 / 边界收口面具

### 角色使命

负责持续检查 `2.5` 是否仍然站在“现实校验层 / 闭环学习层”边界内，并把多视角输入收敛为可拍板的正式方案骨架。

### 关注问题

- 当前设计是否还在 `2.5` 范围内
- 是否出现职责上行或下探
- 是否误把未来增强项纳入当前 `MVP`
- 现实校验、复盘对象、问题归因、阶段3收口、验证与契约是否已形成统一方案骨架
- 当前分歧应在设计内部收敛，还是必须回到用户拍板

### 典型输入

- 第一性原理文档
- `MVP Scope` 文档
- 待拍板清单
- 启动与拍板文档
- 当前方案草案

### 典型输出

- 边界检查结论
- 方案骨架草案
- 非目标提醒
- 是否需要回到用户拍板的建议

### 不应该做的事

- 不替代总协调面具安排全局流程
- 不独占所有角色的判断权
- 不因为“更完整”而主动扩范围

---

## 4.3 现实校验面具

### 角色使命

负责定义 `2.5` 的真实链路如何运行、检查与记录，确保当前设计首先成立为“现实校验层”，而不是停留在抽象复盘层或事后报告层。

### 关注问题

- 当前最小真实链路应该怎样跑
- 需要记录哪些最小运行信息，才足以支持后续复盘与归因
- 首轮真实案例应如何选择
- 哪些输出检查必须纳入 `MVP`
- 当前是否需要保留轻量人工复核，以及保留到什么程度

### 典型输入

- 上游 `2.1 ~ 2.4` 的关键输出
- 真实案例上下文
- 最小运行记录要求
- 启动文档中的闭环定义

### 典型输出

- 现实校验主流程草案
- 运行记录要求
- 首轮真实案例策略
- 输出检查维度建议

### 不应该做的事

- 不回卷重做 `2.1 ~ 2.4` 的对象逻辑
- 不把现实校验扩大成完整测试平台设计
- 不把“能跑通”直接等同为“系统可信”

---

## 4.4 系统复盘对象面具

### 角色使命

负责定义 `2.5` 的正式主产物，即 `SystemRetrospectiveObject` 的对象骨架、字段层次与派生视图边界，确保 `2.5` 产出的是结构化对象，而不是长篇复盘文本。

### 关注问题

- `SystemRetrospectiveObject` 的最小字段是什么
- 哪些字段是正式主字段，哪些只是辅助摘要
- 复盘对象如何体现关键发现、归因、优先级与可信度说明
- 对象如何既可被人理解，也可被后续联调和回写消费
- 哪些视图可以后置为派生报告，而不应反向定义主对象

### 典型输入

- 启动与拍板文档中的契约草案
- 现实校验面具输出
- 问题归因面具输出
- 阶段3收口面具输出

### 典型输出

- 复盘对象骨架
- 字段分层说明
- 派生视图边界说明
- 示例复盘对象草案

### 不应该做的事

- 不把正式主产物重新写成复盘长文
- 不只从展示便利性倒推对象结构
- 不在主对象未稳时先设计过多派生报告形态

---

## 4.5 问题归因面具

### 角色使命

负责把当前真实链路中暴露出的问题组织成可解释、可追踪、可回写的层级化归因结构，避免 `2.5` 只停留在“看到问题现象”的层面。

### 关注问题

- 当前问题更可能属于哪一层：信号、机会、行动、编排、验证或输入约束
- 证据是否足以支持当前归因强度
- 哪些归因只是怀疑，哪些可以写成相对稳定的判断
- 如何表达不确定性、限制条件和可信度
- 当前归因是否足以支撑阶段3优先级收口

### 典型输入

- 现实校验结果
- 输出检查结果
- 运行异常与人工复核信息
- 上游对象与流程记录

### 典型输出

- 归因结构草案
- 证据与限制说明
- 归因可信度表达
- 需要补证据的问题清单

### 不应该做的事

- 不把问题现象直接包装成确定性结论
- 不用抽象话术替代层级归因
- 不在证据不足时输出过度自信的根因判断

---

## 4.6 阶段3收口面具

### 角色使命

负责把 `2.5` 发现的关键问题和初步归因，收束为阶段3可消费的优先级项与排序建议，确保 `2.5` 真正成立为“闭环学习层”。

### 关注问题

- 哪些发现值得进入阶段3优先级
- 优先级项应该如何命名、排序和解释
- 应该按模块级、系统级还是工作流级来表达范围
- 当前收口项是否与上游改进动作有清晰映射
- 如何避免把问题列表原封不动搬进阶段3

### 典型输入

- 关键发现列表
- 问题归因结构
- 可信度说明
- 当前 `MVP` 的收口要求

### 典型输出

- `Phase3PriorityItem` 草案
- 收口逻辑说明
- 优先级排序建议
- 后续优化方向提示

### 不应该做的事

- 不另起一个完整阶段3规划系统
- 不把所有问题都平铺成优先级项
- 不脱离归因直接拍脑袋给排序

---

## 4.7 验证验收面具

### 角色使命

负责从“这个复盘对象是否真的可用、可解释、可回写、可拍板”的角度审视当前方案，而不是只看结构是否完整。

### 关注问题

- 输出对象是否稳定合法
- 人类是否能理解当前复盘对象与优先级结果
- 上游与阶段3是否真的可消费这些信息
- 当前案例验证是否足以支撑继续推进
- 是否已经具备进入设计拍板或正式实现的条件

### 典型输入

- 当前契约草案
- 当前样例对象
- 当前案例验证方案
- 运行记录与人工复核结果

### 典型输出

- 验收检查表
- 验证建议
- 典型失败模式清单
- 是否建议继续推进的结论

### 不应该做的事

- 不负责主流程设计
- 不因为少数失败样例就直接推翻整个模块定位
- 不直接进入实现细节替代实现面具

---

## 4.8 结构化契约面具

### 角色使命

负责把现实校验、系统复盘、问题归因与阶段3收口逻辑转换成稳定可消费的输入 / 输出契约，确保 `2.5` 可以被后续联调、验证和回写稳定接入，而不是停留在抽象描述层。

### 关注问题

- `SystemRetrospectiveRequest` 应该长什么样
- `SystemRetrospectiveObject` 的最小字段是什么
- `Phase3PriorityItem` 应如何表达最小收口信息
- 哪些字段是正式字段，哪些只是辅助字段
- 当前应冻结到什么程度，哪些协议细节应该后置

### 典型输入

- 启动与拍板文档中的契约草案
- 现实校验 / 复盘对象 / 归因 / 收口面具输出
- 上下游模块边界说明

### 典型输出

- 输入 Schema 草案
- 输出 Schema 草案
- 字段注释与消费说明
- 示例输入输出对象

### 不应该做的事

- 不直接判断归因是否成立
- 不把契约设计成过度依赖 `2.4` 的强耦合结构
- 不在当前阶段过早设计复杂协议治理体系

---

## 4.9 实现落地面具

### 角色使命

负责把已经基本稳定的契约与流程方案转化为可执行实现路径，并在运行后把结果回写到治理链路中。

### 关注问题

- 哪些部分已经足够稳定，可以开始做
- 先实现哪条最小路径
- 如何组织样例输入输出
- 运行结果要回写到哪些文档中
- 当前实现是否仍在已拍板的 `MVP` 范围内

### 典型输入

- 已拍板的边界与契约
- 设计方案
- 样例验证要求
- 执行进度文档要求

### 典型输出

- 实现拆解建议
- 样例运行记录
- 风险与返工点
- 回写建议

### 不应该做的事

- 不在拍板未完成时反向决定边界
- 不自行新增当前 `MVP` 之外的大功能
- 不把实现方便性当成模块本质

## 五、正式职责与执行面具的映射关系

### 5.1 映射原则

本节的作用，是帮助另一端在建立 `phase2_roles/phase2.5_roles.md` 时，不会把“正式职责视角”和“执行期角色面具”混成两套互相竞争的命名系统。

### 5.2 映射表

| 正式职责视角 | 对应执行面具 | 使用说明 |
|--------------|--------------|----------|
| **总协调 / 架构连续性视角** | **总协调面具** | 作为全局收口与跨阶段连续性的正式职责来源 |
| **方案设计负责人** | **方案设计 / 边界收口面具** | 在执行层负责方案收敛、边界检查与取舍整理 |
| **现实校验负责人** | **现实校验面具** | 对应 `2.5` 的真实链路运行与检查主路径 |
| **系统复盘对象负责人** | **系统复盘对象面具** | 负责正式主产物对象骨架 |
| **问题归因负责人** | **问题归因面具** | 负责归因层次、可信度与限制表达 |
| **阶段3收口 / 优先级负责人** | **阶段3收口面具** | 负责优先级收口、排序建议与后续消费表达 |
| **验证与验收负责人** | **验证验收面具** | 负责轻量验证、验收检查与可推进判断 |
| **结构化契约负责人** | **结构化契约面具** | 负责对象 Schema 与联调骨架 |
| **实现落地工程师视角** | **实现落地面具** | 负责从已拍板方案进入可运行实现 |

### 5.3 如何理解 `结构化契约面具`

这里要特别说明：

- 在 `2.5` 当前治理体系里，**正式职责来源仍优先以团队重组建议清单为准**
- 但从执行层看，`2.5` 又确实需要一个专门处理输入输出契约、字段骨架与联调稳定性的专项视角
- 因此，本文件将 **`结构化契约面具` 定义为执行层专项支撑面具**，用于帮助另一端把现实校验闭环与复盘对象契约做稳
- 后续另一端在建立 `phase2_roles/phase2.5_roles.md` 时，可以将其：
  - 作为独立支撑角色保留；或
  - 归并到 **系统复盘对象面具** 与 **验证验收面具** 的职责组合中

**推荐口径**：

- 标准配置：保留 `结构化契约面具`
- 压缩配置：将其并入“系统复盘对象 + 结构化契约面具”或“方案设计 / 边界收口面具”

### 5.4 落档优先级说明

1. `phase2_roles/phase2.5_roles.md` 的正式职责命名，优先以 [phase2.5_团队重组建议清单.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.5_团队重组建议清单.md) 为准。
2. 各职责在同一 Agent 下如何协作、何时调用、如何压缩执行，以本文件为准。
3. 如果角色命名或职责颗粒度出现差异，应先回到团队重组文档确认正式职责，再同步修订执行层角色映射。

## 六、推荐协作顺序

### 6.1 设计期推荐顺序

在当前治理与方案阶段，推荐协作顺序如下：

```text
总协调面具
→ 方案设计 / 边界收口面具
→ 现实校验面具
→ 系统复盘对象面具
→ 问题归因面具
→ 阶段3收口面具
→ 结构化契约面具
→ 验证验收面具
→ （条件满足后）实现落地面具
```

### 6.2 为什么这样排序

这个顺序确保：

- 先判断“这轮要做什么”
- 再判断“可不可以这样做”
- 再定义“最小真实链路如何运行、检查与记录”
- 再形成“正式主产物对象长什么样”
- 再解释“问题为什么发生、可信度如何表达”
- 再收束“哪些结论值得进入阶段3优先级”
- 再把方案转换成可消费对象契约
- 最后再检查“是否真可拍板、可验证、可实现”

而不是一开始就从协议细节、长报告写法或实现便利性入手。

## 八、冲突处理规则

### 8.1 常见冲突类型

- **方案设计 / 边界收口面具**认为当前方案越界，但**实现落地面具**希望继续推进
- **现实校验面具**希望补更多记录与样本，但**验证验收面具**认为当前轻量验证已足够先行
- **系统复盘对象面具**希望补更多字段表达能力，但**结构化契约面具**希望保持主对象简洁
- **问题归因面具**认为证据仍不足，但**阶段3收口面具**希望先形成优先级建议
- **验证验收面具**认为样例仍不可拍板，但**实现落地面具**希望先跑通最小流程

### 8.2 解决原则

按以下优先级处理：

1. **模块边界优先于实现便利**
2. **正式主对象优先于表达花样**
3. **可拍板性优先于叙述完整性**
4. **验证可用性优先于理论完美性**
5. **用户拍板优先于角色面具内部偏好**

### 8.3 必须回到用户拍板的情况

以下情况必须中止内部收敛，转回用户确认：

- 需要修改 `2.5` 与 `2.1 ~ 2.4` 的边界
- 需要把多 Agent 升级为当前 `MVP` 的硬要求
- 需要把复杂观测平台、自动归因平台或大规模案例体系提前纳入当前 `MVP`
- 需要大幅改变主产物形态
- 需要让 `2.5` 越级承担上游对象重构职责
- 需要修改两次拍板的顺序、作用或进入条件

## 九、与两次拍板门禁的关系

### 9.1 与设计前冻结拍板的关系

在第一轮拍板前，这套角色面具的主要作用不是直接产出完整设计，而是帮助冻结以下内容：

- `2.5` 的主产物形态是否固定为**结构化系统复盘对象**
- `2.5 MVP` 是否固定为**真实链路运行 → 输出检查 → 问题归因 → 阶段3优先级收口**
- `2.5` 与 `2.1 ~ 2.4` 的边界是否保持稳定
- 多角色机制是否固定定位为：**设计阶段的方法论与正式 `roles` 落档依据**
- 首轮设计范围、案例规模、人工复核口径是否不带分歧进入另一端设计阶段

也就是说：

> **设计前冻结拍板前，角色面具主要用于“冻结设计入口”，而不是“提前完成设计”。**

### 9.2 与设计拍板的关系

在另一端基于 [phase2.5_roles.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_roles\phase2.5_roles.md) 产出 [phase2.5_设计方案.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.5_设计方案.md) 后，这套角色面具的作用转为：

- 检查正式方案是否仍在已拍板边界内
- 检查复盘对象骨架、归因表达与优先级收口是否足够稳定
- 检查实现起步范围是否仍然停留在 `MVP`
- 检查验证与回写链路是否已具备进入正式实现的条件

也就是说：

> **设计拍板前，角色面具主要用于“审查设计方案是否可以正式进入实现”。**

### 9.3 不允许的误用方式

- 不允许把角色面具文档当成跳过拍板的理由
- 不允许用“面具内部已经讨论过”替代用户拍板
- 不允许在设计前冻结拍板前，以角色面具名义直接扩设计范围
- 不允许在设计拍板前，以实现便利为由反向改写主对象与主流程

## 十、另一端接手时的使用方式

另一端后续接手时，不应把这份文档理解为“必须每轮都演完整角色戏”，而应理解为：

- 在做设计时，知道当前需要从哪些视角检查方案
- 在写方案时，知道哪些判断应由哪个角色面具负责
- 在发现冲突时，知道该按什么顺序处理
- 在进入实现时，知道什么时候应减少高层角色，更多交给实现与验证视角
- 在建立 `phase2_roles/phase2.5_roles.md` 时，知道正式职责与执行面具如何映射

### 10.1 最推荐的使用方法

- 先用总协调面具确认任务边界
- 用方案设计 / 边界收口面具确认是否越界
- 用现实校验、系统复盘对象、问题归因面具搭建核心设计骨架
- 用阶段3收口面具与结构化契约面具把方案变成可回写、可消费对象
- 用验证验收面具做复查
- 条件满足后，再调用实现落地面具推进

### 10.2 最不推荐的使用方法

- 把所有角色一次性拉满并要求同时输出
- 让角色面具彼此辩论但没有明确产物
- 在角色尚未收敛前直接写实现
- 把角色面具文档误当成运行时多 Agent 系统设计稿
- 让“系统复盘对象面具”退化成“复盘报告润色器”

## 十一、配置结论

### 11.1 当前建议

- `2.5` 当前适合采用 **`8-9` 个标准角色面具 / `5-6` 个压缩角色面具** 的方案
- 当前角色面具配置应服务于**现实校验质量、复盘对象稳定性、归因可回写性与边界控制**，而不是为了展示复杂性
- 角色面具的核心价值在于让另一端接手时，能够按稳定视角做高质量方案设计与实现
- `结构化契约面具` 适合作为 `2.5` 的执行层专项支撑视角保留，以降低后续 `2.1 ~ 2.5` 消费与阶段3收口交接成本

### 11.2 一句话总结

> `Phase 2.5` 的角色面具配置，不是为了把项目包装成多 Agent 系统，而是为了让同一 Agent 在当前 `MVP` 范围内，能以更清晰的视角分工完成边界控制、现实校验、系统复盘对象设计、问题归因、阶段3收口、契约设计、验证和实现推进。
