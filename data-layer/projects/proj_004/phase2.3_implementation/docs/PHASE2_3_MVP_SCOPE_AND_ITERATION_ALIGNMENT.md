## 一、为什么要单独补这份文档

在当前围绕 `2.3` 的讨论中，我们已经补上了一层非常关键的基础：

- [PHASE2_3_FIRST_PRINCIPLES_AND_ROLE_ESSENCE.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2.3_implementation\docs\PHASE2_3_FIRST_PRINCIPLES_AND_ROLE_ESSENCE.md)

这意味着，`2.3` 已经不再只是“生成决策建议”这样一个泛化任务，而是被重新收口为：

> **把 `2.2` 的机会判断，转化为组织可分阶段承诺、可动态调整、可必要时及时止损的行动结构。**

一旦这个定义成立，接下来最重要的问题就不再是“理论上 `2.3` 还能做什么”，而是：

> **当前哪些能力必须进入 `2.3 MVP`，哪些能力虽然值得做，但不应该在第一轮就并入范围。**

如果这个边界不提前写清楚，`2.3` 特别容易出现两种偏差：

- 一种是把路径规划、预算估算、风险分析、长篇建议模板、复杂案例映射、细粒度资源模型、自动化汇报等内容一次性全塞进 MVP，最终把 `2.3` 做成一个“大而全计划书生成器”；
- 另一种是因为担心复杂度太高，把当前其实最应该进入 MVP 的“行动姿态判断、阶段承诺结构、Go/No-Go 门槛、退出条件”也一起后置，最后做出一个只有口头建议、没有行动骨架的半成品。

因此，这份文档的目的，就是先把当前阶段的判断收口：

> **`2.3` 当前最合理的推进方式，不是直接做成完整战略计划产品，而是先以“行动设计层 / 资源承诺层”的最小闭环为核心，完成输入契约、输出契约与最小行动骨架的收口，再把复杂资源模型、长文档包装、更多自动化与特化策略作为明确后置的增强项。**

## 二、当前 `2.3` 处在什么阶段

结合 [phase2.3_目标说明.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.3_目标说明.md) 与 [PHASE2_3_FIRST_PRINCIPLES_AND_ROLE_ESSENCE.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2.3_implementation\docs\PHASE2_3_FIRST_PRINCIPLES_AND_ROLE_ESSENCE.md) 的当前结论，`2.3` 可以被概括为：

> **模块本质已经基本明确，但工程范围、输入输出契约和 MVP 边界仍需要进一步拍板。**

当前已经比较明确的内容包括：

- `2.3` 的本质不是决策建议写作模块，而是**行动设计层 / 资源承诺层**；
- `2.3` 的上游主要来自 `2.2` 的结构化机会判断对象，以及 `2.4` 的行动支撑证据包；
- `2.3` 的输出应该优先是**结构化行动决策对象**，而不是只有自然语言建议书；
- `2.3` 的职责重点是：行动姿态判断、分阶段承诺设计、Go/No-Go 节点、风险约束映射、退出路径设计；
- 预算、时间线、风险章节、建议文档都重要，但必须服务于“承诺设计”，不能反过来成为模块本体。

这意味着当前不应直接把 `2.3` 看成“完整决策建议产品开发阶段”，而应看成：

- 定义已基本成立；
- 现在需要做 MVP 范围拍板；
- 接着再进入实现方案设计与验证阶段。

## 三、当前争议的核心是什么

当前讨论的重点，实际上集中在两类问题：

### 3.1 `2.3` 的核心产物到底是什么

也就是 `2.3` 当前到底应该优先做成：

- 长篇决策建议文档生成器
- 路径规划器
- 预算估算器
- 风险报告生成器
- 还是结构化行动决策对象生成层

如果这个问题不先拍板，后续设计很容易各做各的：

- 有人会优先做短中长期路径模板；
- 有人会先做资源估算表；
- 有人会先做风险矩阵章节；
- 有人会先做 Markdown 模板；
- 最后发现并没有一个稳定的“行动对象”作为中间主产物。

### 3.2 哪些能力必须在当前阶段进入 MVP

`2.3` 很容易让人自然联想到很多“看起来都很需要”的能力，例如：

- 更细的人力模型
- 更复杂的预算拆解
- 更多行业特化路径模板
- 更完整的短中长期计划书
- 更强的自然语言汇报能力
- 多 Agent 协作式路线设计

这些能力未来都可能值得做，但当前真正要先判断的是：

> **在不偏离第一性原理的前提下，`2.3` 当前最小可验证闭环到底是什么。**

只有这个问题先答清楚，后面的范围、节奏和优先级才能真正稳定。

## 四、结论先说：哪些该进入当前 MVP，哪些不该

## 4.1 应进入当前 MVP 的内容

### A. `2.3` 的输入契约最小化定义

这是当前最明确的 MVP 必做项。

原因是：

- 没有输入契约，就无法知道 `2.3` 消费的到底是 `2.2` 的什么结果；
- 没有输入契约，就无法和 `2.4` 约定“哪些内容只是参考，哪些内容会直接进入行动设计”；
- 没有输入契约，后续路径规划、预算估算、风险约束和建议生成都只能建立在模糊输入之上。

当前建议优先锁定的最小输入包括：

- 来自 `2.2` 的结构化机会判断对象
  - `opportunity_title`
  - `opportunity_thesis`
  - `supporting_evidence`
  - `counter_evidence`
  - `why_now`
  - `key_assumptions`
  - `uncertainty_map`
  - `priority_level`
  - `next_validation_questions`
- 来自 `2.4` 的行动支撑上下文 / context packet
  - 决策模型
  - 分阶段资源模板
  - 风险矩阵
  - 相似案例中的推进路径
  - 常见失败模式与缓解策略
- 来自系统 / 组织的最小约束条件
  - 资源上限假设
  - 风险偏好
  - 时间窗口要求
  - 是否允许进入验证 / 试点 / 升级阶段

### B. `2.3` 的结构化行动决策对象输出契约

这部分同样应该进入当前 MVP，而且优先级极高。

更准确地说，`2.3` 当前最需要先拍板的，不是“建议文档写几段”，而是：

> **结构化行动决策对象至少要包含哪些字段。**

当前建议作为 MVP 主骨架进入输出契约的字段包括：

- `opportunity_title`
- `decision_posture`
- `why_this_posture`
- `phased_plan`
  - `stage`
  - `objective`
  - `key_assumptions_to_test`
  - `actions`
  - `resources`
  - `milestones`
  - `go_no_go_criteria`
  - `exit_conditions`
- `top_risks`
- `resource_commitment_logic`
- `fallback_path`

自然语言建议文档可以存在，但更适合作为结构化对象的派生视图，而不是当前 MVP 的唯一主产物。

### C. 行动设计的四项核心能力

这部分是 `2.3 MVP` 最核心的能力闭环，建议直接纳入当前范围。

#### 1. 行动姿态判断

能在机会进入 `2.3` 后，形成明确的行动姿态，例如：

- `watch`
- `validate`
- `pilot`
- `escalate`
- `hold`
- `stop`

重点不是标签本身，而是它背后的资源动作和组织动作。

#### 2. 分阶段承诺结构设计

能明确给出：

- 当前阶段要验证什么
- 当前阶段允许投入什么资源
- 下一阶段在什么条件下解锁
- 哪些资源必须建立在上一阶段结果成立之后

也就是说，能形成“验证优先、承诺递增”的最小推进路径。

#### 3. Go/No-Go 节点与退出条件设计

能清楚说明：

- 什么时候继续推进
- 什么时候暂停推进
- 什么时候降级观察
- 什么时候应该止损退出

这部分是 `2.3` 区别于普通建议文本生成器的关键。

#### 4. 风险约束映射

能把风险直接写进行动结构，而不是只在后面附加一个“风险分析”章节。

也就是说，风险要真正影响：

- 路径选择
- 节奏安排
- 资源释放
- 退出门槛
- 备选路径

### D. 轻量资源承诺表达

这部分建议进入当前 MVP，但保持轻量。

更适合的方式是：

- 先表达阶段级资源承诺，而不是一次性做完整预算系统；
- 用“角色 / 带宽 / 时间 / 粗粒度预算级别”来表达；
- 先绑定关键假设验证，再考虑更细财务颗粒度。

当前阶段更应该回答的是：

- 这一阶段大概要投入几类角色；
- 这类投入是轻量验证还是中度试点；
- 为什么需要这类投入；
- 如果验证失败，哪些资源不应继续释放。

### E. 轻量案例验证

这部分也建议进入当前 MVP，但保持轻量。

更适合的方式是：

- 先选少量真实 `2.2` 机会对象；
- 验证 `2.3` 是否真的能把机会对象转成“可拍板的行动对象”；
- 重点验证输出是否便于人工理解和便于 `2.5` 复盘，而不是先追求计划书文本的华丽程度。

## 4.2 可以进入当前 MVP，但应作为“条件触发项”的内容

### F. 简版自然语言决策建议

这部分可以保留，但不建议当前阶段做得过重。

更适合的方式是：

- 生成简版说明或摘要；
- 帮助人工快速理解结构化行动对象；
- 服务于验证与协作沟通。

但不建议在当前阶段就把“符合完整 9 章节建议模板”作为 `2.3 MVP` 的主验收条件。

### G. 轻量资源估算规则

这部分可以有，但应是**轻规则、弱精算**。

更适合的方式是：

- 基于阶段目标给出粗粒度资源带宽；
- 保留少量可解释规则；
- 明确这是“承诺设计辅助”，而不是“财务预算系统”。

例如可以有：

- `validation_stage_team_size_hint`
- `pilot_stage_budget_band`
- `estimated_time_band`

但不建议一开始就做成完整的人力、成本、现金流细分模型。

### H. 简单的路径模板库

这部分也可以作为当前 MVP 的辅助项。

适合做的程度是：

- 提供少量通用推进模板；
- 覆盖验证型、试点型、升级型三类基本路径；
- 帮助 `2.3` 避免完全从零生成动作结构。

但不建议当前阶段就建立大规模、行业特化、场景细分很深的模板体系。

## 4.3 不建议作为当前 MVP 硬需求的内容

### I. 完整的大而全资源模型

当前不建议把完整预算 / 招聘 / 现金流 / 多角色配置 / 全周期时间线模型作为 `2.3 MVP` 硬需求。

原因不是这些内容不重要，而是：

- `2.3` 当前最大的缺口不是“资源模型不够细”，而是“缺少稳定的行动对象与承诺逻辑”；
- 如果在行动骨架未稳定前先引入过细资源模型，会迅速增加实现复杂度和归因难度；
- 这些更适合作为 `v1.1 / v1.2` 的增强工程，而不是当前 MVP 的成立前提。

### J. 完整长篇战略计划书 / 经营计划书模板对齐

这块不应作为当前 MVP 硬范围。

原因是：

- 当前最重要的是行动对象是否成立；
- 如果过早把精力放在章节完整度和语言包装上，很容易把 `2.3` 做成展示层，而不是行动设计层；
- 文档更适合作为结构化行动对象稳定后的派生产物。

### K. 复杂的行业特化策略库

例如：

- 投资型机会专属路径树
- 研发预研型机会专属资源模板
- 生态合作型机会专属里程碑体系
- 多业务线差异化风险偏好模型

这些能力未来都值得做，但当前不应进入第一轮 MVP 硬范围。

### L. 多 Agent 化的行动设计编排

`2.3` 当然也存在多 Agent 的合理想象空间，比如：

- 路径推进视角 Agent
- 风险约束视角 Agent
- 资源现实性视角 Agent
- 管理层拍板视角 Agent

但当前不建议把它并入 `2.3 MVP` 硬需求。

原因不是它没有价值，而是：

- `2.3` 当前最大的缺口不是“缺少多角色辩论”，而是“缺少最小行动对象与承诺结构”；
- 在对象和边界未稳定前引入多 Agent，会进一步提高编排复杂度和调试成本；
- 多 Agent 更适合在行动骨架稳定后，作为“提高方案稳健性和反方约束覆盖”的增强机制引入。

## 五、推荐的优先级排序

如果把当前阶段要做的事按优先级排序，我会建议：

### P0：先锁定 `2.3` 的输入契约和输出契约

核心目标：

- 明确 `2.3` 消费什么；
- 明确 `2.3` 产出什么；
- 为后续实现、评估和下游对接建立统一基线。

### P1：实现行动设计最小闭环

核心目标：

- 至少能完成“姿态判断 → 分阶段承诺 → Go/No-Go 节点 → 风险约束映射”这一条最短链路；
- 证明 `2.3` 不是建议改写器，而是真正的行动设计层。

### P2：做轻量案例验证

核心目标：

- 用少量真实机会对象检查输出质量；
- 重点看是否支持人工拍板与后续 `2.5` 验证，而不是先追求文本表现力。

### P3：根据验证结果决定是否引入轻量资源估算增强或简版建议视图

核心目标：

- 用问题驱动增强；
- 保持每次增加的复杂度都能解释其收益。

### P4：在行动对象稳定后，再讨论更细资源模型、模板库扩展或多 Agent 增强

核心目标：

- 把复杂度用在真正需要的地方；
- 让每一步增强都能回答“它具体解决了什么问题”。

## 六、为什么我不建议现在把复杂资源模型和长文档包装并进 `2.3 MVP`

这个判断不是因为这些东西不重要，而是因为当前阶段更重要的是三件事。

### 6.1 先证明 `2.3` 的基本行动对象是否成立

如果连结构化行动决策对象都还没稳定，就直接上细预算、长时间线、全章节报告，很容易出现：

- 输出看起来很完整；
- 文本很多；
- 预算也像模像样；
- 但最后仍然没有一个稳定的行动对象可供组织拍板和下游验证。

### 6.2 先建立可解释的迭代顺序

一个好的 `2.3` 迭代节奏应是：

- 先定对象；
- 再定行动骨架；
- 再做轻量案例验证；
- 最后再决定要不要加细预算、更多模板、更多自然语言包装或多 Agent。

这样每一步都能解释“为什么要加”和“加了后是否真的更好”。这也更符合你希望项目实践强调**真正学会 AI-native 思维与可讲述工程判断，而不是单纯让系统自动生成一切**的方向 [[memory:rebp86gg]]。

### 6.3 先避免 MVP 范围膨胀

如果现在把：

- 细粒度预算系统
- 完整经营计划书模板
- 多场景策略树
- 长中短期深度时间线
- 多 Agent 编排

全部并进 `2.3 MVP`，那么当前阶段就会迅速从“最小可验证闭环”变成“增强版产品化迭代”。

这不利于启动拍板，也不利于你后续真正讲清楚：

- `2.3` 的本体是什么；
- 哪些增强是基于问题而不是基于想象追加的；
- 哪些设计选择是因为理解了模块职责，而不是因为看起来更完整。

## 七、那复杂能力更适合放在哪里

如果从工程经历与模块职责两方面一起考虑，这些能力依然完全有机会进入项目，只是不一定要放在 `2.3 MVP`。

### 7.1 作为 `2.3` 的 P1 / P2 增强项

这是最自然的一种放法。

当 `2.3` 已经有：

- 稳定输入；
- 稳定行动对象；
- 初步案例验证；

就可以把复杂能力引入为：

- 更细的预算与资源模板；
- 更多场景化路径树；
- 更强的风险应对策略映射；
- 更完整的建议文档与汇报视图；
- 多 Agent 的行动方案互相校验。

这种做法的好处是：

- 增强收益更容易被观察；
- 不会把基础对象层搞乱；
- 更像是在真实产品里，为解决具体问题而引入复杂能力，而不是先为了显得完整而做。

### 7.2 `2.5` 更适合作为复杂协作流与现实验证能力的展示场景

如果你的目标里有一项是：

> **希望项目里不仅有单模块能力，还能体现完整工作流的整合、验证与协作价值。**

那从工程表达和项目展示角度看，`2.5` 反而是一个非常合理的复杂能力落点。

因为 `2.5` 本身就天然涉及：

- 多模块整合；
- 端到端工作流；
- 真实案例运行；
- 结果对照与复盘；
- 系统级失真点定位；
- 改进建议输出。

在这个阶段里，复杂编排、多角色校验、现实反馈对照会更容易被讲清楚，也更符合系统闭环的语义。

## 八、推荐的当前阶段工作定义

基于以上判断，我会把当前 `2.3` 阶段定义成：

> **`2.3 MVP` 的范围收口与行动骨架建立阶段。**

这个定义比“开始做决策建议功能”更准确，因为它同时包含：

- 输入输出契约明确化；
- 结构化行动决策对象定义；
- 最小行动设计闭环实现；
- 轻量案例验证；
- 复杂资源模型、长文档包装和多 Agent 能力的明确后置。

但它也明确排除了：

- 一上来就做完整资源计划系统；
- 一上来就做完整战略计划书对齐；
- 一上来就做多 Agent 行动设计编排；
- 一上来就做大规模行业特化策略库。

---

## 九、推荐的对齐说法

如果需要和另一端快速对齐，我建议直接使用下面这段表达：

> 当前 `2.3` 的第一性定位已经明确，其本质是“行动设计层 / 资源承诺层”，而不是长篇建议生成器或大而全计划书系统。下一步不建议把细粒度预算、完整战略计划模板、复杂策略树和多 Agent 编排一次性并入 MVP，而应先完成 `2.3` 的输入契约、结构化行动决策对象输出契约，以及“行动姿态判断 / 分阶段承诺结构 / Go-No-Go 节点 / 风险约束映射”四项核心能力的最小闭环，并用少量真实机会对象做轻量验证。待这个闭环稳定后，再决定哪些复杂资源模型、报告增强或协作编排值得进入 `2.3 P1 / P2`，或放在 `2.5` 的整合验证阶段更完整地呈现。

这段说法的好处是：

- 边界清晰；
- 不否定后续增强；
- 同时保留了后续工程表达空间。

## 十、当前阶段的结论

如果用一句话总结这份文档的核心结论，可以写成：

> **`2.3` 现在最合理的推进方式，不是把行动建议系统一次性做满，而是先以“结构化行动决策对象 + 最小行动设计闭环”为中心，完成可解释的 MVP 收口，再决定哪些增强值得进入下一阶段。**

也就是说：

- 输入 / 输出契约与行动闭环：**现在做**
- 轻量资源承诺、风险映射与案例验证：**现在做**
- 细预算、长文档、复杂模板、多 Agent：**后续做**

## 十一、建议的 MVP 交付最小清单

为了让这份范围文档更容易指导后续落地，我建议把 `2.3 MVP` 的最小交付物进一步收口为下面几项。

### 11.1 必须交付的结构化产物

- `action_decision_schema.json` 或等价的 Schema 定义
- `decision_posture` 枚举与说明
- `phased_plan` 的最小字段定义
- `go_no_go_criteria` 与 `exit_conditions` 的表达规范
- `resource_commitment_logic` 的最小表达规范

### 11.2 必须交付的最小能力实现

- 从 `2.2` 机会对象生成 `decision_posture`
- 生成至少 1-2 个阶段的 `phased_plan`
- 对关键假设绑定验证动作
- 产出至少一组 Go/No-Go 门槛
- 产出至少一组退出条件与 fallback 路径

### 11.3 必须交付的验证材料

- 至少 2 个真实案例的行动对象输出
- 一份轻量验证记录
- 对“是否降低行动歧义”的简要判断
- 对“是否便于 `2.5` 复盘”的简要判断
