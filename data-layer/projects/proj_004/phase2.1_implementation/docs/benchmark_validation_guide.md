## 一、Benchmark 验收关键注意事项

### 1.1 样本构建的关键点

#### 样本数量与分布

**必须严格遵守**：
- 总量：30 条（不多不少）
- 输入类型分布：
  - news（新闻）：10 条
  - report（报告）：10 条
  - announcement（公告）：10 条
- 信号类型分布：
  - technical：8 个
  - market：8 个
  - team：7 个
  - capital：7 个

**为什么这个分布？**
- 覆盖三类输入源，验证通用性
- 四类信号均衡分布，避免偏向
- 30 条是 MVP 阶段的平衡点（太少无代表性，太多拖慢启动）

#### 样本质量要求

**正例样本（约 24-26 条）**：
- ✅ 必须有明确的范式信号
- ✅ 证据充分，不模糊
- ✅ 覆盖典型场景（如"UE5 升级"、"A 轮融资"、"前暴雪制作人加入"）

**边界样本（约 2-3 条）**：
- ⚠️ 信号存在但模糊（如"市场传闻融资"）
- ⚠️ 用于测试 confidence_score 的区分度

**负例样本（约 1-2 条）**：
- ❌ 纯营销软文，无实质信号
- ❌ 用于测试是否会误报

#### 样本来源建议

**选项 A：真实样本（推荐）**
- 从游戏行业新闻网站收集
- 优点：真实性强，更有说服力
- 缺点：需要脱敏处理

**选项 B：合成样本**
- 基于典型模式人工编写
- 优点：可控性强，快速构建
- 缺点：可能不够真实

**选项 C：混合方式（最推荐）**
- 70% 真实样本 + 30% 合成样本
- 平衡真实性与可控性

## 二、标注规则的关键点

### 2.1 标注口径必须明确

**准确率定义**：
```
准确率 = 正确抽取的信号数 / 总抽取信号数

什么算"正确"？
✅ 信号类型正确（technical/market/team/capital）
✅ 信号标签合理（如"UE5 升级"而非"技术变化"）
✅ 描述准确
✅ 证据充分（evidence_text 能支撑结论）
✅ 评分合理（在评分口径范围内）
```

**召回率定义**：
```
召回率 = 正确抽取的信号数 / 应抽取的信号总数

什么算"应抽取"？
- 人工标注时，先标记原文中所有应该被抽取的信号
- 然后对比系统实际抽取了多少
```

### 2.2 评分标准的一致性

**关键问题**：不同标注者对评分的理解可能不一致

**解决方案**：
1. **使用明文评分规则表**（已在 `schemas.py` 中定义）：
   ```
   intensity_score（强度）:
     1-3: 弱信号，影响有限
     4-7: 中等信号，有一定影响
     8-10: 强信号，重大影响

   confidence_score（可信度）:
     1-3: 传闻、未证实
     4-7: 可信来源、间接证据
     8-10: 官方确认、直接证据

   timeliness_score（时效性）:
     1-3: 过时信息（>6 个月）
     4-7: 近期信息（1-6 个月）
     8-10: 最新信息（<1 个月）
   ```

2. **评分容差**：
   - 如果标注评分是 7，系统输出 6 或 8，算正确
   - 容差范围：±1 分
   - 如果差距 ≥2 分，算错误

### 2.3 标注示例

**示例 1：资本信号**
```
原文：某游戏工作室宣布完成 A 轮融资 500 万美元

人工标注：
- 应抽取信号数：1
- signal_type: capital ✓
- signal_label: "A 轮融资" 或 "A 轮融资 500 万美元" ✓
- intensity_score: 7-8 (中等偏强)
- confidence_score: 9-10 (官方宣布)
- timeliness_score: 根据发布时间判断
```

**示例 2：技术信号**
```
原文：某游戏工作室宣布项目从 Unity 迁移到 Unreal Engine 5

人工标注：
- 应抽取信号数：1
- signal_type: technical ✓
- signal_label: "引擎迁移至 UE5" 或 "Unity 迁移到 UE5" ✓
- intensity_score: 8-9 (强信号，技术栈重大变更)
- confidence_score: 9-10 (官方宣布)
- timeliness_score: 根据发布时间判断
```

**示例 3：边界样本**
```
原文：市场传闻某公司正在洽谈新一轮融资，但官方尚未确认

人工标注：
- 应抽取信号数：1
- signal_type: capital ✓
- signal_label: "融资传闻" ✓
- intensity_score: 4-5 (中等偏弱，未确认)
- confidence_score: 2-3 (传闻，未证实)
- timeliness_score: 根据发布时间判断
```

**示例 4：负例**
```
原文：游戏画面精美，玩法有趣，敬请期待后续消息

人工标注：
- 应抽取信号数：0
- 无范式信号（纯营销软文）
```

## 三、测试执行的关键点

### 3.1 测试环境

**必须记录**：
- API 模型：Claude Opus 4.6
- Prompt 版本：v1.0
- 解码器版本：v1.0
- 测试时间：2026-03-XX
- 温度设置：0.0

**为什么？**
- 后续如果调整 Prompt，需要对比不同版本的效果
- 基线测试必须可复现

### 3.2 测试流程

```python
# 伪代码
for sample in benchmark_samples:
    # 1. 运行解码器
    result = decoder.decode(sample)

    # 2. 记录原始输出
    save_raw_output(result)

    # 3. 对比人工标注
    accuracy = compare_with_annotation(result, sample.annotation)

    # 4. 记录错误案例
    if accuracy < threshold:
        save_error_case(sample, result)
```

### 3.3 错误分析

**必须记录的错误类型**：

1. **漏报（False Negative）**：
   - 原文有信号，但系统没抽取
   - 影响召回率
   - 示例：原文提到"前暴雪制作人加入"，但系统未识别

2. **误报（False Positive）**：
   - 原文无信号，但系统抽取了
   - 影响准确率
   - 示例：原文只是营销软文，但系统识别为"市场信号"

3. **分类错误**：
   - 信号存在，但类型错误
   - 示例：融资信号被识别为"market"而非"capital"

4. **评分偏差**：
   - 信号正确，但评分不合理
   - 示例：官方确认的消息，confidence_score 只给了 5

## 四、验收报告的关键点

### 4.1 必须包含的内容

**1. 基本信息**：
- 测试时间
- 样本数量
- 解码器版本
- Prompt 版本

**2. 核心指标**：
```
Schema 合法率：XX% (目标 100%)
准确率：XX% (目标 ≥80%)
召回率：XX% (目标 ≥75%)
平均处理耗时：XX ms (目标 <30s)
```

**3. 详细分析**：
- 各信号类型的准确率/召回率
- 各输入类型的表现
- 错误案例分析（至少 3-5 个典型案例）

**4. 结论与建议**：
- 是否达到验收标准？
- 如果不达标，原因是什么？
- 下一步优化方向

### 4.2 验收判断标准

**通过验收**：
- ✅ Schema 合法率 = 100%
- ✅ 准确率 ≥ 80%
- ✅ 召回率 ≥ 75%
- ✅ 平均处理耗时 < 30s

**不通过验收**：
- ❌ 任一指标未达标
- 触发后置拍板：
  - 引入规则 + LLM 混合链路
  - 调整 Prompt 策略
  - 增加 few-shot 样例

## 五、常见陷阱与注意事项

### 5.1 陷阱 1：标注不一致

**问题**：不同标注者对同一样本的标注结果不同

**解决**：
- 先标注 5 条样本
- 对比标注结果
- 统一标注口径
- 再标注剩余 25 条

### 5.2 陷阱 2：样本偏向

**问题**：30 条样本都是"强信号"，缺少边界样本和负例

**解决**：
- 必须包含 2-3 条边界样本
- 必须包含 1-2 条负例
- 测试系统的区分能力

### 5.3 陷阱 3：过度拟合 few-shot

**问题**：样本与 few-shot 样例过于相似，无法测试泛化能力

**解决**：
- 样本应覆盖不同的表述方式
- 避免与 few-shot 样例雷同

### 5.4 陷阱 4：忽略处理耗时

**问题**：只关注准确率/召回率，忽略性能

**解决**：
- 记录每条样本的处理耗时
- 计算平均值和最大值
- 如果超过 30s，需要优化

## 六、执行时间线建议

**第一步：构建标注规范**（1-2 小时）
- 基于 `schemas.py` 的评分口径
- 编写 3-5 个标注示例
- 明确"正确"的判断标准

**第二步：收集样本**（2-3 小时）
- 优先使用真实样本（游戏行业新闻）
- 补充合成样本（覆盖边界情况）
- 确保分布符合要求

**第三步：人工标注**（3-4 小时）
- 标注前 5 条，统一口径
- 标注剩余 25 条
- 记录标注时间

**第四步：运行测试**（1 小时）
- 批量运行解码器
- 记录原始输出
- 对比人工标注

**第五步：出具报告**（2-3 小时）
- 计算核心指标
- 分析错误案例
- 给出优化建议

**总耗时预估**：1-2 天
