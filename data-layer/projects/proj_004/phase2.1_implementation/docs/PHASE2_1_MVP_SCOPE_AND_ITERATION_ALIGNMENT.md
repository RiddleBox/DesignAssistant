## 一、为什么要单独补这份文档

在最近围绕 `2.1` 的讨论中，我们已经补上了几层非常重要的背景材料：

- 模块定义
- 岗位本质
- `2.4 -> 2.1` 上下游协议
- benchmark 升级方向
- Prompt 角色设定与 few-shot 升级方向

这说明 `2.1` 已经不再只是“做没做出来”的问题，而开始进入：

> **哪些应该立刻进入当前 MVP，哪些应作为后续迭代保留。**

如果这个边界不提前写清楚，很容易出现两种偏差：

- 一种是把所有未来增强都一口气塞进 MVP，导致实现速度和归因能力一起变差
- 另一种是因为担心复杂度过高，把一些当前其实应该纳入 MVP 的基础工作也后置掉

因此，这份文档的目的就是把当前阶段的判断写清楚：

> **`2.1` 现在最适合的推进方式，不是“全量升级”，而是“以 benchmark baseline 为核心、以轻量升级为优先、以后续增强为清晰后置”。**

## 三、当前争议的核心是什么

当前讨论的重点主要集中在两类升级：

这些方向都成立，但不代表它们要在同一阶段全部完成。

## 四、结论先说：哪些该进入当前 MVP，哪些不该

## 4.1 应进入当前 MVP 的内容

### A. 基础 benchmark 跑通并形成 baseline

这是当前最明确的 MVP 必做项。

原因是：

- 没有 baseline，就无法判断当前版本到底够不够用
- 没有 baseline，就无法判断后续 Prompt / few-shot 升级是否真的有收益
- 没有 baseline，后续所有迭代都会更像主观优化而不是可解释演进

### B. benchmark 的轻量升级版观察项

这部分建议进入当前 MVP，但不建议一上来就做得过重。

更适合的方式是：

- 保留现有基础指标作为正式验收项
- 在样本标注和复盘中补入轻量观察字段
- 先开始记录“高价值变化 / 背景保留 / 噪音 / 边界”相关信息

这一步进入当前 MVP 的意义在于：

- 它不会明显拖慢实现速度
- 却能显著提高首轮 benchmark 的解释力

---

## 4.2 可以进入当前 MVP，但应作为“条件触发项”的内容

### C. Prompt 角色设定的轻量升级

这部分不建议在 benchmark 前先做，但建议保留为当前 MVP 内的条件触发动作。

也就是说：

- 如果 benchmark 暴露出明显误报、边界不稳、命名漂移问题
- 那么就可以在当前阶段做一轮轻量角色设定升级

更推荐的升级幅度是：

- wording 收口
- 边界约束补强
- 明确允许“不抽”或“低置信输出”

而不是直接重写整套 Prompt 策略。

### D. 少量定向 few-shot 补强

同样建议作为条件触发项，而不是预设硬需求。

如果首轮 benchmark 暴露出清晰错误模式，则可以：

- 补 2-4 条高针对性的 few-shot
- 优先补边界样本、反例、命名标准例
- 避免在没有错误画像前盲目扩容样例库

---

## 4.3 不建议作为当前 MVP 硬需求的内容

### E. 完整系统化的 few-shot 资产体系

这类工作虽然值得做，但更像 v1.1 / v1.2 的迭代工程，而不是当前 MVP 的必要条件。

原因是它通常意味着：

- 样例类型全面重构
- 维护规则全面建立
- 错误类型映射体系正式建立
- Prompt 资产进入长期版本化管理

这些都需要建立在真实 benchmark 反馈基础上，而不是先验设计。

### F. `2.4 -> 2.1` 的动态上下文增强联调

这块更明确地不应纳入当前 MVP 硬范围。

原因是：

- [PHASE2_1_CONTEXT_PACKET_CONSUMPTION_DESIGN_DRAFT.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2.1_implementation\docs\PHASE2_1_CONTEXT_PACKET_CONSUMPTION_DESIGN_DRAFT.md) 当前仍然是设计草案
- 当前 `retrieval_context` 还属于预留接口而非正式依赖
- 如果现在强行纳入 MVP，会让边界失焦，也会让收益归因更困难

## 五、推荐的优先级排序

如果把当前阶段要做的事按优先级排序，我会建议：

### P0：先跑首轮 benchmark，建立 baseline

核心目标：

- 弄清楚当前 `2.1` 已有版本的真实水平
- 明确主要误报、漏报和边界问题在哪里

### P1：在 benchmark 中补入轻量观察项

核心目标：

- 让首轮评测不只是“字段正确率报告”
- 同时开始积累高价值变化 / 噪音 / 边界样本的判断素材

### P2：根据 benchmark 结果决定是否轻量升级 Prompt

核心目标：

- 用问题驱动 Prompt 迭代
- 保持每次修改都有明确归因

### P3：根据错误类型补少量高价值 few-shot 样例

核心目标：

- 用 few-shot 解决真实高频错误，而不是堆例子

### P4：再讨论更系统化的 few-shot 资产建设

核心目标：

- 在已有 baseline 和第一轮优化结果基础上，再决定系统化维护方式

### P5：最后再进入 `2.4` 增强联调阶段

核心目标：

- 在 `2.1` 自身 baseline 稳住后，再判断外部增强是否带来净收益

## 六、为什么我不建议现在把所有升级都并进 MVP

这个判断不是因为这些升级不重要，而是因为当前阶段的目标不同。

当前阶段更重要的是三件事：

### 6.1 先证明当前 MVP 是否成立

如果连当前版本都没有被 benchmark 验证，就很难判断后续增强到底是在“修问题”还是“堆复杂度”。

### 6.2 先建立可解释的迭代顺序

一个好的迭代节奏应该是：

- 先测
- 再改
- 再复测
- 再决定是否引入更复杂增强

这样后续每一步都更容易讲清楚“为什么要做”和“做了之后是否真的更好”。这也更符合你希望项目过程强调真正学会 AI-native 思维、形成面试可讲述能力，而不是由 AI 一把做完的方向 [[memory:rebp86gg]]。

### 6.3 先避免 MVP 范围膨胀

如果现在把：

- benchmark 升级完整版
- Prompt 角色重构完整版
- few-shot 体系化完整版
- `2.4` 增强联调

全部并进 MVP，那么当前阶段会迅速从“最小可验证闭环”变成“增强版产品化迭代”。

这不利于节奏，也不利于后续和另一端对齐边界。

## 七、推荐的当前阶段工作定义

基于以上判断，我会把当前阶段定义成：

> **`2.1 MVP` 的验证与轻量升级阶段。**

这个定义比“继续做功能”更准确，因为它同时包含：

- baseline 验证
- 评测口径补强
- Prompt / few-shot 的条件触发式轻量优化

但又明确排除了：

- 系统化重构
- 联调级深增强
- 过早复杂化

---

## 八、推荐的对齐说法

如果需要和另一端快速对齐，我建议可以直接用下面这段表达：

> 当前 `2.1` 的工程 MVP 主骨架已经成立，下一步不建议把所有未来增强一口气并入 MVP，而是先完成 benchmark baseline，并轻量补入“高价值变化 / 背景保留 / 噪音 / 边界样本”的观察口径。若 benchmark 暴露出明显误报或边界不稳，再在当前阶段内做一轮轻量 Prompt 角色设定与少量 few-shot 定向补强。完整的 few-shot 资产体系建设和 `2.4 -> 2.1` 动态增强联调，建议作为 MVP 之后的迭代项处理。

这段说法的好处是：

- 边界清晰
- 优先级清楚
- 既不否定增强方向，也不让当前范围失控

## 九、当前阶段的结论

如果用一句话总结这份文档的核心结论，可以写成：

> **`2.1` 现在最合理的推进方式，不是把所有增强项立即做满，而是以 benchmark baseline 为中心，先完成可解释的轻量升级，再决定哪些增强值得进入下一阶段。**

也就是说：

- benchmark 基线与轻量观察增强：**现在做**
- Prompt 轻调与少量 few-shot：**看 benchmark 再做**
- few-shot 体系化与 `2.4` 动态增强：**后续做**
