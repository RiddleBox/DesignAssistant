## 二、当前 Prompt 体系已经做对了什么

在讨论优化前，先明确当前版本并不是错的。

当前 Prompt 体系至少已经做对了三件事：

### 2.1 已有明确任务角色

当前系统设定已经把模型限定在：

- 游戏行业语境
- 非结构化情报输入
- 范式信号提取任务

这比泛用摘要型 Prompt 要强很多。

### 2.2 已有 few-shot 样例作为输出锚点

当前 few-shot 的意义在于：

- 告诉模型什么样的变化值得被抽成 `Signal`
- 给出字段级格式参照
- 初步帮助模型形成四类 `signal_type` 的边界感

### 2.3 已有 Schema 和后处理兜底

这意味着 Prompt 不是独自工作，而是与结构契约配合。

这非常重要，因为 `2.1` 的目标不是“写得像”，而是“输出稳定的可消费对象”。

所以这次升级应该理解为：

> **从一个工程上可运行的 Prompt-first MVP，进一步升级成一个更贴近岗位本质、可被 benchmark 指导迭代的 Prompt 资产体系。**

## 三、当前 Prompt 体系的主要不足

结合当前模块定位和未来增强方向，当前 Prompt 体系更像“通用抽取器”，还没有完全体现以下语义：

### 3.1 角色设定还不够精确

当前角色设定大意是：

> 你是一个游戏行业情报分析专家，负责从非结构化文本中提取范式信号。

这个版本作为 MVP 起点没有问题，但仍然偏：

- 分析师
- 研究员
- 抽取执行者

还不够像：

- 战略工作流的前端信号解码器
- 高价值变化的筛入 / 噪音的压制者
- 需要保持证据边界和低歧义输出的中间层

### 3.2 Few-shot 体系更像“有例子”，还不是“有结构”

当前 few-shot 的问题不一定是质量差，而是：

- 例子数量较少
- 样例类型还不够覆盖价值边界
- 负例和边界例的体系化程度不够
- 暂时还不能很好指导后续误差复盘

也就是说，它更像“让模型先有感觉”，但还不是一个可长期沉淀、可按错误类型补强的资产体系。

### 3.3 缺少对“什么不该抽”的强约束表达

未来 `2.1` 最大的产品风险之一，不是只漏一些边角料，而是：

- 把普通资讯误报成正式信号
- 把 PR 语言误报成 `market` / `technical`
- 对边界样本给出虚高 `confidence_score`

这些问题如果不在 Prompt 中显式压制，仅靠 few-shot 正例并不稳。

### 3.4 缺少“价值密度”与“背景保留”的语义

当前 Prompt 更强调“识别信号”，但未来需要更明确地区分：

- 正式信号
- 背景信息
- 行业噪音
- 弱证据边界内容

否则模型容易倾向于“看到了就抽”。

## 四、未来 Prompt 角色设定应该如何升级

## 4.1 升级方向：从“行业分析专家”升级为“战略工作流前端的范式信号解码器”

如果只用一句话描述更适合的角色设定，我会建议收口成：

> **你不是在写游戏行业综述，也不是在给出投资结论；你负责把非结构化外部情报解码成可进入战略研究、项目孵化与生态投资工作流的结构化范式信号。**

这个升级有几个作用：

- 压制泛评论、泛分析倾向
- 强化“结构化信号而不是长结论”的输出意识
- 强化“服务后续工作流”的中间层定位
- 自然引入“高价值变化 / 噪音抑制 / 证据可追溯”的边界

## 4.2 角色设定中建议显式加入的语义

未来 Prompt 角色设定建议至少明确这几层含义：

### 任务本质

- 你负责识别值得进入后续判断流程的正式信号
- 你不是做普通资讯总结
- 你不是做机会评估和投资结论

### 输入边界

- 原文是主证据
- 外部上下文仅用于帮助理解，不可替代原文事实
- 只有原文明确支持的变化才能进入 `signals`

### 输出边界

- 输出的是结构化 `Signal`
- `evidence_text` 必须尽量回到原文
- `description` 需要表达清楚“发生了什么变化”
- `confidence_score` 需要和证据强弱相匹配

### 价值与克制

- 高价值变化优先
- 普通背景信息不必强行升级成信号
- PR / 营销 / 空泛趋势表述应谨慎处理
- 模糊、传闻、弱证据样本应压低 `confidence_score`

## 五、建议的 Prompt 核心原则

### 5.1 先强调“什么是正式信号”，再强调“怎么输出字段”

很多 Prompt 容易先讲格式、再讲任务，但对 `2.1` 来说，更重要的是先统一“什么值得被抽出来”。

否则模型容易变成：

- 很会输出 JSON
- 但抽出的内容战略价值不高

所以 Prompt 未来应先讲：

- 什么是正式信号
- 什么只是背景
- 什么属于噪音 / PR / 边界表达

再讲字段格式。

### 5.2 先强调“证据和边界”，再强调“生成像人话”

`2.1` 不是内容写作模块，因此 Prompt 不应鼓励模型：

- 美化表达
- 自行补全因果链
- 用行业常识替代原文证据

未来更合适的导向是：

- 证据优先
- 边界清楚
- 变化事实优先于总结性评论

### 5.3 明确允许“不抽”

这是一个非常关键但经常被忽略的 Prompt 原则。

如果 Prompt 没有清楚告诉模型：

- 有些文本可以不抽
- 有些内容只适合保留为背景
- 不确定时应降低 `confidence_score` 或保持克制

模型就很容易出现“必须产出”的倾向。

对 `2.1` 来说，这种倾向非常危险。

## 六、Few-shot 体系应该如何升级

## 6.1 从“覆盖信号类型”升级为“覆盖决策边界”

当前 few-shot 主要帮助模型理解四类信号和输出格式，这是必要的。

但下一步 few-shot 更应该覆盖的是：

- 什么该抽
- 什么不该抽
- 什么应低置信输出
- 什么应保留为背景

也就是说，few-shot 的重点应从：

> **教模型分类**

逐步升级为：

> **教模型做信号级准入判断。**

## 6.2 推荐的 few-shot 样例类型结构

未来更完整的 few-shot 体系，建议至少包含以下类型：

### A. 高质量正例

作用：

- 锚定什么是典型高价值变化
- 帮助模型稳定 `signal_type`、`label`、`description`

推荐覆盖：

- `technical`
- `market`
- `team`
- `capital`

### B. 边界正例

作用：

- 教模型在证据一般但仍值得保留时，如何输出低 / 中 `confidence_score`
- 教模型避免把模糊变化写成绝对结论

### C. 反例 / 负例

作用：

- 明确哪些文本不应产出正式信号
- 压制“看到变化就输出”的冲动

推荐覆盖：

- 纯宣传文案
- 泛行业空话
- 普通版本更新但不构成范式变化

### D. 背景保留例

作用：

- 教模型识别“有信息量，但不应升级为正式信号”的内容
- 为未来 `background only` 语义打基础

### E. 命名标准例

作用：

- 让 `signal_label` 更可比较
- 减少命名漂移
- 避免过长、过评论化的标签

---

## 6.3 Few-shot 不应只是“凑数量”，而应与 benchmark 错误类型对应

未来 few-shot 最理想的演化方式，不是主观往里加例子，而是按错误模式补：

- 如果 `PR 误报` 多，就补反例
- 如果 `signal_label` 漂移，就补命名标准例
- 如果边界样本 `confidence_score` 偏高，就补边界正例
- 如果某类 `signal_type` 混淆，就补消歧样例

也就是说：

> **few-shot 应该是 benchmark 驱动的资产，而不是拍脑袋堆料。**

## 七、推荐的 Prompt 结构升级方向

当前 `2.1` 的 Prompt 结构已经具备 MVP 可用性，但后续更推荐升级为下面的逻辑顺序：

```text
[1] 角色设定与任务本质
[2] 什么属于正式信号 / 什么不属于正式信号
[3] 输出 Schema 与字段要求
[4] 评分口径与边界约束
[5] 静态 few-shot 样例
[6] 可选动态 ContextPacket（constraint / glossary / few_shot / boundary_case）
[7] 原始输入文本
[8] 输出要求
```

### 为什么建议这样排

- 先让模型理解自己是谁
- 再让模型理解“什么该抽、什么不该抽”
- 再统一结构与评分
- 最后才用样例和外部上下文辅助执行

这样可以减少“模型只学格式、不学边界”的问题。

## 八、与 `2.4 ContextPacket` 的关系

这份文档不只面向当前静态 Prompt，也面向未来增强模式。

结合 [PHASE2_1_CONTEXT_PACKET_CONSUMPTION_DESIGN_DRAFT.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2.1_implementation\docs\PHASE2_1_CONTEXT_PACKET_CONSUMPTION_DESIGN_DRAFT.md)，未来 Prompt 升级应坚持一个原则：

> **`2.4` 是证据增强层，不是结论替代层。**

因此 Prompt 中动态增强的正确方式应该是：

- `constraint`：补分类和评分边界
- `glossary`：补术语消歧
- `few_shot`：补相似历史样例
- `boundary_case`：补误判防护

而不是：

- 让 `2.4` 直接给出结论
- 让外部上下文替代原文证据
- 让增强内容压过原始输入文本

## 九、推荐的 MVP 升级方式

当前阶段最适合的，不是一次性大改 Prompt 体系，而是采用“轻量升级 + benchmark 驱动”的方式。

### 9.1 MVP 阶段建议做的轻量升级

如果 benchmark 暴露出明显问题，建议优先做：

- 角色设定 wording 收口
- 增加 2-4 个高针对性的边界 / 反例 / 命名样例
- 显式补入“可以不抽”“可以低置信输出”的边界约束

### 9.2 MVP 阶段不建议强行做满的部分

当前阶段不建议立即做成完整系统化版本的内容包括：

- 大规模动态 few-shot 调度
- 复杂 Prompt 路由策略
- 多版本角色体系并行
- 过早绑定 `2.4` 作为硬依赖

原因是：

- 当前最需要的是 benchmark baseline
- Prompt 资产体系应由错误模式驱动演化
- 过早复杂化会让收益归因变困难

## 十、一个更接近未来方向的角色设定示意

下面不是要求立刻替换当前代码里的 Prompt，而是一版更靠近未来方向的角色设定示意：

> 你是一个服务于战略研究、项目孵化与生态投资工作流的游戏行业范式信号解码器。你的职责不是写行业综述，也不是给出投资结论，而是把新闻、报告、公告等非结构化外部情报，解码为可追溯、可评分、可被后续工作流稳定消费的结构化信号。你应优先识别高价值变化，谨慎对待 PR 话术、空泛趋势表述和弱证据内容。只有原文明确支持的变化才能进入正式信号；不确定时应保持克制，并通过较低的 `confidence_score` 反映证据强度。

这段示意的重点不在措辞本身，而在它强调了四件事：

- 工作流服务对象
- 模块边界
- 信号准入意识
- 证据优先和克制原则

## 十一、推荐的迭代节奏

为了让 Prompt 升级真正可讲、可复盘、可解释，最推荐的节奏是：

### 第一阶段：保留当前 Prompt 体系，先跑 benchmark baseline

目标：

- 明确当前误报、漏报、边界不稳的主要问题在哪里

### 第二阶段：做一次轻量 Prompt / few-shot 定向升级

目标：

- 针对首轮 benchmark 暴露的问题修正
- 不追求大而全，只追求问题对问题

### 第三阶段：形成 Prompt 资产化维护方式

目标：

- 建立“错误类型 → few-shot 补强策略”映射
- 为 `2.4` 的动态增强接入留出清晰接口

### 第四阶段：再考虑更复杂的增强策略

目标：

- 在已有 benchmark 和联调收益基础上，再决定是否值得做更复杂的上下文注入、Prompt 路由或动态 few-shot

## 十二、当前阶段的结论

如果用一句话总结这份文档的核心判断，可以写成：

> **`2.1` 的 Prompt 升级方向，不是让模型“更会说”，而是让它更像一个面向战略工作流的信号解码器：更懂价值边界、更克制噪音、更稳定输出可追溯信号。**

因此，下一步最值得做的并不是立刻重写整套 Prompt，而是：

> **先用升级版 benchmark 看清错误画像，再用轻量、可归因的方式升级角色设定与 few-shot 样例。**

---

## 十三、下一步建议

基于当前阶段，最推荐的动作是：

1. **先跑首轮 benchmark，确认主要误差类型分布**
2. **若误报或边界问题明显，再升级角色设定 wording**
3. **按错误类型补充 2-4 个高价值 few-shot 样例**
4. **将 few-shot 维护方式与 benchmark 错误分类建立映射关系**
5. **待 `2.4` 协议更清晰后，再收口增强模式 Prompt 装配方式**
