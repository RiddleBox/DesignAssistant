## 二、执行优先级拍板

### 2.1 总体结论

**当前阶段采用“双线并行，但主次明确”的执行策略。**

更具体地说：

- **主线任务**: 优先修补 `2.4` 的关键阻塞项
- **副线任务**: 并行启动 `2.1` 中不深度依赖 `2.4` 真实能力的骨架工作
- **核心原则**: 先补“可被下游真实依赖”的最小能力，再扩大上层模块开发范围

### 2.2 正式判定语句

> 当前最合理的推进方式，不是完全停下 `2.1` 等待 `2.4`，也不是在 `2.4` 真实能力未稳定时全面转向 `2.1`；而是以 `2.4` 真能力补齐为主线，以 `2.1` 低耦合骨架建设为副线并行推进。

---

## 三、为什么当前应以 `2.4` 为主线

当前 `2.4` 已完成的部分主要是：

- 接口骨架
- 最小数据模型
- Mock 检索与 Mock 生成闭环
- 基础验证和 benchmark 框架

但以下关键能力仍未完成真实验收：

- 真实 embedding 检索路径
- 真实 LLM 生成路径
- 知识库数量达标
- 对下游可稳定依赖的输出协议

因此，`2.4` 当前更像“接口和链路已打通的基础版”，而不是“已可作为上层模块长期依赖的稳定底座”。如果在这一状态下把主精力直接切到 `2.1`，后续大概率会出现以下返工风险：

- 上游输出字段变化导致 `2.1` 适配返工
- Mock 质量与真实质量差异导致 prompt 或 schema 误判
- 检索结果结构变化导致评测标准失效

## 四、A/B 双线并行执行方案

## 4.1 A 组：`2.4` 修补主线

### 本组目标

**把 `2.4` 从 Mock 可联调版，补到下游可以真实接入的最小可用版。**

### 本周任务

1. **打通真实检索链路**
   - 接入真实 embedding 服务
   - 关闭或绕过关键路径中的 Mock 检索
   - 重建真实索引
   - 运行最小 smoke test，确认查询结果来自真实向量检索

2. **打通真实生成链路**
   - 接入真实 LLM 服务
   - 跑通最小闭环：输入问题 → 检索上下文 → 生成回答
   - 留存 `3-5` 组真实样例，证明输出不是占位结果

3. **处理知识库规模问题**
   - 优先补齐到 `100` 条知识文档
   - 如短期不能补齐，则必须正式修订目标并说明下调原因

4. **拍板对下游输出协议**
   - 明确 `2.4 -> 2.1` 的固定输出字段
   - 建议采用“结构化 JSON + 可读摘要”双输出形态
   - 至少包含：`matched_terms`、`evidence_chunks`、`confidence`、`summary`、`source_ids`

5. **补齐最小验收证据**
   - 保留真实调用日志或截图
   - 保留测试样例与结果
   - 更新文档，区分已完成项、未完成项、延后项

### 本组交付物

- 真实检索可运行
- 真实生成可运行
- 固定输出协议 `v1`
- 验收样例集
- 更新后的实施/验收文档

## 七、问答归档：关于真实 embedding

### Q1: “接入真实 embedding”具体指什么？

**A**: 指的是**让知识文档与查询文本都通过真实的 embedding 模型生成向量，并基于这些真实向量建立和查询索引**，而不是继续使用模拟向量、伪造向量或仅用于流程演示的 Mock 向量。

它通常至少包含以下动作：

1. 选定实际使用的 embedding 模型或服务
2. 用真实模型把知识库文档转成向量
3. 重新构建向量索引
4. 在检索时用同一个真实模型把用户查询转成向量
5. 用真实查询向量去索引中检索结果
6. 用真实结果重新验证召回质量、稳定性和耗时

### Q2: 之前出现 API 400 报错，说像是 memory 到上限，这和“真实 embedding 接入是否合格”有关吗？

**A**: **大概率无直接关系。**

如果报错发生在对话框交互阶段，表现为“当前会话后续所有输入都持续报错，只有新开对话才能恢复”，那它更像是：

- 对话系统上下文过长
- 请求体超过平台限制
- 服务端会话状态异常
- 某个请求参数在当前会话中持续污染

这类问题通常属于：

- 会话层问题
- 平台侧请求限制问题
- API 请求打包方式问题

而不是“embedding 业务语义是否正确”的验收问题。

### Q3: 那这类 400 问题和真实 embedding 完全没关系吗？

**A**: 不能说完全没关系，但**不能直接据此判断 embedding 业务不合格**。

更准确地说：

- **如果 400 发生在聊天平台对话上下文层**，通常和 embedding 质量无关
- **如果 400 发生在你调用 embedding 服务接口时**，则可能与 embedding 接入实现有关，例如：
  - 请求字段不符合接口要求
  - 输入文本过长
  - 批量大小超过限制
  - 模型名称错误
  - 鉴权或配额异常

因此，判断标准不是“有没有 400”，而是要看：

- 报错发生在哪一层
- 是聊天会话报错，还是 embedding 服务本身返回报错
- 报错响应体里具体写了什么

### Q4: 当前项目里对“真实 embedding 已接入”的验收标准应该是什么？

建议至少满足以下四条：

1. **非 Mock 向量来源**：索引中的向量来自真实 embedding 模型
2. **非 Mock 查询向量**：检索时查询向量来自同一真实模型
3. **最小召回验证通过**：至少用若干固定样例证明召回结果基本符合预期
4. **链路证据留档**：保留配置、日志、测试结果，能证明当前跑的是 Real Embedding Mode

## 八、问答归档：关于真实 LLM

### Q1: “接入真实 LLM”是指什么？

**A**: 指的是**生成模块在实际运行时，调用真实的大语言模型服务来完成回答生成、摘要生成、信息组织或结构化输出**，而不是使用：

- Mock 返回值
- 占位字符串
- 规则拼接模板
- 仅做流程验证的伪响应

### Q2: “现在应该使用的 API 接口吧，这不行吗？”

**A**: **要区分“有 API 接口”与“真实 LLM 已接入并完成最小验收”这两件事。**

如果当前只是：

- 代码里预留了调用接口
- 有生成模块的函数入口
- 可以从接口层返回一个演示答案
- benchmark 或测试里仍然走 Mock

那么这只能说明：

- **生成链路骨架存在**
- **接口形式存在**
- **流程上预留了真实接入点**

但**还不能说明真实 LLM 已经作为正式能力接入成功**。

### Q3: 那什么才算“真实 LLM 已接入”？

建议至少满足以下条件：

1. **实际调用了真实模型服务**
2. **模型返回内容确实来自真实推理，而不是 Mock 数据**
3. **最小闭环可运行**：问题输入 → 检索结果 → 组装上下文 → LLM 输出
4. **留存样例**：至少保留 `3-5` 个真实问答样例
5. **记录失败模式**：如超时、空回答、幻觉、格式不稳定等

### Q4: 也就是说，API 接口“可以调用”还不够？

**A**: 对。**“接口可调用”只证明通道存在；“真实 LLM 已接入”要求证明这条通道实际连到了真实模型，并且最小质量已被验证。**

换句话说：

- 有 API 入口 = 有门
- 真正调到模型并返回可用结果 = 门后面真的接上了系统
- 有样例、有日志、有验证 = 可以证明这扇门不是摆设

## 九、对当前推进的直接指导

基于以上判断，当前最实际的执行顺序应为：

1. 先确认 `2.4` 的真实 embedding 接入点、报错层级和索引重建路径
2. 再确认 `2.4` 的真实 LLM 调用点、输出样例和最小质量验证方式
3. 同步冻结给 `2.1` 的输出协议 `v1`
4. 在此基础上扩大 `2.1` 的正式开发范围

---

## 十、归档结语

本次拍板的关键不是“否定已有 MVP 成果”，而是明确：

**当前最值得优先补的，不是再堆更多上层功能，而是把 `2.4` 从“能演示”补到“能被真实依赖”。**

同时，`2.1` 不需要完全停摆，可以继续推进低耦合骨架，但不应在 `2.4` 真实能力未冻结前做深度绑定式开发。

因此，当前阶段的统一执行口径应为：

> **主线修补 `2.4` 真实能力，副线启动 `2.1` 骨架建设；先补底座，再扩上层。**
