## 一、为什么要补这份文档

当前 [phase2.2_目标说明.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.2_目标说明.md) 已经给出了比较明确的实现任务分解，包括：

而如果这一层不先收口，`2.2` 很容易变成一个：

- 形式上很完整
- 输出上很热闹

- 且容易和 `2.1 / 2.4 / 2.3` 互相越界

这份文档的目的，就是把 `2.2` 从“机会评估功能集合”重新收口为一个更贴近真实岗位本质的系统组件。

## 二、先回到岗位的第一性原理

如果把整个岗位从更上层压缩，不管它表面上看起来是在做：

- 行业研究
- 趋势跟踪
- 机会分析
- 投资扫描
- 新方向预判

它更本质地都在服务一件事：

> **帮助组织更早、更稳、更可解释地识别哪些变化值得被升级为下一步行动。**

这件事里真正困难的，不是“看到了什么信息”，而是：

- 哪些变化只是背景噪音
- 哪些变化构成了值得持续跟进的线索
- 哪些线索已经足以被组织视作“机会候选”
- 哪些机会候选值得进入更正式的研究、验证或资源配置流程

如果用更系统的话来表达，可以写成：

> **这个岗位的本质，是为战略研究、项目孵化与生态投资，建立一套从“外部变化”到“机会判断”再到“行动升级”的工作流。**

在这条工作流里：

- `2.1` 负责把高熵外部情报压缩成低歧义信号；
- `2.4` 负责给判断过程补充证据级上下文；
- `2.2` 负责把信号与证据升级成可被组织进一步处理的“机会假设对象”；
- `2.3` 再把这些对象推进到更明确的行动建议与决策路径。

也就是说，`2.2` 不属于“信息读取层”，也不属于“最终决策层”，而是恰好位于最关键的中间环节：

> **把变化信号升级为机会判断。**

## 三、从岗位本质反推 `2.2` 的本质

沿着上面的逻辑继续往回推，`2.2` 就不应该只被理解成：

- 评估报告生成器
- 打分器
- 多 Agent 辩论系统
- 行业分析写手

这些都可能是实现方式，但都不是 `2.2` 的第一性原理本质。

更准确地说，`2.2` 在整个系统里承担的是：

> **把 `2.1` 输出的结构化信号，与 `2.4` 输出的证据级上下文，组织成可解释、可比较、可质疑、可升级的机会假设对象。**

这里有几个关键词非常关键。

### 3.1 不是“再解释一遍信息”，而是“把信号升级成机会”

`2.1` 已经回答了一个问题：

- 这段外部信息里，发生了哪些值得进入下游流程的高价值变化

`2.2` 不应该回去重复做这件事。

`2.2` 真正要回答的是：

- 这组变化为什么重要
- 这些变化是否构成值得跟进的机会候选
- 这类机会对我们意味着什么
- 当前判断最关键的不确定性在哪里
- 它是否值得进入下一层决策流程

所以 `2.2` 的目标不是“把信号写成一份更长的说明”，而是“把信号组织成一个可判断、可比较、可升级的机会对象”。

### 3.2 不是“给出漂亮结论”，而是“给出可被质疑的判断对象”

真正有用的机会判断，不应该只是：

- 这个方向很好
- 这个机会值得关注
- 这个团队潜力很大

这些都太像结论语气，而不够像工作流中的中间层对象。

`2.2` 更理想的产出应该能够清楚表达：

- 机会论点是什么
- 支撑它的证据是什么
- 反对它的证据是什么
- 它依赖哪些关键假设
- 哪些部分仍然不确定
- 为什么它应该被排在更前面或更后面

所以 `2.2` 不只是“判断器”，还是“可被复核的判断组织器”。

### 3.3 不是“黑箱评估”，而是“判断升级层”

从整个业务链路看，`2.2` 的价值不在于“替组织一锤定音”，而在于：

> **帮助组织对机会候选做出“继续观察 / 深挖验证 / 升级关注 / 暂不推进”的判断分流。**

这意味着 `2.2` 真正服务的，不是报告模板，而是：

- 战略研究是否升级
- 预研 / 孵化是否进入进一步验证
- 生态 / 投资线索是否提升优先级

因此它更接近一个“机会升级层”，而不是一个终局裁决器。

## 四、如果从第一性原理重新定义 `2.2`

如果让我现在给 `2.2` 写一句更接近岗位本质的产品定义，我会写成：

> **`Phase 2.2` 不是评估报告生成模块，而是面向战略研究、项目孵化与生态投资的“机会判断层”。**

如果再具体一点：

> **它的职责，是将 `2.1` 输出的结构化信号与 `2.4` 输出的证据级上下文整合为可解释、可比较、可升级的机会假设对象，为后续资源配置与决策建议提供判断基础。**

这个定义比“机会评估模块”更严格，因为它同时明确了三件事：

- 输入是什么
- 输出是什么
- 边界在哪里

而一旦这层定义立住，很多后续设计选择就会自然变清晰：

- 为什么它不应该重做 `2.1`
- 为什么它不能把判断直接外包给 `2.4`
- 为什么它不该越级替代 `2.3`
- 为什么多 Agent 只能是手段，不是模块本体
- 为什么输出最好是结构化机会对象，而不只是自然语言报告

## 五、`2.2` 真正服务的对象是谁

从工程接口上看，`2.2` 的输入来自：

- `Phase 2.1` 的结构化信号
- `Phase 2.4` 的证据级上下文

而它的下游通常会是：

- `Phase 2.3` 决策建议模块
- 阶段性人工复核或深度研究
- 项目孵化 / 投资 / 战略跟进动作

但如果从岗位本质看，`2.2` 真正服务的不是某个模板文件，而是三类“升级判断活动”。

### 5.1 服务于战略研究升级

战略研究不会对每条信号都投入同等资源。

它真正需要判断的是：

- 哪些变化已经从“值得知道”升级为“值得系统研究”
- 哪些变化只是背景趋势，还不应占用太多精力
- 哪些机会候选值得形成正式专题

这要求 `2.2` 输出的是一类**可被排序和升级的判断对象**，而不是信息堆叠。

### 5.2 服务于预研 / 孵化升级

预研和孵化更关心：

- 这是不是一个值得验证的方向
- 当前机会成立依赖哪些关键假设
- 哪些不确定性还需要进一步收集信息
- 是否值得进入概念验证、接触验证或专题拆解

这要求 `2.2` 的输出不仅要有正向论点，还要有反向证据和不确定性地图。

### 5.3 服务于生态 / 投资升级

生态狩猎和投资扫描关心的是：

- 哪些团队 / 项目 / 路线值得优先跟进
- 为什么这个机会比别的更值得先看
- 当前证据是否足够支持资源投入升级
- 哪些信息仍需被验证，才能进入更高优先级

这要求 `2.2` 不只是“给分”，而是提供可解释的排序依据。

## 六、`2.2` 应该做什么，不应该做什么

如果认同上面的定位，那么 `2.2` 的职责边界就必须明确收口。

### 6.1 `2.2` 应该做的事情

#### 6.1.1 信号聚合

把分散的 `signal` 组织成同一个机会主题下的判断对象。

例如：

- 技术信号 + 团队信号 + 市场信号
- 多篇来源不同但指向同一变化方向的材料
- 单一高强度信号与若干辅助信号的组合

`2.2` 需要把这些线索聚起来，而不是逐条孤立处理。

#### 6.1.2 机会假设构建

`2.2` 需要把“发生了什么变化”升级为“这构成了什么机会”。

这一步的重点不是堆砌描述，而是明确形成一个可检验的机会论点。

#### 6.1.3 正反证据组织

一个成熟的机会判断必须同时说明：

- 支持它成立的证据有哪些
- 反对它成立的证据有哪些
- 哪些只是背景噪音，哪些是真正的反例

这可以显著减少“只会包装机会、不会质疑机会”的偏差。

#### 6.1.4 关键假设与不确定性识别

真正重要的问题不是“这是不是好机会”，而是：

- 这个机会成立需要哪些前提为真
- 当前证据最弱的环节在哪里
- 哪些问题不验证清楚，就不应该升级

因此，`2.2` 的输出里应该天然包含：

- `key_assumptions`
- `uncertainty_map`
- `next_validation_questions`

#### 6.1.5 优先级与升级判断

`2.2` 最终应该帮助组织做出的是：

- 继续观察
- 深挖验证
- 升级关注
- 暂不推进

这不是最终决策，但它是进入最终决策前最关键的分流层。

### 6.2 `2.2` 不应该做的事情

#### 6.2.1 不重做 `2.1`

`2.2` 不应该回头从原始文章重新做情报抽取，也不应该把自己变成第二个“信号识别器”。

它消费的是 `2.1` 的结果，而不是替代 `2.1`。

#### 6.2.2 不把判断外包给 `2.4`

`2.4` 的本质是证据级上下文供应层，它可以提供：

- 规则
- 术语
- 案例
- 背景知识
- 边界条件

但 `2.4` 不能替 `2.2` 做机会判断。

否则 `2.2` 的职责会变虚，整个系统也会失去可诊断性。

#### 6.2.3 不越级替代 `2.3`

`2.2` 不应该直接给出最终投资结论、最终立项结论或完整行动方案。

它输出的是**机会判断基础对象**，而不是最终行动包。

#### 6.2.4 不做泛行业分析写手

`2.2` 的任务不是把材料写成一份“看起来很像咨询报告”的长文，也不是泛泛评论行业趋势。

它的工作必须始终围绕“是否构成值得升级的机会候选”这个问题展开。

## 七、这会如何改变我们对 `2.2` 的判断标准

如果认同上面的定位，那么 `2.2` 的好坏就不能只看：

- 是否生成了完整报告
- 是否打出了几个维度分数
- 是否跑通了多 Agent 流程

这些都重要，但都不是最底层的标准。

更关键的是看下面几件事。

### 7.1 它是否真的把“信号”升级成了“机会判断”

也就是说，接入 `2.2` 后，下游是否更容易判断：

- 这是不是值得继续研究的机会
- 哪些候选更该被优先处理
- 哪些候选还停留在证据不足阶段

如果 `2.2` 只是把 `2.1` 的信号改写成更长的描述，那它并没有真正完成模块职责。

### 7.2 它是否保留了反方与不确定性

`2.2` 最常见的产品偏差，是过快进入“包装机会”模式。

真正稳健的 `2.2`，必须能够系统性保留：

- 反对证据
- 关键假设
- 风险点
- 当前无法下结论的部分

否则输出虽然很像评估，但实际上对组织帮助有限。

### 7.3 它是否支持可解释排序

一个真正有业务价值的机会判断层，不只是说“这个机会不错”，而是要说明：

- 为什么它比另一个机会更值得先看
- 为什么当前优先级应该是 `watch` 而不是 `escalate`
- 分数、结论和证据之间能否互相对齐

### 7.4 它是否真的帮助后续行动分流

`2.2` 的最终价值，不在于写出一份好看的文本，而在于帮助后续动作更清晰：

- 是继续观察
- 还是进入专题研究
- 还是做更深验证
- 还是交给下游模块生成行动建议

也就是说，`2.2` 是判断流中的枢纽层，而不只是展示层。

## 八、这会如何反推 `2.2` 的输入设计

如果沿着岗位本质往回推，`2.2` 的主输入不应该是“原始文章全文”，而应该是更接近工作流中间层的对象。

### 8.1 来自 `2.1` 的结构化信号

`2.2` 最核心的上游输入应该包括：

- `signal_type`
- `signal_label`
- `description`
- `evidence_text`
- `entities`
- `intensity_score`
- `confidence_score`
- `timeliness_score`

这些字段的意义在于：

- `2.1` 已经对“变化事实”做过第一层解码
- `2.2` 可以基于这些字段做信号聚合与机会论点构建
- `2.2` 不必每次都回到原文重新做理解

### 8.2 来自 `2.4` 的证据级上下文

`2.4` 更适合给 `2.2` 提供：

- 历史案例
- 相似失败 / 成功路径
- 方法论框架
- 术语边界
- 行业结构背景
- 用于质疑当前机会判断的反例或限制条件

从第一性原理看，`2.2` 更适合消费的是**证据包 / context packet**，而不是 `2.4` 直接替它生成的“现成答案”。

因为如果 `2.2` 直接吃答案，它的判断职责会变得不清晰。

### 8.3 来自内部的评估约束

除了外部信号与外部证据，`2.2` 还应该逐渐消费组织内部的判断框架，例如：

- 关注方向
- 机会分级规则
- 跟进优先级标准
- 升级到 `2.3` 的门槛条件

否则它只能做“泛机会判断”，而很难做“对我们有意义的机会判断”。

## 九、这会如何反推 `2.2` 的输出设计

如果 `2.2` 的本质是机会判断层，那么它最合理的输出，不应只是自然语言评估报告，而应优先是一种**结构化机会评估对象**。

例如，它至少应该能表达下面这些层次：

```json
{
  "opportunity_title": "...",
  "opportunity_thesis": "...",
  "related_signals": ["..."],
  "supporting_evidence": ["..."],
  "counter_evidence": ["..."],
  "why_now": "...",
  "key_assumptions": ["..."],
  "uncertainty_map": ["..."],
  "evaluation": {
    "change_significance": 0,
    "capture_feasibility": 0,
    "timing_window": 0,
    "evidence_strength": 0
  },
  "priority_level": "watch / research / deep_dive / escalate",
  "next_validation_questions": ["..."]
}
```

这里最关键的不是字段名本身，而是它表达的含义：

- 不是“我觉得这个机会很好”
- 而是“这是一个什么机会、为什么成立、哪里还不确定、下一步该怎么验证”

自然语言报告当然可以继续保留，但更适合被视为：

- 结构化对象的可读呈现层
- 或面向人类阅读的派生视图

而不是模块唯一的主产物。

## 十、这会如何反推 `2.2` 的评分逻辑

当前 [phase2.2_目标说明.md](f:\AIProjects\DesignAssistant\data-layer\projects\proj_004\phase2_plan\phase2.2_目标说明.md) 中使用的是四维视角：

- 市场
- 技术
- 团队
- 时间窗口

这个方向本身没有问题，但如果从第一性原理继续往下压，`2.2` 需要的不只是几个打分维度，而是一套更底层的判断框架。

我会建议至少回到下面五个问题：

1. **发生的变化是否足够重要**
   - 不是普通噪音，而是真变化
2. **这个变化是否可能演化为机会**
   - 是否形成新的价值网络、能力杠杆或结构性窗口
3. **这个机会是否是我们能抓住的**
   - 与组织能力、资源、位置是否匹配
4. **为什么是现在**
   - 时间窗口是否成立
5. **当前判断到底有多稳**
   - 证据是否足够强，不确定性是否过高

这意味着：

- 现有四维评分可以作为实现层保留
- 但上层必须补一个“判断框架层”
- 否则 `2.2` 很容易退化成“打分器”，而不是“判断器”

## 十一、这会如何反推 Multi-Agent 的定位

如果沿着这套逻辑继续推，**多 Agent 辩论不是 `2.2` 的本质，只是一个可选实现手段。**

也就是说：

- `2.2` 的本质不是“乐观派 vs 悲观派”
- `2.2` 的本质是“让机会判断更完整、更抗偏差、更可解释”

因此，多 Agent 只有在满足这些条件时才真正有价值：

- 能显著提升反方证据覆盖
- 能减少单一路径偏见
- 能让假设与反假设更清楚
- 能产出对下游更有用的结构化结果

如果只是为了“看起来更智能”，那它就会偏离第一性原理。

换句话说：

> **多 Agent 在 `2.2` 中应该是“提升判断质量的可选机制”，而不是定义 `2.2` 身份的核心标签。**

## 十二、`2.2` 的非目标应尽早写清楚

`2.2` 是整个链路里最容易“听起来很聪明、实际上最容易越界”的模块，所以非目标必须尽早明确。

当前阶段建议明确排除以下定位：

- 不做新闻摘要器
- 不做泛行业分析写手
- 不做直接投资结论黑箱
- 不做为了辩论而辩论的多 Agent showcase
- 不做脱离证据的机会包装器
- 不做直接替代 `2.3` 的行动决策模块

这些非目标越早写清楚，后续实现越不容易走偏。

## 十三、如果再压缩成一句最重要的判断

如果问：`2.2` 从第一性原理看最重要的到底是什么？

我的答案会是：

> **不是“把信号写成报告”，而是“把信号升级为可被组织采取下一步动作的机会判断”。**

这句话很重要，因为它会直接影响后续所有取舍：

- 为什么 `2.2` 不该只是评估报告生成器
- 为什么它必须保留支持与反对证据
- 为什么它不能越级给出最终决策
- 为什么它需要消费 `2.4` 的证据，而不是让 `2.4` 替它判断
- 为什么它的输出应该优先是结构化机会对象

## 十五、`2.2 MVP` 最合理的样子是什么

如果按 MVP 来看，我会建议 `2.2` 先只做到下面四件事。

### 15.1 机会组装

能把 `2.1` 的多个信号组织成一个机会候选，而不是只逐条点评信号。

### 15.2 机会判断

能明确给出：

- 机会论点
- 支持证据
- 反对证据
- 关键假设
- 不确定性

### 15.3 可解释排序

能说明为什么这个机会比另一个更值得优先看，而不是只给单个静态评分。

### 15.4 升级建议

能给出类似下面的判断分流：

- `watch`
- `research`
- `deep_dive`
- `escalate`

做到这四件事，`2.2` 就已经是一个真正有业务意义的中间层。

而以下这些内容，更适合作为 `P1 / P2`：

- 完整多 Agent 辩论流程
- 更复杂的评分体系
- 更花的自然语言报告模板
- 更强的角色扮演式交互设计
