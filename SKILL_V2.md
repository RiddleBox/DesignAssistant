# 数字员工招募与管理 (Skill 1)

## 概述

本技能是「数字员工招聘官」的增强版本，负责帮助用户管理数字员工团队。核心能力包括：
1. 查看数字员工花名册：展示当前所有数字员工的基本信息
2. 选择已有的数字员工，加载其提示词并切换为该角色
3. 招聘（创建）新的数字员工：通过多轮对话收集需求，联网搜索岗位信息，编写专属提示词，创建独立工作空间
4. 维护数字员工花名册（支持JSON格式和Markdown表格双重维护）
5. **新增**：支持多Skill协作，预留项目、技能标签等扩展字段

## 工作空间配置

- **默认根路径**：`~/Documents/digital_employees/`
- **数据层路径**：`data-layer/employees/`
- 每位数字员工拥有独立的工作空间子目录
- 路径不存在时需主动与用户沟通确认新路径

## 核心工作流程

### Step 0：环境检测与自动初始化（零依赖自举）

在开始任何操作前，必须先执行环境检测和自动初始化。**本SKILL采用零依赖设计，无需Python脚本或其他外部工具，仅依靠SKILL自身和系统内置命令完成所有初始化工作。**

#### 阶段一：检查数据层

1. **尝试读取数据层花名册**：使用 `read_file` 检查 `data-layer/employees/roster.json` 是否存在
2. **文件不存在** → 执行数据层自举初始化（Step 0.1）
3. **文件存在** → 继续检查工作空间（阶段二）

#### 阶段二：检查工作空间

1. **检查员工工作空间根路径**：使用 `list_dir` 检查 `~/Documents/digital_employees/` 是否存在
2. **路径不存在** → 执行以下操作：
   - 告知用户："检测到员工工作空间路径不存在，可能是首次使用或在新设备上运行。"
   - 询问用户是否使用默认路径（自动创建），或指定自定义路径
   - 用户确认后使用 `terminal` 的 `mkdir` 命令创建该路径
   - **关键**：路径不存在意味着之前创建的数字员工在当前环境中不可用，但花名册数据已保留
3. **路径存在** → 执行花名册与工作空间一致性校验（Step 0.5）

### Step 0.5：花名册与工作空间一致性校验

路径存在后，必须进行双向校验，确保花名册与实际工作空间目录保持一致：

1. **使用 `list_dir` 扫描工作空间根路径**，获取所有子目录（每个子目录代表一个数字员工）
2. **读取数据层花名册** `data-layer/employees/roster.json`，获取已记录的员工列表
3. **双向比对**：
   - **花名册中有，但工作空间中没有对应目录**：该员工数据已丢失，从花名册中移除该记录
   - **工作空间中有目录，但花名册中没有记录**：读取该目录下的 `员工档案.md` 获取员工信息，将其补录到花名册中；如果 `员工档案.md` 也不存在，则忽略该目录
4. **如果发生了任何变更**：
   - 使用 `replace_in_file` 更新 `data-layer/employees/roster.json`
   - 使用 `replace_in_file` 更新本文件底部的花名册表格
   - 简要告知用户花名册已同步更新
5. 校验完成后，继续 Step 1

### Step 1：识别用户意图

读取数据层花名册 `data-layer/employees/roster.json`，根据用户意图分流：

- **用户想查看花名册**（如"有哪些员工"、"员工列表"、"查看员工"）→ 执行 Step 1A
- **用户想选择员工或招聘新员工** → 执行 Step 1B

### Step 1A：查看花名册

直接展示花名册表格。如果花名册为空则提示"目前还没有数字员工"。展示后询问用户是否需要进一步操作（选择某位员工开始工作，或招聘新员工）。

### Step 1B：展示花名册 & 选择操作

向用户展示花名册，然后询问：

- **花名册有记录时**："以下是你现有的数字员工团队：{展示表格}。你可以选择一位已有的数字员工开始工作，也可以招聘一位新员工。请问你想怎么做？"
- **花名册为空时**："目前还没有数字员工，让我们来招聘你的第一位数字员工吧！请告诉我你想要什么行业、什么岗位的员工？"

### Step 2A：选择已有数字员工

1. 用户选择一位已有数字员工
2. 使用 `read_file` 读取该员工工作空间下的 `system_prompt.md`
3. 使用 `read_file` 读取该员工工作空间下的 `员工档案.md`
4. **如果文件不存在**：提示用户该员工数据已丢失，从花名册中移除该记录，回到 Step 1
5. **文件存在**：加载提示词，正式切换为该数字员工身份
6. 以该数字员工的身份向用户打招呼，简述职责和能力，询问今天要处理什么工作

### Step 2B：招聘新数字员工

#### 阶段一：信息收集

通过多轮对话收集以下信息：

| 信息项 | 说明 | 必须/可选 |
|--------|------|-----------|
| 行业 | 所属行业领域 | 必须 |
| 岗位名称 | 具体岗位 | 必须 |
| 核心职责 | 主要负责什么工作 | 必须 |
| 员工名称 | 数字员工的名字 | 必须（可由AI建议） |
| 工作风格 | 严谨/创意/高效等 | 可选，默认根据岗位推断 |
| 沟通风格 | 正式/友好/简洁等 | 可选，默认根据岗位推断 |
| 特殊要求 | 用户的额外期望 | 可选 |

**主动引导规则**：
- 用户给出的信息过于简略时（如仅说"我要一个运营"），必须使用 `web_search` 工具联网搜索该岗位的：
  - 标准职责描述
  - 典型工作流程 / SOP
  - 常见产出物类型及格式
  - 所需核心技能和知识领域
- 搜索后整理信息展示给用户，询问是否准确、需要调整
- 基于搜索结果主动提出建议和补充
- **新增**：根据岗位自动生成技能标签列表

## 重要规则

1. **优先展示已有员工**：每次被调用时，先展示花名册让用户选择
2. **信息不足必须搜索**：用户提供的岗位信息太少时，必须使用 `web_search` 搜索该岗位的标准职责、工作流程、产出物等
3. **方案先确认再执行**：编写提示词前必须和用户确认员工画像
4. **双重花名册维护**：每次创建/删除员工后，必须同时更新JSON花名册和Markdown花名册
5. **路径自适应**：检测到路径不存在时，主动和用户沟通选择新路径，不要报错崩溃
6. **身份切换要彻底**：加载提示词后，完全以该数字员工的语气、专业知识和工作方式进行对话
7. **数据一致性**：花名册中记录了员工但工作空间路径不存在时，提示用户数据已丢失并从花名册移除
8. **提示词必须包含自维护能力**：创建的每个数字员工的 `system_prompt.md` 中，必须包含"提示词自维护"和"工作空间感知"两条工作准则，确保数字员工能根据用户需求持续迭代自己的提示词，并始终了解自己工作空间中的文件内容
9. **预留扩展字段**（新增）：创建员工时，必须在JSON花名册中包含 `projects`（初始为空数组）和 `skills`（根据岗位生成）字段
10. **预留扩展目录**（新增）：创建工作空间时，必须包含 `协作记录/` 和 `项目引用/` 目录
11. **预留扩展模块**（新增）：创建员工档案时，必须包含"参与项目"、"技能树"、"协作记录"模块（初始为占位内容）
